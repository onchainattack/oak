# OAK-T10.006 — Cross-Chain Governance Relay Attack

**Parent Tactics:** OAK-T10 (Bridge and Cross-Chain)
**Maturity:** emerging
**Chains:** Cross-chain (any protocol stack where governance actions are relayed across chains via a message bridge — LayerZero, Wormhole, Chainlink CCIP, Hyperlane, Axelar); target chains typically EVM or EVM-compatible
**First documented:** 2023–2024 (class characterised in bridge-security literature; specific governance-bridge bypass incidents from 2024 onward)
**Aliases:** "governance-bridge attack", "cross-chain governance hijack", "message-relay governance bypass", "governance-message forgery", "cross-chain proposal injection"

## Description

An attack in which the adversary exploits a cross-chain governance message relay — the infrastructure that allows a governance action (parameter change, upgrade, treasury transfer) approved on chain A to be executed on chain B — to execute a governance action on the target chain that was never approved (or was fraudulently represented as approved) on the source chain. The load-bearing surface is the message-relay infrastructure that bridges the governance decision from the source chain's governance contract to the target chain's executor contract: if the relay's message verification, replay protection, or source-chain finality check is incomplete, the attacker can inject a malicious governance message that the target chain's executor accepts as legitimately sourced.

T10.006 is structurally distinct from T10.002 (Message-Verification Bypass): T10.002 covers the general case of a bridge's message-verification predicate being bypassed (e.g., Nomad's `0x00` trusted-root); T10.006 covers the specific case where the relayed message is a **governance action** — the attacker targets not the bridge's asset-custody layer but the protocol's cross-chain governance authority. The canonical T10.002 incident (Nomad, August 2022) bypassed the bridge's message verification to drain the bridge's asset pool; a canonical T10.006 incident would bypass the relay's governance-message verification to execute an unauthorised parameter change, upgrade, or treasury transfer on the target chain while leaving the bridge's asset pool untouched. T10.006 is distinct from T9.003 (Governance Attack — single-chain): T9.003 covers governance capture within a single chain (proposal quorum manipulation, flash-loan vote-buying, malicious-proposal snowballing); T10.006 covers the case where the governance decision crosses chains via a relay and the attack targets the **relay itself** rather than the source-chain governance process.

The attack surface comprises: (a) the message-relay protocol's on-chain verification contracts on the target chain — do they correctly verify that the governance message was emitted by the legitimate governance contract on the source chain? (b) the governance-message format and replay protection — can a valid governance message from chain A be replayed on chain B with modified parameters? (c) the source-chain finality assumption — does the target-chain executor accept a governance message before the source chain's block containing that message has reached finality? (d) the relayer/executor permission model — who is authorised to deliver governance messages, and can an unauthorised relayer deliver a validly signed but maliciously parameterised message?

## Observed indicators

- A governance message executed on the target chain whose source-chain proposal cannot be located in the source chain's governance contract event log — the message was never approved on the source chain.
- A governance message whose parameters on the target chain differ from the corresponding proposal parameters on the source chain — the message was replayed or modified in transit.
- The target-chain executor contract accepts governance messages from a relayer address that is not the designated cross-chain relayer — the executor's access control permits relay by an attacker-controlled address.
- The message-relay protocol's on-chain verification contract lacks a source-chain finality check — it accepts messages from source-chain blocks that have not yet reached finality, opening a reorg-based injection window.
- The governance bridge's architecture concentrates authority in a single relayer or a small multisig without on-chain verification that the multisig's action matches a verifiable source-chain governance event.

## Detection signals

- Pre-deployment audit of the governance-relay infrastructure: verify that (a) the target-chain executor contract verifies the source-chain governance contract's address, (b) the message format includes a nonce or source-chain proposal ID to prevent replay, (c) the executor requires source-chain finality (N block confirmations) before executing, and (d) the relayer permission model is explicitly gated to the designated bridge relayer set.
- Per-message monitoring: for each governance message executed on the target chain, cross-reference against the source chain's governance contract event log — any target-chain execution whose source-chain proposal ID is absent or whose parameters differ from the source-chain proposal is a T10.006 alert.
- Relayer-address permission audit: monitor the set of addresses authorised to deliver governance messages to the target-chain executor; alert on any change to the authorised relayer set that is not accompanied by a verifiable source-chain governance proposal.
- Cross-chain replay monitoring: maintain a registry of executed governance messages indexed by (sourceChain, sourceProposalId); alert on any message whose (sourceChain, sourceProposalId) tuple has already been executed — this is a structural T10.006 replay signal.

## Real-world examples

- **LayerZero governance-relay architecture — cross-chain — 2023–2024.** LayerZero's Omnichain Fungible Token (OFT) standard and related governance-relay patterns have been the subject of security analysis by audit firms and protocol integrators. The governance-relay surface is explicitly documented by LayerZero: protocols using LayerZero for cross-chain governance must correctly configure the target-chain executor's message-verification parameters (source-chain governance contract address, relayer authorisation, message-format validation). Misconfigurations where the executor trusts a broader set of source addresses or relayers than intended constitute the T10.006 surface. OAK does not at v0.1 name a specific exploited LayerZero governance-bridge incident at the loss-scale of Ronin (T10.001) or Nomad (T10.002); the class is documented at the architectural-review and audit-finding layer.
- **Wormhole governance-relay surface — cross-chain — 2023–2024.** Wormhole's generic message-passing infrastructure supports governance-relay patterns; protocols deploying Wormhole-based cross-chain governance (e.g., token-bridge governance parameter updates, cross-chain DAO proposal execution) must correctly gate the target-chain governance executor to Wormhole-verified messages from a specific source-chain governance contract. Misconfigurations in this gating are the T10.006 surface on the Wormhole stack.
- [`examples/2024-07-compound-cross-chain-governance-relay.md`](../examples/2024-07-compound-cross-chain-governance-relay.md) — Compound cross-chain governance relay misconfiguration — audit-finding class — 2023–2024
- **Audit-firm literature — cross-chain governance relay misconfiguration class — 2023–2025.** Trail of Bits, OpenZeppelin, and Spearbit have each published advisories flagging cross-chain governance-relay misconfigurations as a distinct audit class. The surface includes: insufficient source-chain address verification; missing nonce/replay protection; lack of finality-gating; overly permissive relayer authorisation. The class is listed here as emerging rather than stable because the public named-exploit record is sparse relative to the audit-finding record.

## Mitigations

- **Protocols (OAK-M01 — governance-relay pre-deployment audit):** for every cross-chain governance deployment, audit the target-chain executor's verification of: (a) source-chain governance contract address, (b) message nonce / proposal-ID replay protection, (c) source-chain finality requirement (minimum block confirmations), (d) relayer authorisation (only the designated bridge relayer set may deliver).
- **Protocols (OAK-M09 — multi-relay-path governance quorum):** where the governance-risk budget allows, require governance messages to arrive via at least two independent relay paths (e.g., LayerZero + Wormhole) with agreement required before execution. This converts a single-relay compromise into a multi-relay coordination problem.
- **Protocols (OAK-M11 — target-chain governance-execution timelock):** impose a timelock on target-chain governance execution after message delivery; this provides a window for governance participants to observe and challenge an anomalous target-chain execution, analogous to a multisig timelock.
- **Risk teams:** maintain per-protocol inventories of cross-chain governance-relay deployments; for each, verify the source-chain governance contract address, the relay protocol, the executor's permission model, and the finality-gating configuration. Treat any governance relay that permits execution without verified source-chain finality as elevated T10.006 risk.

## Discussion

T10.006 is the governance-specific analogue of T10.002: where T10.002 covers the general case of bridge message-verification bypass (any message type, including asset-transfer messages), T10.006 covers the specific case where the relayed message is a governance action. The blast radius of a successful T10.006 attack is qualitatively different from a T10.002 asset-drain — the attacker does not drain the bridge's asset pool but instead gains control of the target-chain protocol's governance parameters, which may include upgrade-authority, fee-recipient, treasury-management, or token-minting capabilities. The extraction may be deferred (the attacker upgrades the target-chain protocol to an attacker-controlled implementation, then drains over time) rather than single-transaction.

The class is marked emerging rather than stable because (a) cross-chain governance is still a relatively young deployment pattern (the first major deployments of Wormhole-, LayerZero-, and CCIP-based governance bridges are 2023–2024), (b) the public named-exploit record at extraction scale is sparse (most misconfigurations are caught at audit), and (c) the governance-relay surface is expanding as more DAOs deploy across multiple chains. Contributors writing future T10.006 entries should anchor any new named exploit against the public forensic record and clearly distinguish the relay-bypass layer from the governance-capture layer (T9.003 / T16.x).

A structural observation: T10.006 is most likely to surface in the "long tail" of protocols that deploy cross-chain governance via a standard bridge SDK without a governance-specific security review — the bridge SDK's generic message-passing is correctly implemented, but the protocol's governance-executor gating (which source address? which relayer set? which message format?) is configured permissively by default. The mitigation surface is primarily at the protocol's governance-deployment audit layer, not at the bridge protocol's infrastructure layer.

False-positive considerations: a legitimate governance message that arrives via a different relay path than expected (e.g., the designated relayer delegates delivery to a backup relayer) is not T10.006. The distinguishing signal is that the source-chain governance event cannot be located in the source-chain governance contract's event log at all, or that the target-chain message parameters materially differ from the source-chain proposal parameters. A governance message that arrives via an unexpected relayer address but whose source-chain proposal is verifiable and whose parameters match is a relayer-delegation event (possibly misconfigured, but not T10.006). The per-message cross-chain correlation (source-chain governance event log vs. target-chain execution event) is the load-bearing forensic artefact for distinguishing T10.006 from legitimate cross-chain governance delivery.

T10.006 also composes with T9.003 (Governance Attack — single-chain): an attacker who captures the source-chain governance process (via flash-loan vote-buying, delegation takeovers, or malicious proposals) can then use the legitimate cross-chain governance relay to propagate the captured governance action to target chains — the relay is functioning correctly, but the source-chain governance decision it relays was captured. This is a T9.003/T16.x attack chained with the cross-chain governance infrastructure, not a T10.006 attack on the relay itself. The OAK classification convention assigns the case to the load-bearing-surface technique: if the relay verification was bypassed, T10.006; if the source-chain governance was captured, T9.003/T16.x.

v0.x deferred items: a systematic inventory of major cross-chain governance-relay deployments (DAO → bridge → target chain), enumerating the source-chain governance contract, the bridge/relay protocol, the target-chain executor, and the verification parameters (source-address check, nonce/replay protection, finality-gating, relayer authorisation) for each; an integration-test framework that validates governance-relay message integrity end-to-end (source-chain proposal emission → bridge verification → target-chain execution) against a configurable threat model.

Attribution-strength considerations: T10.006 attribution is typically at the `pseudonymous-attacker` level for operational exploits — the attacker address that submitted the malicious governance message to the target-chain executor is on-chain attributable. The source-chain governance event's absence from the governance contract's event log is verifiable from on-chain data; the target-chain execution event is on-chain attributable. The forensic record is split across two chains, requiring cross-chain event correlation — an analytical task that is well-supported by forensic-provider infrastructure but requires explicit T10.006 detection logic rather than general-purpose cross-chain tracing.
