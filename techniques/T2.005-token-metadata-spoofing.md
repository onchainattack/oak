# OAK-T2.005 — Token Metadata Spoofing

**Parent Tactics:** OAK-T2 (Token-Level Exploits), OAK-T6 (Defense Evasion)
**Maturity:** observed
**Chains:** EVM (primary — ERC-20 name/symbol/decimals can differ from the canonical token the contract impersonates); Solana (secondary — SPL Token metadata URI manipulation); BSC (PancakeSwap token-impersonation wave 2021)
**First documented:** 2020-2021 (token-impersonation wave on Uniswap V2/V3; fake USDT/USDC/ETH tokens with manipulated name/symbol/decimals fields)
**Aliases:** "token symbol spoofing", "fake token metadata", "ERC-20 name impersonation", "counterfeit token branding", "metadata deception"

## Description

A token-deployment attack in which the adversary deploys a token contract whose ERC-20 `name()`, `symbol()`, and `decimals()` return values impersonate a legitimate high-value token (USDT, USDC, WBTC, ETH), while the token's actual behaviour differs materially — it may be a honeypot, a high-tax token, or a token whose transfer function routes through an attacker-owned liquidity pool. The wallet UI and block explorer display the impersonated name/symbol, misleading the user into believing they are interacting with the legitimate asset.

The impersonation surface is the ERC-20 metadata primitives themselves — the three public-view functions `name()`, `symbol()`, and `decimals()` — which are not standardised against a canonical registry at the wallet or block-explorer display layer. A token contract that returns `name() = "Tether USD"`, `symbol() = "USDT"`, and `decimals() = 6` will display identically to the legitimate USDT contract in virtually every wallet UI and block explorer, regardless of the token's actual transfer logic, supply distribution, or pool routing. The user has no visual indication that the token they are approving or swapping is not the canonical USDT at `0xdAC17F958D2ee523a2206206994597C13D831ec7`; the wallet displays "USDT" and the user proceeds.

T2.005 is structurally distinct from **OAK-T6.006 (Counterfeit Token Impersonation)**. T6.006 covers the full fake-token deployment with matching name/symbol/logo as the victim-facing lure — the adversary constructs a complete counterfeit-brand surface spanning on-chain metadata and off-chain brand assets (logo, website, social-media presence, CoinMarketCap/CoinGecko listing impersonation). T2.005 covers the specific on-chain metadata-primitive (ERC-20 name/symbol/decimals) as the impersonation mechanism — the token may not fully impersonate the original's brand at the off-chain layer but exploits the wallet/explorer's display of on-chain metadata to mislead. T6.006 is the full-spectrum brand-counterfeit; T2.005 is the metadata-layer mechanism that makes the display-level impersonation possible. The two Techniques co-occur in most incidents but are structurally separable: a token can spoof metadata (T2.005) without a full off-chain brand-counterfeit operation (T6.006), and a full counterfeit operation (T6.006) always includes metadata spoofing (T2.005) as its on-chain impersonation primitive.

T2.005 is structurally distinct from **OAK-T1.001 (Modifiable Tax / Anti-Sell Transfer Restriction)** because the impersonation surface is the metadata return values, not the transfer logic. T1.001 spoofs the transfer-function behaviour; T2.005 spoofs the display-layer metadata that labels the token in UIs. A token can have clean transfer logic (no anti-sell, no modifiable tax) and still be a T2.005 impersonation of a legitimate asset; conversely, a legitimate-asset impersonation with anti-sell logic bundles T2.005 + T1.001.

The BSC PancakeSwap token-impersonation wave of 2021 is the canonical chain-scale cohort: attackers deployed tokens with `name()/symbol()` matching top-volume PancakeSwap pairs, seeded thin liquidity, and routed swaps through attacker-owned pools where the received token was the impersonator rather than the legitimate asset.

## Observed indicators

- A token's `name()` and `symbol()` match a well-known asset (USDT, USDC, WBTC, ETH) but its contract address differs from the legitimate asset's known deployment address.
- A token's `name()` and `symbol()` match a well-known asset but its `decimals()` return value differs from the legitimate asset's canonical decimals (e.g., a token claiming to be USDC with `decimals() = 18` instead of `6`).
- The wallet UI or block explorer displays the token with the impersonated name/symbol; the user is presented with "USDT" in the token selector with no visual indicator that the contract address differs from the canonical deployment.
- The token's `transfer()` function includes a fee, rebase, max-transaction limit, or other logic not present in the legitimate asset — the metadata says "USDT" but the behaviour says "honeypot."
- The token's liquidity pool is routed through a non-canonical factory or a pool whose token0/token1 pair pairs the impersonator token against a legitimate base asset, creating a swap path where the user receives the impersonator instead of the canonical token.
- On Solana, the SPL Token metadata URI (on-chain or via Metaplex Token Metadata Program) points to a URI whose off-chain JSON payload contains `name`/`symbol` fields matching a legitimate token while the on-chain mint address differs.

## Detection signals

- **Static (token-metadata cross-reference against canonical token registries).** For every `name()` + `symbol()` + `decimals()` tuple observed on-chain, check whether the tuple matches a known asset in a canonical token registry (Uniswap Token List, CoinGecko token list, 1inch Token List, Solana Token List, chain-specific canonical address lists). Flag any token whose metadata tuple matches a registry entry but whose contract address does not match the registry's canonical deployment address.
- **Static (decimals mismatch as leading indicator).** A token returning `symbol() = "USDC"` with `decimals() = 18` is an immediate metadata-spoofing flag because the legitimate USDC uses `decimals() = 6`. Decimals mismatch is the cheapest static check and catches a non-trivial fraction of naive impersonators.
- **Static (wallet-side token-address verification).** Wallet UIs should cross-reference any token displayed in the token selector against the chain's canonical deployment address for that symbol and surface a warning when the address differs. The absence of this cross-reference at the wallet layer is the primary reason T2.005 remains effective.
- **Static (block-explorer-level metadata verification).** Block explorers can flag tokens whose name/symbol match a known asset but whose address differs from the canonical deployment, and surface an "impersonation warning" banner on the token's page.
- **Behavioural (transfer-function simulation).** Simulate a transfer from a non-deployer address and compare the actual token balance change against the expected amount. If the transfer deducts a fee, triggers a rebase, or fails, the token is flagged regardless of its metadata claims.
- **Behavioural (pool-routing cross-reference).** For every swap routed through a DEX aggregator, verify that the output token's contract address matches the canonical deployment for the symbol the user selected. Aggregators that route swaps without this verification yield T2.005 surfaces at the routing layer.

## Reference implementations

- MG `mg-detectors-rs` — coverage **gap** at v0.1. The token-metadata cross-reference primitive against canonical token registries (Uniswap Token List, CoinGecko, 1inch, Solana Token List) is a v0.x detector target. The static `name()/symbol()/decimals()` extraction primitive exists in the token-introspection layer but is not currently wired to a registry-cross-reference workflow.
- GoPlus — partial via token-name/symbol display and honeypot detection; registry-cross-reference against a canonical token list is not surfaced as a first-class check.
- TokenSniffer / Honeypot.is — partial via honeypot detection at the transfer-function level (catches the T1.001 co-occurrence); metadata-registry-cross-reference is not a first-class detection axis.
- Wallet-side: MetaMask token-import warning, Phantom token-flagging, and Rabby token-risk scoring represent partial wallet-side implementations; none cross-references the token's name/symbol/decimals tuple against a canonical registry and surfaces an impersonation warning at the token-selector level.
- Aggregator-side: 1inch, Matcha, and Odos perform some degree of token-address verification in their token-selector UIs; the coverage is not standardised across aggregators and the implementation details are not publicly-documented to a level that permits an OAK v0.1 coverage assessment.

## Mitigations

- **Wallet-side metadata-verification against canonical token registries (highest-leverage mitigation).** Wallet UIs should cross-reference every token displayed in the token selector against a canonical token registry (Uniswap Token List, CoinGecko, 1inch Token List, Solana Token List, chain-specific canonical address lists) and surface an impersonation warning when a token's metadata tuple matches a registry entry but its contract address differs. This is a display-layer mitigation — it does not require any on-chain change and closes the primary T2.005 surface at the user-interaction point.
- **User-side verification of token contract address before approving.** Users should verify the token's contract address against the canonical deployment address on the protocol's official documentation or a trusted block explorer before approving any token spend. This is the user-side fallback when wallet-side verification is absent.
- **Block-explorer-level impersonation warning banners.** Block explorers should maintain a canonical token-address registry per chain and flag tokens whose name/symbol match a canonical entry but whose address differs, surfacing an impersonation warning on the token's explorer page.
- **DEX aggregator token-address verification at the routing layer.** Aggregators should verify that the output token of every swap route matches the canonical deployment address for the symbol the user selected in the UI, and reject routes that route to a non-canonical address when the symbol matches.
- **Canonical token-registry standardisation.** The absence of a universally-adopted canonical token registry per chain is itself a T2.005 surface. Wallet, explorer, and aggregator adoption of a shared canonical registry (with transparent inclusion criteria and community-governed maintenance) would close a large fraction of the display-layer impersonation surface.
- **Cross-comparison with T6.006.** T6.006 mitigations (full-spectrum brand-counterfeit detection including off-chain logo/website/social-media impersonation checks) are complementary: T6.006 catches the full counterfeit operation, while T2.005 catches the metadata-primitive mechanism even when off-chain brand-counterfeit signals are absent.

## Discussion

T2.005 addresses a structural gap in OAK v0.1's T2 coverage. The 2020-2021 Uniswap V2 token-impersonation wave and the 2021 BSC PancakeSwap fake-token wave are the canonical historical cohorts that demonstrate the metadata-spoofing primitive at scale. The technique is "observed" rather than "stable" because the wallet-side and explorer-side detection infrastructure has not yet standardised around registry-cross-reference verification — the surface remains open on most wallet UIs and block explorers as of OAK v0.1.

The ERC-20 metadata design choice — that `name()`, `symbol()`, and `decimals()` are arbitrary return values with no on-chain canonical-registry enforcement — is the structural root cause. The ERC-20 standard intentionally leaves these fields as informative display hints rather than as load-bearing identity claims; the vulnerability arises because wallet UIs and block explorers treat them as load-bearing identity claims without cross-referencing the contract address against a canonical registry. The metadata primitives are not malicious per se — any deployed token is entitled to return any name/symbol string — but the *combination* of a well-known-asset-matching metadata tuple with a non-matching contract address, displayed to a user without a cross-reference warning, is the T2.005 surface.

Solana's SPL Token Metadata Program adds a second layer: the `name`/`symbol` fields are stored in an off-chain JSON file referenced by an on-chain URI, creating a two-hop metadata-resolution path (mint address -> metadata account -> URI -> JSON -> name/symbol). The off-chain JSON hop adds an updateability surface — the metadata JSON can be modified by the metadata-account update authority without an on-chain mint-address change — that does not exist in the ERC-20 single-hop model. Solana-wallet T2.005 detection requires verifying both the metadata-account on-chain fields and the off-chain JSON contents against a canonical mint-address registry.

Promotion to `stable` will follow once (a) wallet-side registry-cross-reference verification becomes standardised across major wallet UIs, (b) per-chain canonical token registries achieve broad adoption, and (c) the per-incident worked-example anchor set expands beyond the single multi-year cohort file to include standalone named-incident worked examples with on-chain deployment transactions.
