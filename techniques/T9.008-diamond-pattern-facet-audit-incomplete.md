# OAK-T9.008 — Diamond-Pattern Facet-Audit Incomplete

**Parent Tactics:** OAK-T9 (Smart-Contract Exploit), OAK-T6 (Defense Evasion)
**Maturity:** emerging
**Chains:** EVM (primary; EIP-2535 diamond pattern is EVM-native); non-EVM diamond-pattern analogues (proxy-facet architectures on Solana, Sui, Aptos) are pipeline anchor candidates at v0.x
**First documented:** ~2022-07 (Li.Fi exploit ~$600K, v1 diamond-pattern facet-addition vulnerability); class-level calibration anchor Li.Fi July 2024 (~$10M, post-audit facet addition)
**Aliases:** "diamond-proxy facet gap", "post-audit facet addition", "unaudited-facet exploit", "EIP-2535 audit-scope gap", "facet-upgrade-path exploitation", "diamond-cut attack"

## Description

A diamond-proxy (EIP-2535) pattern where the project truthfully claims audit coverage for some facets but deploys additional unaudited facets — or where the facet upgrade authority is not bound by the audit scope. The diamond pattern's multi-facet architecture creates a **structural audit-scope gap**: an audit of Facets A, B, and C does not cover Facet D added post-audit, even when D holds treasury-routing or upgrade authority. The audited facets match their audited bytecode; the gap is the unaudited facets, not drift in audited ones.

T9.008 is **structurally distinct** from sibling Techniques at the *audit-scope* level:

- **T6.003 (Audit of Different Bytecode Version)** — covers the case where audited bytecode differs from deployed bytecode. T9.008's audited facets *match* their audited bytecode — the audit scope simply does not include the post-audit-added facets. The audit is truthful about what it covers; it does not cover enough.
- **T6.005 (Proxy-Upgrade Malicious Switching)** — covers proxy-implementation switching to malicious bytecode, typically via operator-key-compromise or on-chain message forgery. T9.008's diamond-upgrade (`diamondCut`) is a legitimate design primitive — the gap is that the upgrade-authority path adds facets outside the audit scope, not that the upgrade itself is maliciously triggered.
- **T6.002 (Fake-Audit Claim)** — covers false audit attestation. T9.008's audit claim is truthful: the audited facets were audited and match their deployed bytecode. The gap is that the project deploys *additional* facets that are not covered by the audit, and the public-facing audit claim does not make the scope limitation clear.
- **T6.004 (Audit-Pending Marketing Claim)** — covers claiming an audit that does not exist. T9.008's audit exists and is legitimate; the scope is the problem.
- **T9.004 (Access Control Misconfiguration)** — covers access-control flaws. Some T9.008 cases manifest as access-control flaws in the unaudited facet (e.g., a missing `onlySelf` check in a diamond-cut-added facet), but the structural feature is the audit-scope gap, not the access-control primitive itself.

The diamond pattern (EIP-2535) is a proxy architecture where a single `Diamond` proxy contract delegates calls to multiple implementation contracts called "facets." Facets are added, replaced, or removed via the `diamondCut` function, which is typically gated behind a contract owner or governance contract. The pattern's structural properties create the T9.008 audit-scope gap:

1. **Unbounded facet count.** The diamond can support an arbitrary number of facets. An audit may cover N facets at deployment; the project can add N+1 post-audit without re-auditing.
2. **Upgrade-authority concentration.** The `diamondCut` authority (contract owner or governance multisig) can add facets with arbitrary logic, including facets that hold treasury-routing, fee-extraction, or upgrade-authority-transfer functionality. The audit scope is typically bounded by the facets present at the time of the audit; the upgrade authority is outside the scope.
3. **Facet-selector collision surface.** Two facets can register overlapping function selectors; the diamond's selector-resolution logic (typically first-registered-wins or a configurable lookup table) determines which facet handles a call. A post-audit facet can shadow an audited facet's function selectors, redirecting calls to unaudited logic.
4. **Cross-facet storage collision surface.** The diamond pattern uses a shared storage layout (Diamond Storage / AppStorage pattern). A post-audit facet that incorrectly accesses storage slots used by audited facets can corrupt or overwrite audited-facet state.

Two distinguishable sub-patterns:

- **T9.008.001 — Post-audit facet addition introduces exploit surface.** The project deploys a diamond with audited Facets A, B, C. Post-audit, the operator adds Facet D (e.g., a new bridge integration, a new token-swap router, a new fee module). Facet D is unaudited and contains a vulnerability (missing input validation, unprotected external call, access-control gap) that is exploited. The audit claim ("audited by Firm X on date Y") is truthful for Facets A, B, C but misleading about the diamond's overall security posture because Facet D is not covered. Canonical: Li.Fi July 2024 (~$10M) — a post-audit facet addition introduced a call-verification gap.
- **T9.008.002 — Operator-side rug via post-audit facet addition.** The project operator uses the `diamondCut` authority to add a facet that enables treasury extraction (e.g., a facet that exposes an `emergencyWithdraw(recipient)` function or transfers the `diamondCut` authority to an operator-controlled EOA). The audited facets remain audited; the unaudited-rug-facet is added after the audit is complete. The operator can truthfully claim that the *audited* facets were deployed unchanged, and the audit firm correctly audited the facets present at the time. The rug facet is technically outside the audit scope.

## Observed indicators

- **Diamond-architecture fingerprints:** the proxy contract emits `DiamondCut` events (EIP-2535 standard event); the proxy's implementation surface is split across multiple facet contracts. The diamond's Louper (`facets()`, `facetFunctionSelectors()`, `facetAddresses()`) returns the current facet set.
- **Temporal-audit-gap fingerprints:** the audit report lists N facets; the current diamond state (queryable on-chain via the Louper interface) lists N+M facets. The M post-audit facets were added after the audit date.
- **Facet-deployment-date fingerprints:** the deployment date of post-audit facets (queryable via the facet contract's creation transaction) post-dates the audit report publication date.
- **Upgrade-authority fingerprints:** the `diamondCut` authority is a contract owner EOA or a small multisig rather than a time-locked governance contract with a mandatory delay. The upgrade-authority path is outside the scope of the audit.
- **Audit-report language fingerprints:** the audit report uses language like "audit of the smart contracts listed in Appendix A" without a prominent disclosure that the diamond architecture allows post-audit facet addition outside the scope. The project's marketing materials claim "audited by Firm X" without qualifying the scope.
- **Facet-selector-collision fingerprints:** a post-audit facet registers a function selector that overlaps with an audited facet's selector, redirecting the function call to unaudited logic.

## Detection signals

- **Diamond-Loupe-vs-audit-report comparison (highest-leverage):** defender-side tooling should query the diamond's Louper interface to enumerate the current facet set and compare against the audit report's facet list. Any facet present on-chain but absent from the audit report is a T9.008 gap signal.
- **Post-audit-facet-deployment-date cross-reference:** for each facet in the current diamond state, check the facet contract's deployment transaction timestamp against the audit report publication date. Facets deployed after the audit date are outside the audit scope.
- **Audit-report-scope-language analysis:** audit reports should include a prominent, standardised disclosure that "the diamond architecture permits post-audit addition of facets not covered by this audit. Users should verify the current facet set against this report's scope before interacting with the protocol." Absence of this disclosure is a risk signal.
- **`diamondCut`-event monitoring:** on-chain monitoring of `DiamondCut` events should trigger an alert when new facets are added post-audit, particularly when the added facets introduce external-call surfaces, token-transfer logic, or treasury-routing functionality.
- **Facet-function-selector-collision detection:** tools should detect when a post-audit facet registers a function selector that overlaps with an audited facet's selector, flagging the collision as a potential shadowing attack.
- **Upgrade-authority-path analysis:** tools should flag diamonds where the `diamondCut` authority is an EOA or a small multisig without a time-lock, as this enables single-actor post-audit facet addition. Governance-contract-gated `diamondCut` with a mandatory time-lock is the safer pattern.

## Real-world examples

- **Li.Fi diamond-facet exploit — EVM — 2024-07 (canonical T9.008.001 case)** — ~$10M stolen via a post-audit facet addition in Li.Fi's diamond-proxy architecture. The protocol had previously been audited, and the audited facets matched their audited bytecode. A post-audit facet addition introduced a call-verification gap in the bridge-integration facet — the new facet called an external contract without validating the contract address, allowing an attacker to supply a malicious contract address that drained approved tokens from users who had granted infinite approvals to the Li.Fi diamond. The audit did not cover the post-audit facet; the audit claim was truthful about the audited facets but misleading about the diamond's overall security posture. Attribution: `confirmed` at the protocol-operator level (the post-audit facet was added by Li.Fi's operator, not by an external attacker); the vulnerability was in the unaudited facet, not in the operator's intent. Per Li.Fi's post-mortem, the facet was added to integrate a new bridge provider; the call-verification gap was a missing `onlySelf` / sender-validation check in the new facet's external-call logic.
- **Li.Fi v1 facet vulnerability — EVM — 2022-07** — An earlier Li.Fi exploit (~$600K) involving a diamond-pattern facet-addition vulnerability. The v1 incident was smaller in scale but established the diamond-facet-audit-gap pattern as a recurring vulnerability class for the same protocol. The July 2024 incident is the calibration anchor; the 2022 incident establishes the recurring pattern.
- **Operator-side rug via post-audit facet addition — various diamond-pattern protocols — 2023–2025 (pipeline anchors at v0.x)** — Multiple diamond-pattern protocols where the operator used the `diamondCut` authority to add a post-audit facet enabling treasury extraction. The audited facets remained unchanged and audited; the rug facet was added outside the audit scope. The operator's claim that the protocol was "audited by Firm X" was technically true but operationally misleading. Per-incident worked-example anchors still forming at v0.1; held as pipeline anchors for v0.x.

## Mitigations

- **Audit-report diamond-facet-scope disclosure (highest-leverage):** audit firms auditing diamond-pattern protocols should include a standardised, prominent disclosure that (a) the audit covers only the facets listed in the report's scope, (b) the diamond architecture permits post-audit addition of facets not covered by the audit, (c) users should verify the current facet set against the report's scope before interacting, and (d) the audit does not cover the `diamondCut` upgrade-authority path's security properties. This disclosure should be a required element of any diamond-protocol audit report.
- **Protocol-side post-audit-facet-re-audit commitment:** diamond-pattern protocols should commit to re-auditing any post-audit facet addition that introduces external-call surfaces, token-transfer logic, or treasury-routing functionality before the facet is deployed to production.
- **Protocol-side time-locked `diamondCut` with mandatory delay:** the `diamondCut` authority should be a governance contract with a mandatory time-lock (e.g., 48-hour delay) and a public notification of the planned facet addition before execution. This prevents single-actor post-audit facet addition and gives the community time to review the new facet's bytecode.
- **User-side diamond-Louper-vs-audit-report verification:** tools (Louper.dev, wallet-integrated diamond-verification surfaces) should allow users to query the current diamond facet set and compare it against published audit reports before interacting with the protocol. The query should surface: "This diamond has N facets. M of them are covered by Audit Report X (date). The remaining K facets are not covered by any published audit."
- **Facet-function-selector-collision prevention:** the diamond's `diamondCut` logic should reject facet additions whose function selectors collide with existing facets' selectors, unless the addition is explicitly a replacement (not an addition). Collision detection should be enforced at the diamond-contract level, not left to operator diligence.
- **Auditor-side diamond-facet-set baseline publication:** audit firms should publish a machine-readable baseline of the diamond's facet set at the time of the audit (facet addresses + function selectors + bytecode hashes) that can be programmatically compared against the current on-chain state. This enables automated "has this diamond diverged from its audit baseline?" queries.

OAK Mitigation cross-references at v0.1 are a **gap** — no existing OAK-M* entry directly maps to diamond-facet-audit-scope disclosure or user-side diamond-Louper-vs-audit-report verification. v0.x candidate Mitigation entries: "OAK-M-x audit-report diamond-facet-scope mandatory disclosure"; "OAK-M-x post-audit-facet-re-audit commitment"; "OAK-M-x time-locked diamondCut with mandatory delay"; "OAK-M-x user-side diamond-Louper-vs-audit-report verification".

## Discussion

T9.008 addresses a structural gap in OAK's coverage of audit-scope limitations in diamond-proxy (EIP-2535) architectures. Existing T6 / T9 Techniques that cover audit-scope issues (T6.003 — Audit of Different Bytecode Version; T6.002 — Fake Audit Claim; T6.004 — Audit-Pending Marketing Claim) all share the assumption that the *deployed bytecode* is the unit of audit-scope analysis. T9.008's structural feature is that the diamond pattern's multi-facet architecture makes the *facet set* the relevant unit of audit-scope analysis, not the individual bytecode — and the facet set can change post-audit without invalidating the existing audit claim.

The structural OAK lesson is that **audit claims on diamond-pattern protocols should be indexed by facet set, not by protocol name**. "Protocol X is audited" is a misleading claim when Protocol X uses a diamond pattern where post-audit facets can be added without re-audit. The correct claim is "Facets A, B, C (as listed in Audit Report Y, published Z) are audited. Query the diamond's Louper interface for the current facet set. Facets added after Z are not covered."

The Li.Fi July 2024 incident (~$10M) is the calibration anchor. The protocol had a legitimate audit; the audited facets matched their audited bytecode; the vulnerability was introduced via a post-audit facet addition that the audit did not cover. Li.Fi's post-mortem acknowledged the gap — the new facet was added to integrate a new bridge provider, and the missing `onlySelf` / sender-validation check was an oversight in the new (unaudited) code. The incident is not a case of operator malfeasance (T9.008.002) but of the diamond pattern's structural audit-scope gap (T9.008.001) combined with an operator-side failure to re-audit the new facet before deployment.

The operator-rug sub-pattern (T9.008.002) is structurally more severe because the operator's intent is malicious — the post-audit facet is added specifically to enable value extraction. The audit claim is weaponised: the operator uses the audit as a trust signal while adding unaudited facets that subvert the protocol's security. This sub-pattern chains into the broader T6 (Defense Evasion) Tactic because the audit claim is the defense-evasion surface.

The class is included at v0.1 with `emerging` maturity rather than `stable` because (a) the canonical anchor (Li.Fi July 2024) is a single incident with a confirmed pattern recurrence (Li.Fi 2022 v1 at lower value), but independent multi-protocol anchors are still forming, (b) the operator-rug sub-pattern (T9.008.002) has per-incident anchors tracked in rug-pull databases but structured per-incident worked examples for OAK are still pipeline, (c) audit-report diamond-facet-scope disclosure is not yet standardised across audit firms, and (d) user-side diamond-Louper-vs-audit-report verification tooling is not yet operationalised. Promotion to `stable` will follow once multi-protocol independent anchors land and audit-firm standardisation of diamond-facet-scope disclosure reaches material coverage.

The class structurally generalises beyond EIP-2535 diamonds to any proxy architecture where the implementation surface is a *set* of contracts rather than a single implementation contract, and where the set can be extended post-audit. Diamond-pattern analogues on non-EVM chains (Solana's upgradeable-program pattern with multiple program-derived addresses; Sui's module-upgrade pattern; Aptos's resource-account-upgrade pattern) have the same structural audit-scope gap and are pipeline anchor candidates at v0.x. The structural condition is: (1) the protocol uses a multi-implementation proxy architecture, (2) the implementation set is extensible post-deployment, (3) the audit covers the implementation set at a point in time, and (4) the upgrade-authority path allows post-audit addition of implementations outside the audit scope. These conditions are architecture-agnostic and chain-agnostic.
