# OAK-T9.015 — Degenerate-Input Signature-Verification Bypass

**Parent Tactics:** OAK-T9 (Smart-Contract Exploit)
**Adjacent Techniques:** OAK-T10.002 (Message-Verification Bypass — the application-layer sibling: the verifier admits a message it should reject because a *business-logic* check is missing or wrong, rather than because the cryptographic primitive returned true on a degenerate input), OAK-T9.001 (Oracle Price Manipulation — the downstream outcome when the forged-authorised message is a price update), OAK-T10.005 (Light-Client Verification Bypass), OAK-T9.004 (Access-Control Misconfiguration)
**Maturity:** emerging
**Chains:** chain-agnostic (any runtime whose contracts verify signatures or proofs over an elliptic curve — EVM, Hedera, Cosmos, Solana, Move-family; the defect is in the verifier's input validation, not in the chain)
**First documented:** 2026-07 (Bonzo Lend / Supra on-demand oracle verifier — first documented on-chain crypto-theft anchor). The underlying cryptographic hazard is long-known in the protocol-engineering literature: subgroup-membership and identity-element handling are explicit requirements in the IRTF CFRG BLS-signature specification (`KeyValidate`, subgroup checks), and small-subgroup / invalid-curve attacks predate crypto by well over a decade.
**Aliases:** "zero signature bypass", "identity-element forgery", "point-at-infinity attack", "small-subgroup attack", "missing subgroup check", "degenerate pairing input", "zeroed signature"

## Description

A verifier accepts a **mathematically degenerate input** — the zero scalar, the identity element (the point at infinity), or a point lying outside the correct prime-order subgroup — and its verification equation evaluates to true *as designed*, because the equation holds trivially for those inputs. Nothing is broken cryptographically and no key is stolen. The verifier simply never asked whether the values it was handed were legitimate group elements before running the check, and the contract then reads a trivially-true result as proof that an authorised party signed.

The shape a defender should recognise in pairing-based (BLS) verification: the check tests whether `e(sig, g2) == e(H(m), pk)`. If both `sig` and `pk` resolve to the identity element, each side collapses to the identity of the target group, and the equation is satisfied for **any** message `m`. An attacker who controls both the signature field and the public-key reference — or who can reach any code path where the committee public key resolves to the zero point — can have arbitrary content accepted as committee-signed. The same class covers non-pairing settings: ECDSA verifiers that fail to reject `r = 0` or `s = 0`, verifiers that skip `KeyValidate` and accept a public key of infinity, and verifiers that accept points off the curve or in a small subgroup so that a cofactor-multiplication step annihilates them.

The defender's question is therefore not "is the signature valid?" but **"did the verifier validate its inputs before deciding validity?"** This is what makes T9.015 worth separating from its neighbours: the defect is invisible to every control aimed at the authorisation *policy* (who may sign, what quorum is required, which key is authoritative). All of those can be correct and correctly enforced. The failure is one level below them, in the primitive that answers whether the signature satisfies the policy, and it converts an unauthenticated caller into an authorised committee without any key compromise, market movement, or governance action.

T9.015 is **structurally distinct** from the adjacent classes:

- **T10.002 (Message-Verification Bypass).** T10.002's defect lives at the application layer — a missing source-channel check (Secret Network, 2026-06), a trusted root initialised to zero (Nomad, 2022-08), an unvalidated guardian account (Wormhole, 2022-02). The cryptographic primitive, where one is used at all, behaves correctly; the contract asks the wrong questions. T9.015's defect is that the primitive itself returns true on inputs it should have rejected outright, so the contract can ask exactly the right questions and still be told "yes". The two classes co-occur and the boundary is occasionally genuinely ambiguous (Nomad's zero-root is the degenerate-input shape expressed at the application layer); contributors should map both when the boundary is unclear and say which layer was load-bearing.
- **T9.004 (Access-Control Misconfiguration).** There, the authorisation check is absent, guards the wrong role, or was left open. In T9.015 the check is present, correctly scoped, and runs — it just resolves in the attacker's favour on degenerate inputs.
- **T9.001 (Oracle Price Manipulation).** T9.001 is mitigated at the oracle-resilience layer (TWAP windows, multi-venue feeds, deviation breakers) because its premise is that the attacker moved a real reported value. T9.015 needs no market and no capital: the value is asserted directly and the verifier vouches for it. When the forged message is a price update, the incident presents as T9.001 downstream while its mechanism is T9.015 — map both, and note that oracle-resilience controls only bound the blast radius here, they do not close the hole.
- **T11.001 / T10.001 (signer-key compromise classes).** Those require the attacker to obtain key material. T9.015's defining property is that **no key material is ever needed**.

The class earns its own identifier because its mitigation is specific, cheap, and mechanical — reject zero, identity, off-curve and off-subgroup inputs at the verifier boundary before any pairing or curve operation — while none of the controls that defend its neighbours touch it. A protocol can run threshold signing with perfect operator separation, rotate keys on schedule, and enforce a strict quorum, and still be drained by an all-zero byte string.

## Observed indicators

- A verification-bearing call whose signature, proof, or public-key field is **all zeros**, or whose committee/key reference resolves to the identity element (point at infinity).
- A state update attributed to a signing committee for which **no committee member has any corresponding off-chain signing record** — the on-chain artefact claims a signature that was never produced.
- An authorised-looking update whose payload is wildly outside any plausible range (the degenerate input carries no constraint on message content, so attackers typically assert extreme values rather than subtle ones).
- Successful verification from a caller with **no prior relationship** to the signing set — no stake, no registration, no key rotation history.
- In pairing settings: calldata containing curve points that fail a subgroup-membership test, or a cofactor-cleared value that lands on the identity.
- The legitimate signer set continuing to publish correct values **concurrently** with the forged one — the real committee is untouched and unaware, so genuine and forged updates interleave.

## Detection signals

- **Verifier-boundary input validation (highest leverage — this is prevention, not detection).** Reject `sig == 0`, `pk == identity`, points off the curve, and points outside the prime-order subgroup before evaluating any pairing or verification equation. `KeyValidate`-equivalent checks are mandated by the CFRG BLS specification precisely because their omission is exploitable; treat their absence in an audited verifier as a finding, not a nit.
- **Degenerate-calldata screening.** Flag any transaction to a verification-bearing entrypoint whose signature/proof/key fields are all-zero or identity-valued. This is a cheap, near-zero-false-positive rule: legitimate signatures are never zero.
- **Signature-provenance reconciliation.** For each accepted committee-signed update, confirm a matching off-chain signing event exists among the signer set. A verified-on-chain update with no off-chain counterpart is the direct signature of this class and catches it even when the degenerate value is not literally zero.
- **Value-plausibility circuit breaker on consuming protocols.** A downstream consumer (lending market, perp DEX) should reject or quarantine an oracle value that deviates from its previous accepted value beyond a bound, regardless of how well-signed it appears. This does not prevent T9.015 but bounds it — the Bonzo case turned a verifier bug into a $9M loss because a price of 1 followed by 30 zeroes was consumed as collateral valuation without a sanity bound.
- **Time-to-borrow anomaly at consumers.** Extraction against a forged value is typically near-instant (Bonzo: borrow within eight seconds of the price landing). Flag borrow/settle actions that follow an oracle write from the same or an adjacent block, especially against thin collateral.
- **Third-party verifier inventory.** Protocols consuming an external oracle or bridge verifier inherit its input-validation quality. Maintain an explicit inventory of which verifier contracts your protocol trusts, and whose audit scope actually covered them.

## Real-world examples

- [`examples/2026-07-bonzo-lend-supra-oracle-zeroed-signature-drain.md`](../examples/2026-07-bonzo-lend-supra-oracle-zeroed-signature-drain.md) — Bonzo Lend / Supra on-demand oracle — Hedera — 2026-07-12 — ~\$9.05M. **Canonical anchor.** Supra's on-demand verifier did not reject zero, identity, or off-subgroup inputs; a price update carrying an all-zero signature field `[0,0]` against a zero-point public-key reference satisfied the pairing equation trivially and was written on-chain as committee-signed. The attacker inflated SAUCE by roughly 12 orders of magnitude and borrowed ~\$9.05M against 250 SAUCE (worth a few dollars) on Hedera's largest lending market.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. The actionable control is inside the verifier contract (input validation before pairing), which is a pre-deployment / audit-scope surface rather than a runtime risk-engine surface. The consuming-protocol half (value-plausibility breaker, oracle-write-to-borrow proximity) is expressible as a runtime rule and is the plausible near-term coverage target.
- Static analysis / formal verification: subgroup-membership and identity-element checks are tractable for pre-deployment verification tooling, and the CFRG BLS specification gives an unambiguous conformance target. Coverage across mainstream Solidity/Move analysers is **partial and untested** at v0.1 — contributors with concrete detector rules should extend this row rather than assume coverage.
- Vendor coverage: **unassessed** at v0.1 for the verifier-boundary check. No public claim of a shipping detector for degenerate curve inputs in third-party oracle verifiers is known to OAK; do not read this row as "no coverage exists".

## Mitigations

- **OAK-M16 (Pre-Deployment Audit and Formal Verification):** highest-leverage. Input validation at the verifier boundary — reject zero, identity, off-curve, and off-subgroup elements before any pairing — is mechanically checkable and belongs in every audit of a signature- or proof-verifying contract. The Bonzo case's structural lesson is one of **audit scope**: the consuming protocol was not the defective component, and the defective component was a third-party dependency whose verifier internals the consumer never reviewed.
- **OAK-M09 (TWAP / Multi-Venue Oracle with Deviation Circuit Breaker):** bounds the blast radius rather than closing the class. A deviation breaker on the consuming market would have rejected a 12-orders-of-magnitude jump no matter how convincingly it was signed. Load-bearing for any protocol that consumes an external price feed.
- **OAK-M11 (Rate-Limiting and Per-Block Caps):** caps single-block extraction against a value that has just changed, converting an instant total drain into a partial, observable one.
- **OAK-M39 (Cross-Protocol Watcher Network):** signature-provenance reconciliation — an accepted committee-signed update with no matching off-chain signing event — is a watcher-shaped signal that the signing committee itself is positioned to raise.
- **OAK-M34 (Pause-by-Default / Emergency Pause):** Bonzo paused its lending pool 50 minutes after the forged price landed; the pause is what stopped it from being worse, not what prevented it.

**Canonical mitigations:** OAK-M16, OAK-M09, OAK-M11.

## Discussion

OAK mints T9.015 because the Bonzo Lend case had no honest home. Mapping it to T9.001 describes the *outcome* — an inflated price consumed as collateral — while implying an oracle-resilience mitigation (TWAP, multi-venue, deviation breakers) that addresses a different premise: those controls assume the attacker moved a real market. Nobody moved SAUCE. The value was asserted and the verifier vouched for it, so the entire oracle-design mitigation family only bounds the damage. Mapping instead to T10.002 captures the mechanism honestly — a verifier admitted a message it should have rejected — but T10.002 sits under T10 (Bridge / Cross-Chain), and its worked examples are uniformly application-layer logic defects at a bridge boundary. Bonzo was neither a bridge nor an application-layer logic defect. The distinction OAK wants to preserve is **application-layer check missing (T10.002) versus cryptographic primitive returning true on degenerate input (T9.015)**, because the two have different owners, different audit scopes, and different fixes.

The reason to expect recurrence rather than treat this as a curiosity is that the hazard is structural, not novel. Subgroup-membership and identity-element validation have been explicit requirements in BLS standardisation for years, and invalid-curve and small-subgroup attacks are old results in the wider cryptographic-engineering literature. What is new is the deployment surface: on-chain verifiers for pairing-based oracle attestations, threshold committees, and proof systems have proliferated fast, they are frequently third-party dependencies whose internals the consuming protocol never reads, and the gap between "the paper says validate your inputs" and "the shipped verifier validates its inputs" is where this class lives. Supra's verifier had presumably been reviewed as a component; Bonzo, the party that actually lost the money, had no practical way to see the defect and no control that would have caught it. That asymmetry — **the protocol that bears the loss is not the protocol that shipped the bug** — is the durable teaching point and the reason the mitigation set leans on audit scope for third-party verifier dependencies and on consumer-side plausibility bounds as the compensating control.

The class is `emerging` with a single worked anchor. It should be promoted to `stable` once a second independent on-chain anchor lands, and contributors encountering *any* incident where a verification passed on zero/identity/off-subgroup input — in an oracle, a bridge, a threshold committee, or a proof verifier — should map it here and say which primitive and which omitted check were load-bearing. A candidate sub-shape worth watching, but not yet minted: **degenerate inputs to zero-knowledge proof verifiers** (identity-element or off-subgroup elements in a Groth16/PLONK verifier's proof or verifying key), which is the same defect class against a different primitive and would justify a sub-Technique once anchored.
