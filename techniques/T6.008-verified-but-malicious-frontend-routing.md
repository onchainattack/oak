# OAK-T6.008 — Verified-but-Malicious Frontend Routing

**Parent Tactics:** OAK-T6 (Defense Evasion)
**Maturity:** emerging
**Chains:** EVM (primary); cross-chain DEX-router analogues (Solana Jupiter routing-manipulation, cross-chain bridge routing-path insertion) are pipeline anchor candidates at v0.x
**First documented:** ~2024 (SwapKit router impersonation cohort; class-level visibility accelerated through 2025 with Uniswap routing-manipulation incidents)
**Aliases:** "malicious routing frontend", "helper-contract injection", "router-impersonator phishing", "verified-contract routing attack", "intermediate-hop extraction"

## Description

Phishing UIs that **bind to real, audited, verified smart contracts** at the user's wallet confirmation screen but route the transaction through attacker-owned helper contracts that extract value via routing logic invisible to the user at sign-time. The user inspects the wallet confirmation screen, sees the legitimate protocol's verified contract address (e.g., the Uniswap Universal Router), and approves the transaction. The extraction occurs in the routing path — swap-path manipulation, fee-injection intermediate contracts, malicious hop contracts inserted between the user's approval and the protocol's settlement.

T6.008 is **structurally distinct** from sibling Defense Evasion Techniques because the *evasion surface is in the routing path, not in the contract address the user inspects*:

- **T4.002 (Compromised Front-End Permit Solicitation)** — the attacker compromises a real platform's frontend. T6.008's frontend may **not be compromised** — the malicious routing is the load-bearing primitive, and the frontend may be an attacker-operated UI that happens to interact with real protocol contracts.
- **T4.008 (Fake-DEX Clone-Frontend Phishing)** — the attacker deploys a counterfeit UI clone that connects to a drainer backend. T6.008's UI interacts with **real protocol contracts** (e.g., the actual deployed Uniswap router); the extraction happens in the intermediate routing logic, not in a drainer contract the user directly authorises.
- **T6.001 (Source-Verification Mismatch)** — the attacker deploys unverified or mismatched bytecode. T6.008's protocol contracts are genuinely verified and audited — the malicious code is in the **helper contracts that sit in the routing path**, which the user never inspects.
- **T6.003 (Audit of Different Bytecode Version)** — the audit covers different bytecode. T6.008's audited contracts match their audited bytecode; the unaudited helper contracts in the routing path are structurally outside the audit scope.
- **T6.005 (Proxy-Upgrade Malicious Switching)** — the attack vector is a proxy upgrade. T6.008 requires no upgrade — the helper contracts are deployed independently and the routing path is constructed at transaction-construction time.

The novel structural feature is **routing-path extraction** — the extraction is invisible at the wallet confirmation screen because the user's transaction is addressed to a legitimate protocol contract. The user's mental model is "I am swapping via Uniswap" and the wallet confirmation screen confirms the Uniswap router address. The routing path that extracts value is embedded in the transaction's calldata — a structure most wallet UIs do not surface to the user at sign-time.

Two distinguishable sub-patterns:

- **T6.008.001 — Swap-path manipulation.** The attacker constructs a swap route that passes through an attacker-owned intermediate contract (e.g., an attacker-deployed liquidity pool, a fee-injection contract, or a malicious aggregator hop) inserted between the user's input token and the output token. The user signs a transaction to the real Uniswap / Jupiter / 1inch router; the swap path constructed by the attacker's frontend routes through the malicious intermediate contract. The legitimate router executes the swap path as constructed; the extraction occurs at the intermediate contract.
- **T6.008.002 — Fee-injection helper-contract routing.** The attacker deploys a helper contract that wraps the legitimate router call, injecting an additional fee-transfer or output-redirect step before or after the legitimate swap. The user signs a transaction to the helper contract (not the router directly), but the helper contract's verified source or its external-call surface is not inspected by the user. The wallet confirmation screen may show the helper contract's address; the user does not recognise it as malicious because the transaction simulation shows the expected swap outcome (the fee extraction may be hidden in a secondary token transfer or a slippage-adjusted output).

## Observed indicators

- **Transaction-construction fingerprints:** the transaction `to` address is a legitimate, verified, audited protocol contract (Uniswap Universal Router, Jupiter aggregator, 1inch router). The malicious routing path is embedded in the calldata, not in the `to` address.
- **Routing-path fingerprints:** the swap path includes an intermediate contract address that is (a) not a known liquidity pool from a major DEX, (b) recently deployed, (c) not verified on the chain explorer, or (d) not listed in the protocol's canonical pool registry.
- **Frontend-side fingerprints:** the frontend's domain is a typosquat / near-miss of the legitimate protocol's domain, but the frontend's swap-path construction differs from the legitimate frontend's path-construction for the same input/output pair.
- **Wallet-confirmation-screen fingerprints:** the wallet confirmation screen shows the legitimate protocol router address; the user's mental model is satisfied. The calldata — which defines the routing path — is typically truncated or not human-readable in standard wallet UIs.
- **Output-token fingerprints:** the user receives fewer output tokens than the legitimate frontend's quote for the same input/output pair at the same block; the difference is extracted in the intermediate routing path.

## Detection signals

- **Wallet-side routing-path transparency (highest-leverage):** wallet UIs should surface the full routing path to the user at sign-time, including all intermediate contract addresses, fee percentages, and output-token estimates at each hop. MetaMask / Rabby / Phantom / OKX Wallet pre-trade simulation should flag routing paths that include unverified or recently-deployed intermediate contracts.
- **Swap-path comparison:** defender-side tooling should compare the swap path constructed by the frontend against the canonical routing path the legitimate protocol's API returns for the same input/output pair. Path divergence — particularly the presence of intermediate contracts not in the protocol's known-pool registry — is a high-confidence detection signal.
- **Domain-reputation services:** ScamSniffer / Web3 Antivirus / SEAL ISAC should flag domains hosting frontends whose swap-path construction includes unverified intermediate contracts, even when the transaction's `to` address is a legitimate protocol contract.
- **Protocol-side canonical-router verification:** protocols should publish a canonical list of router contract addresses, pool addresses, and allowed intermediate-hop contracts. Wallets and simulation tools should verify that the routing path's intermediate contracts are all members of the canonical set.
- **Calldata-decoding simulation:** pre-trade simulation engines should decode the full calldata of router transactions and surface the routing path in human-readable form, including all intermediate hop addresses and fee parameters.

## Real-world examples

- [`examples/2024-2025-swapkit-router-impersonator-phishing.md`](../examples/2024-2025-swapkit-router-impersonator-phishing.md) — SwapKit router impersonator phishing cohort — 2024–2025
- **SwapKit router impersonator cohort — EVM — 2024–2025 (canonical T6.008 cohort)** — Attacker-operated frontends presented a Uniswap-alike swap interface that constructed transactions addressed to the legitimate Uniswap Universal Router. The swap path embedded in the calldata routed through attacker-owned intermediate contracts that extracted fees or redirected output tokens to attacker-controlled addresses before the remaining output reached the user's wallet. The user's wallet confirmation screen showed the verified Uniswap router address; the extraction was invisible at the confirmation step. The SwapKit brand was impersonated as the aggregation / routing layer — the attacker claimed the frontend was "powered by SwapKit" to explain the intermediate routing path, but the intermediate contracts were attacker-owned. Attribution: `pseudonymous` at cohort scale; per-victim reporting via ZachXBT, ScamSniffer, and SEAL Threat Intel. Per-incident worked-example anchors still forming at v0.1.
- **Uniswap routing-manipulation frontend cohort — EVM — 2025** — Multiple attacker-operated frontends claiming to be "Uniswap interface" or "Uniswap aggregator" constructed swap paths through attacker-deployed intermediate liquidity pools. The user signed a transaction to the verified Uniswap Universal Router; the swap path routed through the attacker's pool, which extracted a spread or fee before forwarding the swap to legitimate Uniswap V3 pools. The legitimate router executed the path as constructed — the router itself was not malicious; the path was. Attribution: `pseudonymous`; operator-cluster attribution partial at v0.1.
- **Cross-chain bridge routing-path injection (pipeline anchor at v0.x)** — Structurally analogous cases on cross-chain bridge frontends where the attacker's UI interacts with real bridge contracts but routes the user's deposit through an intermediate contract that extracts a fee or redirects the output chain / address. Not yet anchored with a per-incident worked example at v0.1; held as a pipeline anchor candidate for v0.x.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T6.008's primary detection surface is at the wallet-side routing-path transparency / calldata-decoding simulation layer, not the on-chain runtime risk-engine surface MG covers. MG's authority-grant detection module may flag the transaction's approval of the legitimate router, but the routing-path extraction primitive is embedded in the calldata and is outside MG's current detection scope.
- Vendor coverage outside MG: wallet-side pre-trade simulation routing-path decoding (MetaMask, Rabby, Phantom, OKX Wallet — partial coverage at v0.1; calldata-decoding for router transactions is not uniformly implemented); ScamSniffer domain-reputation feeds; SEAL ISAC phishing-domain blocklists; protocol-side canonical-router-address registries (Uniswap, 1inch, Jupiter — partial coverage).

## Mitigations

- **Wallet-side routing-path transparency (highest-leverage):** wallet UIs should decode and surface the full routing path for router / aggregator transactions, including all intermediate contract addresses, fee parameters, and estimated output at each hop. The routing path should be surfaced with the same prominence as the `to` address.
- **Wallet-side intermediate-contract verification:** wallet UIs should flag routing paths that include intermediate contracts not present in the protocol's canonical pool / hop registry, with a high-friction warning before the user signs.
- **Protocol-side canonical-router-and-hop-registry publication:** protocols operating router / aggregator contracts should publish and maintain a canonical list of allowed intermediate-hop contract addresses. Wallets and simulation tools should ingest and verify against this list.
- **Pre-trade swap-path comparison:** simulation engines should compare the constructed swap path against the legitimate protocol API's returned path for the same input/output pair at the current block. Divergence should trigger a high-friction warning.
- **Domain-reputation integration:** ScamSniffer / SEAL ISAC should ingest routing-path-manipulation domains as a discrete phishing sub-class, keyed on frontends whose constructed swap paths include unverified intermediate contracts.
- **User-side hygiene:** users should verify the swap path (not just the `to` address) when signing router transactions; prefer the legitimate protocol's canonical frontend URL typed directly rather than clicking search results or social-media links.

OAK Mitigation cross-references at v0.1 are a **gap** — no existing OAK-M* entry directly maps to wallet-side routing-path transparency or protocol-side canonical-hop-registry publication. v0.x candidate Mitigation entries: "OAK-M-x wallet-side routing-path decoding and display"; "OAK-M-x protocol-side canonical hop-registry publication"; "OAK-M-x pre-trade swap-path comparison".

## Discussion

T6.008 addresses a structurally novel Defense Evasion surface that no existing T6 Technique covers. Existing T6 Techniques all share the assumption that the *contract the user inspects* is the locus of deception: T6.001 (source-verification mismatch), T6.002 (fake audit claim), T6.003 (audit of different bytecode), T6.004 (audit-pending marketing claim), T6.005 (proxy-upgrade switching), T6.006 (counterfeit token impersonation), T6.007 (vendor-policy revocation). T6.008 shifts the deception locus to the **routing path** — the contract the user inspects is real, verified, and audited; the extraction occurs in the calldata the wallet UI does not surface.

The structural OAK lesson is that **wallet confirmation screens that display only the `to` address and value are inadequate for router / aggregator transactions**. The user's security model is "verify the contract address" — a model that T6.008 defeats because the contract address is correct. The defender's model must extend to "verify the full routing path, including all intermediate contracts, fee parameters, and output estimates at each hop."

The technique is especially dangerous because it exploits the **gap between the wallet's confirmation-surface fidelity and the transaction's execution-surface complexity**. The wallet shows the user a simple "you are interacting with Uniswap" confirmation; the transaction executes a multi-hop swap path through attacker-owned contracts. The user's mental model and the transaction's execution model are structurally disjoint — the routing path is the gap.

The class is included at v0.1 with `emerging` maturity rather than `stable` because (a) the per-incident anchor cohort is still forming — the SwapKit cohort provides the class-level calibration anchor but individual incident anchors are still being published, (b) wallet-side routing-path decoding is not uniformly implemented across major wallet surfaces, and (c) protocol-side canonical hop registries are not uniformly published. The `emerging` classification reflects that the threat-class characterisation is mature and the operational-cohort anchor is forming; `stable` will follow once per-incident worked-example anchors land and wallet-side routing-path transparency reaches material coverage across the wallet surface.

The class structurally generalises beyond DEX routers — any protocol where the user signs a transaction to a legitimate contract but the transaction's calldata defines a routing path (bridge contracts, aggregator contracts, yield-aggregator vaults, cross-chain message-passing contracts) has the same routing-path extraction surface. The structural condition is: (1) the user signs a transaction to a verified contract, (2) the transaction's calldata defines a routing path through intermediate contracts, (3) the wallet UI does not surface the full routing path to the user at sign-time. These conditions are chain-agnostic and protocol-agnostic.
