# OAK-T12.004 — Timelock-Free Protocol Upgrade Execution

**Parent Tactics:** OAK-T12 (Governance / Protocol Management)
**Maturity:** emerging
**Chains:** EVM (primary); any chain with proxy-upgrade patterns and governance-controlled upgrade authority
**First documented:** 2020–2021 (proxy-upgrade pattern proliferation; timelock-bypass incidents characterized in audit literature)
**Aliases:** "instant upgrade attack", "timelockless governance", "governance-timelock bypass", "immediate proxy upgrade", "unguarded upgrade authority"

## Description

The governance design anti-pattern in which a protocol's upgrade authority (governance multisig, DAO executor contract, or proxy admin) can execute a protocol upgrade — proxy implementation change, parameter modification, or treasury transfer — without a mandatory timelock delay between governance approval and on-chain execution. The absence of a timelock eliminates the window during which governance participants and protocol users can observe an approved upgrade, verify its correctness, and exit their positions before execution.

T12.004 is structurally distinct from T12.002 (Governance Multisig Compromise) in a load-bearing way: T12.002 covers the *compromise of the multisig itself* (key exfiltration, social engineering, insider collusion), while T12.004 covers the *governance design decision* that determines the blast radius of any governance compromise. An attacker who compromises the upgrade authority of a timelock-protected protocol gains the ability to *queue* an upgrade that will execute after the timelock delay — users have the duration of the timelock to observe the queued upgrade, verify its correctness, and exit their positions. An attacker who compromises the upgrade authority of a timelock-free protocol gains the ability to *execute* the upgrade instantly — converting a governance-compromise event into an immediate full-protocol extraction with no user exit window.

The timelock is the load-bearing defense-in-depth primitive at the governance layer:
- It does **not** prevent governance capture — an attacker with sufficient governance power can still queue a malicious upgrade.
- It **does** limit the blast radius by giving users a time window (typically 24–72 hours) to detect the queued upgrade and exit before execution.
- It **transforms** a governance-compromise event from an instantaneous extraction into a time-delayed extraction with a mandatory user-observation window — a categorically different risk profile.

T12.004 covers four structurally adjacent sub-shapes:

- **Proxy-admin instant upgrade.** The proxy admin (an EOA, a multisig, or a DAO executor) can call `upgradeTo` on the proxy in the same transaction as governance approval — no intermediate timelock contract enforces a delay between proposal passage and implementation upgrade.
- **Governance-executor direct execution.** The governance executor contract (the on-chain contract that executes approved governance proposals) can call arbitrary external functions without routing through a timelock — the governance proposal approval and the proposal execution occur atomically or in the same block.
- **Multisig-direct upgrade authority.** The governance multisig is itself the proxy admin, and threshold approval authorizes an immediate upgrade without a separate timelock contract interposed between the multisig and the proxy. The multisig's `executeTransaction` or `execTransaction` path directly calls the proxy's `upgradeTo` function.
- **Governance-overrideable timelock.** The timelock contract's minimum delay is configurable by governance and can be reduced to zero (or a trivially short window) by the same governance that the timelock is meant to constrain — the timelock is present but not load-bearing because governance can override it.

## Observed indicators

- Proxy admin or upgrade-authority contract has no mandatory delay between proposal approval and execution — the upgrade path from governance approval to `upgradeTo` call contains no timelock contract.
- Governance executor can call `upgradeTo` (or equivalent proxy-upgrade function) on the proxy in the same transaction as governance approval — the `ProposalExecuted` event and the `Upgraded` event fire in the same block.
- Multisig threshold can authorize an immediate upgrade without a separate timelock contract — the multisig's authorization path (e.g., Gnosis Safe `execTransaction`) directly targets the proxy's upgrade function.
- Upgrade events fire in the same block as the governance proposal execution event — on-chain monitoring can detect same-block proposal-execution + upgrade-event pairs.
- Timelock contract's minimum delay is governance-configurable and has been reduced below the protocol's stated timelock window — the timelock exists but is not load-bearing.

## Detection signals

- **Pre-deployment governance audit.** Verify that the upgrade path enforces a mandatory timelock: minimum 24–48 hours for parameter changes, 48–72 hours for implementation upgrades. Confirm that the timelock contract is interposed between the governance approval layer and the proxy upgrade function, and that the governance cannot bypass or override the timelock.
- **On-chain same-block upgrade monitoring.** Alert when an `Upgraded` event (proxy implementation change) fires in the same block as a `ProposalExecuted` event (governance proposal execution) — the canonical T12.004 signature at the detection layer.
- **Governance-contract static analysis.** Verify that the executor contract's code routes all state-changing operations through the timelock; confirm that the timelock's minimum delay is hard-coded or governed by a separate, higher-threshold governance process than the timelock itself.
- **Timelock-parameter drift monitoring.** Monitor the timelock contract's minimum-delay parameter over time; alert on reductions that bring the delay below the protocol's documented governance-security window.

## Real-world examples

- [`examples/2020-2021-defi-timelock-free-upgrade-cohort.md`](../examples/2020-2021-defi-timelock-free-upgrade-cohort.md) — DeFi "move fast" era timelock-free protocol upgrade cohort — 2020–2021
- **Compound Timelock (canonical positive example).** Compound's GovernorBravo + Timelock architecture (deployed 2020) established the 48-hour governance timelock as the DeFi standard: governance proposals that pass are queued in the Timelock contract and execute after a minimum 48-hour delay. Users have a 48-hour window to observe queued proposals, verify their correctness, and exit positions before execution. This architecture became the canonical reference implementation for governance-timelock integration — OpenZeppelin's TimelockController, Tally's governance tooling, and most GovernorBravo-based DAOs inherit this pattern. Source: Compound Governance documentation; `[compoundtimelock2020]`.
- **Multiple DeFi protocols deployed without governance timelocks — 2020–2021 ("move fast" era).** During the 2020–2021 DeFi expansion, numerous protocols launched with governance multisigs as direct proxy admins or with governance executors that could call `upgradeTo` without a timelock delay. This was the standard "move fast" deployment pattern before the Compound Timelock became the accepted industry standard. Audit firms (Trail of Bits, OpenZeppelin, ConsenSys Diligence) began flagging timelock-free upgrade paths as critical findings in this period — the audit-firm literature from 2020–2021 is the primary public record for the T12.004 class.
- **OpenZeppelin TimelockController introduction.** OpenZeppelin Contracts 4.0 (released 2021) introduced the `TimelockController` contract as a standardized access-control primitive for timelocked governance execution. The introduction of a standard library contract for timelocked governance is indirect evidence of the pre-existing gap: the standard was created because protocols were observed deploying without timelocks. Source: `[openzeppelintimelock2021]`.

## Mitigations

- **Protocols — OAK-M11 (Governance Timelock).** All governance upgrades must pass through a mandatory timelock with a configurable minimum delay. The timelock contract must be interposed between the governance approval layer and the proxy upgrade function. The minimum delay should be: 24–48 hours for parameter changes (oracle updates, fee adjustments, collateral factor changes); 48–72 hours for implementation upgrades (proxy implementation changes, contract migrations).
- **Protocols — timelock governance immutability.** The timelock's minimum delay should be governed by a separate, higher-threshold governance process than the timelock itself (e.g., a higher multisig threshold, a separate DAO vote with a longer voting period, or a hard-coded minimum that requires a contract migration to change). A timelock whose minimum delay can be reduced to zero by the same governance that the timelock constrains is not load-bearing.
- **Protocols — deploy upgrade authority behind a TimelockController or equivalent.** For new deployments, use OpenZeppelin's TimelockController (or an audited equivalent) as the standard upgrade-authority pattern. The TimelockController should be the proxy admin, not the governance multisig directly.
- **Users / integrators — monitor the governance timelock queue.** Subscribe to `CallScheduled` / `CallExecuted` events from the timelock contract; on observing a queued upgrade that is malicious or unexpected, exit positions before the timelock delay expires. The timelock-protocol social contract is that users have the timelock window to exit — users must actively monitor the timelock queue to exercise this right.
- **Audit firms — flag timelock-free upgrade paths as critical findings.** The absence of a governance timelock should be treated as a critical-severity finding in any protocol audit, regardless of the multisig's reputation or the governance process's perceived security. The blast-radius difference between timelocked and timelock-free governance compromise is categorical.

## Discussion

T12.004 is the governance-design counterpart to T12.002 (Governance Multisig Compromise): T12.002 is the *incident* class (the multisig is compromised), and T12.004 is the *design-decision* class (the protocol's governance design determines the blast radius of any compromise). A protocol that deploys with a Compound-style 48-hour timelock is not immune to governance compromise, but a compromise event is a time-delayed extraction with a mandatory user-observation window — a categorically different risk profile from an instant extraction.

The structural relationship between T12.004 and T16.x (Governance Attack) techniques is an amplification relationship: T12.004 amplifies the extraction payoff of any T16.x attack (malicious proposal, governance capture, vote-buying) by eliminating the user-exit window. A malicious proposal that passes governance (T16.005) on a timelock-protected protocol gives users 48 hours to detect and exit; the same proposal on a timelock-free protocol executes instantly, with no user recourse.

The Compound Timelock's 48-hour standard has become the DeFi governance norm, but the norm is a social convention, not an on-chain guarantee. New protocols, particularly those launched in "fast-follow" mode or by teams that prioritize deployment speed over governance security, may deploy without a timelock. The audit-firm practice of flagging timelock-free upgrade paths as critical findings has created a market incentive for timelock adoption, but the incentive operates at the audit layer, not the deployment layer — unaudited protocols may still deploy without timelocks.

False-positive considerations: a same-block `ProposalExecuted` + `Upgraded` event pair is a high-signal T12.004 indicator, but not all same-block governance-execution + contract-upgrade event pairs are malicious — some governance processes are designed for same-block execution (e.g., emergency DAO operations, pre-announced scheduled upgrades with a social-timelock rather than an on-chain timelock). The detection signal should be interpreted in the context of the protocol's documented governance design: a same-block execution on a protocol whose documentation claims a 48-hour timelock is a strong T12.004 signal; a same-block execution on a protocol whose documentation explicitly states that governance executes without a timelock is a design-choice disclosure, not a detection event.
