# OAK-T11.013 — Legacy-Version Maintenance Attack Surface

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure); structurally adjacent to OAK-T9.007 (Fork-Substrate Vulnerability — both involve unpatched known vulnerabilities in deployed code, but the operational context differs: T9.007 covers downstream forks of a different protocol; T11.013 covers the same operator's own deprecated version)
**Maturity:** emerging
**Chains:** EVM (primary); structurally generalisable to any chain where protocol operators maintain multiple deployed versions
**First documented:** 2023-02 (Yearn V1 — structurally related but with a configuration-rot dimension); 2025-07 (GMX V1 — cleaner instance of the pure legacy-version-maintenance decision class)
**Aliases:** "deprecated-version attack surface", "legacy-version residual vulnerability", "multi-version rollout security gap", "deprecation-without-decommission"

## Description

An operator-side operational decision class in which a protocol operator launches a new protocol version (V2, V3) but **retains the deprecated legacy version (V1) in production** — accepting deposits, maintaining TVL, processing user transactions — without applying known vulnerability fixes to the legacy version. The operator's decision calculus is typically: decommissioning V1 would force LP migration (user friction), fixing V1 would consume engineering resources (opportunity cost), and maintaining V1 unfixed accepts residual exploitable surface (risk acceptance). When the residual vulnerability is exploited, the loss is a direct consequence of the **deprecation-without-decommission** decision.

The structural feature is not the vulnerability class (which may be any T9.x subclass — oracle manipulation, accounting flaw, reentrancy) but the **operator's operational decision to maintain a known-vulnerable deprecated version in production**. The vulnerability was known to the operator (documented in internal communications, disclosed post-incident); the operator made a reasoned decision to accept the risk; the exploit was the materialisation of that accepted risk.

Distinct from sibling Techniques:

- **T9.007 (Fork-Substrate Vulnerability Not Mitigated)** — Protocol B forks Protocol A and fails to track upstream patches. T11.013 involves the **same operator's own legacy version**, not a different operator's fork. The operational gap in T9.007 is between two different teams; in T11.013 the gap is within one team's version-management discipline.
- **T9.004 (Access Control Misconfiguration)** — the standing vulnerability class. T11.013's vulnerability may be of any T9 class; the structural feature is the deprecation-without-decommission decision, not the vulnerability class.
- **T6.007 (Trust-substrate Shift / Vendor-side Promise Revocation)** — the operator changes a policy promise. T11.013 involves no policy change; the operator's deprecation decision was already made and disclosed; the residual risk was accepted but not eliminated.
- **T11.005 (Operator-Side Fake-Platform Fraud)** — the operator is fraudulent. T11.013's operator is genuine; the failure is in version-lifecycle risk management, not in platform authenticity.

## Observed indicators

- **Multi-version protocol deployment:** the protocol operates two or more deployed versions simultaneously (V1 and V2), with V1 explicitly deprecated but still accepting deposits and maintaining material TVL.
- **Known-vulnerability documentation gap:** the operator's internal communications or public post-mortem acknowledge that the vulnerability class exploited on V1 was known to the operator team before the incident — the deprecation-vs-fix tradeoff was a conscious decision.
- **Version-specific exploit targeting:** the attacker specifically targets V1's accounting, pricing, or state-machine logic — logic that was redesigned or hardened in V2, confirming that the vulnerability was version-specific and known to be version-specific.
- **Operator's post-incident acknowledgment:** the operator's post-mortem frames the incident as a consequence of the decision to maintain V1 in production rather than decommission it — the acknowledgment confirms the T11.013 structural diagnosis.
- **No V2 impact:** the exploit is confined to V1; V2 continues to operate normally. The version-isolation of the exploit confirms that the vulnerability was legacy-version-specific, not protocol-wide.

## Detection signals

- **Pre-incident: multi-version deployment audit (highest-leverage).** For every protocol operating multiple deployed versions, audit the legacy version for known vulnerabilities that have been fixed in the current version but remain unpatched in the legacy deployment. The audit surface is the delta between V2's security posture and V1's security posture.
- **Pre-incident: legacy-version TVL monitoring.** Monitor the TVL retained in deprecated protocol versions. Legacy versions with material TVL (>5% of protocol TVL) and known vulnerability deltas from the current version are T11.013 exposure signals.
- **Pre-incident: operator-side deprecation-with-decommission discipline verification.** Auditors and due-diligence providers should verify whether the operator has a documented legacy-version decommissioning timeline and whether known vulnerabilities in the legacy version are patched or the version is fully decommissioned (deposits disabled, migration path enforced).
- **At-incident: version-specific exploit pattern recognition.** When an exploit targets a legacy version's accounting/pricing logic that was redesigned in the current version, the incident is a high-confidence T11.013 event — the attacker explicitly targeted the known delta between versions.

## Real-world examples

- **GMX V1 GLP global-short-tracking exploit — Arbitrum — 2025-07-09 (canonical T11.013 anchor).** GMX V2 launched in late 2023 with redesigned accounting and pricing logic. V1 continued to operate with substantial residual TVL through 2024–2025 because of LP migration friction. The V1 GLP pool used a global-short-tracking accounting structure known to the operator team to have a manipulable surface via crafted swap-and-stake transactions. The operator made a reasoned deprecation-vs-fix decision to accept the residual risk. On 2025-07-09, an attacker exploited the V1-specific accounting flaw, extracting ~$42M. Near-100% recovery via 10% white-hat bounty. See [`examples/2025-07-gmx-v1.md`](../examples/2025-07-gmx-v1.md).

### Additional incidents (pipeline anchors)

- **Yearn V1 — February 2023.** Yearn V1 had been superseded by V2 for an extended period but V1 vaults retained residual TVL. A V1-specific configuration-rot vulnerability was exploited. The case has a configuration-rot dimension that complicates the pure legacy-version-maintenance classification; structurally related but with an additional contributing factor. Pipeline anchor.

## Mitigations

- **Legacy-version decommissioning discipline (highest-leverage).** Protocol operators launching a new version should publish and adhere to a legacy-version decommissioning timeline: disable deposits on the legacy version within a defined window, enforce LP migration to the new version, and fully decommission (disable all state-mutating functions) the legacy deployment once migration is complete. A legacy version that cannot accept new deposits cannot accumulate new TVL-at-risk.
- **Legacy-version vulnerability backporting.** If a legacy version must be maintained in production (e.g., because of irreversible LP lockups or protocol-political constraints), known vulnerabilities fixed in the current version should be backported to the legacy version. The backporting decision should be documented, audited, and verified on-chain via contract upgrade or parameter change.
- **Multi-version security-delta audit.** Pre-deployment and post-deployment security audits should include a specific "version-delta" section: for every known vulnerability fixed in the new version, verify whether the legacy version remains exposed and assess the residual risk.
- **Legacy-version TVL monitoring and risk disclosure.** Protocols maintaining multiple versions should publicly disclose the TVL retained in each version and the known vulnerability delta between versions. Users depositing into a deprecated version should be informed that they are accepting residual vulnerability risk.

## Discussion

T11.013 occupies a structurally distinct slot at the boundary between OAK-T11 (Custody and Signing Infrastructure — operator-side operational decisions) and OAK-T9 (Smart-Contract Exploit — the vulnerability class exploited). The technique is classified under T11 rather than T9 because the **load-bearing primitive is the operator's operational decision** (deprecation-without-decommission), not the smart-contract vulnerability class (which may be any T9.x subclass and varies across incidents). The same operator, managing two versions of the same protocol, chose to accept residual exploitable surface on the legacy version; the exploit was the materialisation of that accepted risk.

The GMX V1 July 2025 case is the canonical anchor because it is the cleanest instance of the pure legacy-version-maintenance decision class: the vulnerability was known to the operator, the deprecation-vs-fix tradeoff was a reasoned and documented decision, the vulnerability was version-specific (V1's GLP global-short-tracking accounting was redesigned out of V2), and the post-incident operator post-mortem explicitly frames the incident as a consequence of the legacy-version-maintenance decision. The near-100% recovery via white-hat bounty does not diminish the structural significance of the case — the loss was realised before recovery, and the operational decision class remains load-bearing regardless of recovery outcome.

The technique is structurally adjacent to T9.007 (Fork-Substrate Vulnerability). Both involve deployed code with known-but-unpatched vulnerabilities, and both involve an operational gap between vulnerability disclosure and patch deployment. The distinction is organisational: T9.007's gap is between two different teams (Protocol A's security team discloses; Protocol B's operator fails to track); T11.013's gap is within one team's version-lifecycle management (the same team that hardened V2 chose not to backport to V1). The distinction matters for mitigation: T9.007 is mitigated by downstream fork operators monitoring upstream disclosures; T11.013 is mitigated by the operator's own legacy-version decommissioning discipline.

Promotion to `observed` requires at least one additional independent worked-example anchor beyond GMX V1. The pipeline is thin because multi-version protocol deployments with material legacy-version TVL are a relatively recent pattern (2023–2025) and because operators who decommission legacy versions eliminate the T11.013 surface before an incident occurs. Yearn V1 (February 2023) is a structurally related pipeline anchor with a configuration-rot dimension; additional clean instances are expected as the DeFi protocol multi-version deployment pattern matures.
