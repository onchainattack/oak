# OAK-T13.001.001 — Paymaster Accounting Drain

**Parent Tactics:** OAK-T13 (Account Abstraction Attacks)
**Maturity:** observed
**Chains:** EVM (ERC-4337 EntryPoint v0.6 / v0.7 / v0.8 / v0.9 deployments); EVM L2s with native account abstraction (zkSync Era, Starknet) where the paymaster role is analogous
**First documented:** 2025 (OSEC paymaster security review enumerates the `postOp` revert / gas-token-mechanic surfaces explicitly); class characterised continuously through audit-firm advisories on deployed paymasters
**Aliases:** "paymaster drain", "postOp revert drain", "gas-token-mechanic accounting drain", "validation-time-debit not unwound"

## Description

The attacker submits UserOps that abuse the paymaster's accounting model to extract sponsored funds beyond what the paymaster intended. The canonical worked sub-pattern: a paymaster pays the EntryPoint from a shared pool during `validatePaymasterUserOp` and attempts to charge the individual user back in `postOp`; if `postOp` reverts (bad state, arithmetic overflow, risky external call, fee-on-transfer or rebasing ERC-20 used as gas token), the validation-time payment is **not** unwound and the paymaster has subsidised a UserOp it did not intend to subsidise `[osecpaymasters2025]`.

The failure mode is that the paymaster's `postOp` is the only path that can charge the user back, but `postOp` can revert for reasons that the paymaster's author did not enumerate at design time — particularly when the gas-token mechanic includes fee-on-transfer (T1.005), rebasing, or pausable / blacklist (T1.004) surfaces. Each revert-path that goes un-enumerated is an accounting-drain surface.

T13.001.001 is part of the OAK-T13.001 (Paymaster Compromise) umbrella; see the parent for the architectural distinction from T9.

## Observed indicators

- A paymaster whose `postOp` can revert (arithmetic, risky external call, fee-on-transfer or rebasing ERC-20 gas token, pausable / blacklist token) without unwinding the validation-time deposit-balance debit.
- A paymaster that supports ERC-20-denominated gas payment without an explicit allowlist of audited gas tokens.
- Drift between the paymaster's deposited balance at the EntryPoint and its accounted-for sponsorship volume — the deposited balance trends downward faster than the accounting model predicts.
- A paymaster contract whose `postOp` arithmetic does not use checked operations (or whose `unchecked` blocks span the user-charge path).

## Detection signals

- **Pre-deployment paymaster contract audit.** Engagement with audit firms with explicit ERC-4337 specialist practice (OpenZeppelin, Trail of Bits, Quantstamp, OSEC / OtterSec, Halborn, ConsenSys Diligence). A paymaster that has not been audited by a 4337-specialist firm is a T13.001.001 surface independently of the underlying EntryPoint version.
- **`postOp` revert-safety review.** Enumerate every revert path in `postOp` (arithmetic, external-call, ERC-20 token-mechanic — fee-on-transfer, rebasing, blacklist / pausable T1.004 surfaces); verify that no revert path allows a validation-time debit to remain un-unwound.
- **Gas-token allowlist verification.** Audit the allowlist of ERC-20 tokens accepted as gas-payment denomination; reject tokens with fee-on-transfer (T1.005), rebasing, pausable, or blacklist mechanics unless the paymaster implements explicit accounting that handles them.
- **Paymaster-fund-balance monitoring.** Runtime metric: paymaster's deposited balance at the EntryPoint, sampled per block, with alerts on drawdown rate exceeding the paymaster's expected sponsorship-throughput envelope. Persistent accounting drain produces a drawdown signature that is statistically distinguishable from organic sponsorship activity.
- **Per-UserOp policy-rule audit.** For high-value paymasters, log per-sponsored UserOp the gas-token used and the `postOp` outcome (success / revert path); reconcile against the paymaster's deposited-balance trajectory; mismatches indicate either accounting-drain (T13.001.001) or policy-bypass (T13.001.002).

## Real-world examples

- [`examples/2025-04-erc4337-paymaster.md`](../examples/2025-04-erc4337-paymaster.md) — ERC-4337 paymaster compromise cohort, April 2025 anchor (~$500K+ drain). The anchor incident centred on a sub-case (a)/(b) hybrid where the `postOp` charge-back path failed to recover the validation-time deposit-balance debit against an attacker-drained smart account — the canonical T13.001.001 drawdown signature.
- **OSEC 2025 paymaster review — class characterisation — 2025** — OSEC / OtterSec's 2025 paymaster security review enumerates the `postOp` revert and gas-token-mechanic accounting-drain surfaces with concrete worked patterns. The advisory is the canonical reference for the T13.001.001 sub-case at v0.1; the surface is well-characterised in audit-firm literature even though no single named large-loss incident has been catalogued in public forensic write-ups `[osecpaymasters2025]`.
- **Cohort framing — 2023 onward** — OAK does not at v0.1 name a single canonical large-loss T13.001.001 incident comparable to T9.001 or T10.001; publicly-disclosed paymaster-accounting-drain flaws have to date been caught at the audit-firm or bug-bounty layer rather than via headline exploits. Contributors with public forensic write-ups of specific named incidents whose primary failure mode is paymaster accounting drain are encouraged to submit them via PR.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1 (paymaster-specific accounting monitoring is out of MG operator-behaviour scope).
- Pre-deployment vendor coverage: ERC-4337 specialist audit firms (see umbrella T13.001 for the canonical list).
- Runtime / operational coverage: deployed-contract monitoring (Forta, OpenZeppelin Defender, BlockSec PhalconHQ) tuned for paymaster balance-drawdown signatures; bundler-vendor reputation systems do not directly cover this sub-case.

## Mitigations

- **Paymaster architects:** treat `postOp` revert-path enumeration as a structural review item; verify no revert path leaves a validation-time debit un-unwound; either reject ERC-20 gas tokens with fee-on-transfer / rebasing / pausable / blacklist mechanics or implement explicit accounting that handles them; use checked arithmetic on the user-charge path; if the paymaster supports ERC-20-denominated gas payment, audit the gas-token mechanic for fee-on-transfer (T1.005) and pausable / blacklist (T1.004) surfaces.
- **Paymaster operators:** monitor deposited-balance drawdown against expected sponsorship throughput; reconcile per-UserOp `postOp` outcome against the balance trajectory; pause sponsorship if drawdown rate exceeds a threshold (circuit breaker).
- **Risk teams:** when reviewing a new paymaster integration, require: (1) an audit report from an ERC-4337-specialist firm covering `postOp` revert-safety, (2) the gas-token allowlist with explicit handling of fee-on-transfer / rebasing / pausable / blacklist tokens, (3) the deposited-balance monitoring posture and circuit-breaker threshold.

## Discussion

T13.001.001 is the most operationally-relevant sub-Technique for paymaster operators because it can occur silently — a paymaster slowly leaks deposited balance over many sponsored UserOps and the operator does not notice until the balance is drained. Pre-deployment audit catches the design-time issue; runtime balance-monitoring catches the production-time exploitation; both are required.

The cross-cuts to T1.004 and T1.005 are deliberate: a paymaster that accepts arbitrary ERC-20 gas tokens inherits the entire token-genesis adversarial surface as a paymaster-accounting surface. This is the v0.1 strongest argument for treating gas-token allowlists as a hard control rather than a configuration choice.
