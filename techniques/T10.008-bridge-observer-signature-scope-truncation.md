# OAK-T10.008 — Bridge Observer Signature Scope Truncation

**Parent Tactics:** OAK-T10 (Bridge and Cross-Chain)
**Maturity:** emerging
**Chains:** Cross-chain (any bridge or cross-chain protocol where a relayer/observer network signs observations of external chain events and the signed payload does not cover all semantically-meaningful wrapper fields — THORChain Bifrost, Chainlink CCIP, LayerZero, Wormhole, Axelar, any MPC-based bridge with observer-consensus architecture)
**First documented:** 2026-05-15 (THORChain Router exploit — Bifrost `GetSignablePayload()` truncation: inner `Tx` signed, `ObservedTx` wrapper direction flag unsigned, proposer flipped inbound→outbound to forge migration observation)
**Aliases:** "observer signature scope attack", "unsigned wrapper field forgery", "Bifrost direction-bit attack", "relayer payload truncation", "incomplete payload signing"

## Description

A class of bridge-infrastructure attacks where the attacker exploits a gap between what the observer/relayer network signs and what the consensus layer interprets. The observer's `GetSignablePayload()` (or equivalent signing function) covers only the inner transaction data, while semantically-meaningful fields in the wrapper — direction flags, destination addresses, chain identifiers, message types — sit outside the signed boundary. A malicious proposer (or relayer node) modifies these unsigned wrapper fields to forge observations that carry valid signatures but represent semantically different operations. The consensus layer accepts the forged observation because the existing signatures check out against the signed portion.

The defining structural feature is **signature scope truncation**: the set of bytes covered by the cryptographic signature is a strict subset of the bytes that determine the observation's semantic effect. The bridge contract enforces signature validity correctly — every accepted observation has valid signatures over the inner transaction. The vulnerability is that wrapper metadata (inbound/outbound direction, destination chain, asset type, migration slot) influences the execution path but was never included in the signature's coverage.

T10.008 is structurally distinct from T10.001 (Validator-Signer Key Compromise): no signing key material is compromised. The TSS keys remain secure. The attacker uses validly-signed observations; they only change unsigned metadata.

T10.008 is structurally distinct from T10.002 (Message Verification Bypass): in T10.002 the verification logic itself is broken (e.g., Nomad's `0x00` sentinel collision — the `process()` function treated all messages as proven). In T10.008 the verification logic is correct for what it covers; the flaw is that it does not cover enough. The signatures verify; the bytes they don't cover are the attack surface.

Sub-cases observed and anticipated:

- **(a) Unsigned direction flag forgery.** The observer signs the inner `Tx` but the `ObservedTx` wrapper's direction flag (inbound/outbound) is unsigned. A malicious proposer takes a legitimate deposit observation, flips direction from inbound to outbound, and L1 consensus accepts it as a valid outbound — redirecting allowances or custody to the attacker's address. (THORChain 2026-05-15)
- **(b) Unsigned destination chain forgery.** The wrapper carries a `destChain` field outside the signed payload. The attacker changes it to redirect cross-chain messages to a chain where they control the recipient.
- **(c) Unsigned asset/memo forgery.** The wrapper carries `asset`, `memo`, or `amount` fields outside the signed payload. The attacker modifies these to change what asset is transferred or to insert a malicious memo that changes the downstream interpretation.
- **(d) Unsigned migration slot forgery.** During vault churn, the wrapper carries the migration slot/block height outside the signed payload. The attacker substitutes a different migration slot to replay a past legitimate migration observation with their own address.

## Observed indicators

- A bridge observer/relayer's `GetSignablePayload()` or equivalent signing function returns only the inner transaction bytes, omitting wrapper fields that determine the observation's direction, destination, or type.
- Multiple observers sign the same inner `Tx` but the final committed observation has different wrapper metadata (direction, chain, address) than what individual observers witnessed — detectable by comparing observer-level signatures against the committed wrapper.
- During vault churn or migration, an observation is committed where the destination address has no prior cryptographic binding to the bridge's TSS pubkey or validator set.
- The committed observation's wrapper metadata is inconsistent with on-chain state: e.g., an "outbound" observation whose inner `Tx` data describes a deposit.
- A proposer node consistently produces observations whose wrapper fields differ from what other observers witnessed for the same event.

## Detection signals

- **Payload coverage audit:** For each bridge observer protocol, enumerate the fields in the signed payload (`GetSignablePayload()` output) versus the fields in the full observation struct that influence execution. Flag any semantically-meaningful field that sits outside the signature boundary.
- **Wrapper-vs-inner inconsistency:** For each committed observation, compare the wrapper metadata (direction, destination, chain, asset) against what the inner signed `Tx` data describes. A direction flag of "outbound" paired with inner transaction data that describes a deposit is an anomaly.
- **Proposer metadata divergence:** Compare the wrapper metadata in a single proposer's committed observation against the wrapper metadata in the same event's observations from other observers. Divergence on unsigned fields from one proposer is a signal.
- **During-churn destination validation:** During vault churn/migration events, verify that the destination address in the committed observation is cryptographically derivable from the bridge's TSS pubkey. Even if the observation is forged, this defense-in-depth check catches wrong-destination migrations.
- **Post-upgrade signature scope regression:** After any bridge observer protocol upgrade, verify that `GetSignablePayload()` coverage has not regressed — i.e., no newly-added wrapper field was left outside the signed boundary.

## Real-world examples

- **THORChain Router — Ethereum/BSC/Base/Bitcoin — 2026-05-15.** Loss ~$7.4M+ ($3.7M ERC20s on Ethereum, remainder on BSC/Base/BTC). THORChain's Bifrost protocol used `GetSignablePayload()` which signed only the inner `Tx` data. The `ObservedTx` wrapper carried a `direction` flag (inbound/outbound) that was **outside the signed payload**. A malicious proposer took a legitimate deposit observation, flipped `direction` from inbound to outbound, and the existing observer signatures remained valid. L1 consensus accepted the forged observation during vault churn `MIGRATE:26146440`, executing `transferAllowance` to the attacker's address (`0x82a5...bDA`). The attacker then operated stealthily for ~48 hours (processing 166 ETH outbounds while holding ERC20 allowances idle) before draining all ERC20s. The `thornode` patch (`af46db22`) fixed `GetSignablePayload()` to sign the full `ObservedTx` wrapper including the direction flag. Full proof: [`investigations/2026-05-15-thorchain-router-exploit.md`](../investigations/2026-05-15-thorchain-router-exploit.md).

- Vendor/research coverage: BlockSec Phalcon (post-hoc attack tracing), MetaSleuth (fund flow), ZachXBT (investigative), PeckShield (on-chain monitoring). ## Mitigations

- **Bridge architects (OAK-M02 — full wrapper signing):** `GetSignablePayload()` and equivalent observer signing functions MUST cover the full observation struct — including all wrapper fields that influence execution (direction, destination chain, asset, memo, migration slot). The patch is surgical: expand the signed byte range from `Tx` to `ObservedTx`. THORChain's `af46db22` is the reference fix.
- **Bridge architects (OAK-M08 — migration destination validation):** During vault churn/migration, require on-chain cryptographic verification that the destination address is derivable from the bridge's TSS pubkey. This is defense-in-depth — it catches forged observations even if the signing scope flaw is not yet discovered.
- **Bridge operators (OAK-M09 — multi-observer wrapper consistency):** Require that a supermajority of observers agree on the full `ObservedTx` wrapper (not just the inner `Tx`) before the observation is committed. A single proposer's wrapper metadata diverging from all other observers is rejected at consensus.
- **Audit teams:** For any bridge with observer/relayer architecture, the first audit question is: "Does `GetSignablePayload()` cover every field in the observation struct that influences execution?" Treat any field outside the signature boundary as a standing T10.008 surface.
- **Risk teams:** Before integrating any cross-chain bridge, require: (a) documentation of the observer signing scope, (b) a diff between the signed payload struct and the full observation struct, (c) evidence that all semantically-meaningful fields are inside the signature boundary. Bridges where wrapper metadata is unsigned carry standing T10.008 exposure.

## Discussion

T10.008 is the first OAK technique that targets the "signature scope boundary" — the architectural seam between what a bridge observer signs and what the consensus layer executes. This seam exists in virtually every bridge architecture where observers sign observations of external chain events: THORChain Bifrost, Chainlink CCIP OCR, LayerZero DVNs, Wormhole Guardians, Axelar verifiers — each has a `GetSignablePayload()` or equivalent function whose coverage determines the attack surface.

The THORChain case illustrates a critical signature-engineering principle: **the set of bytes signed MUST equal the set of bytes that determine the observation's semantic effect.** If a field changes what the observation does but sits outside the signature boundary, that field is unsigned attacker-controlled input regardless of how many observers signed the inner payload.

The dust transactions from lookalike vanity addresses (ending in `bDA`) that initially appeared to be the attack vector were "likely a side effect, not the main attack vector" (per @theoblivionsage's revised analysis). The attacker may have used them as camouflage, but the root cause was the unsigned direction flag. This is an important lesson for OAK: **on-chain address patterns can mislead root-cause analysis when the actual vulnerability is in the protocol's signing architecture.** Future T10.008 investigations should prioritize protocol-level payload coverage analysis before correlating on-chain transaction patterns.

The stealth period (48 hours between allowance receipt and drain) remains a valid detection signal regardless of the root cause: an address that processes only native-currency outbounds while holding ERC20 allowances idle is anomalous for a legitimate vault. But this signal detects the *positioning* after a successful T10.008 exploit, not the exploit itself. The exploit is detected by auditing the signing scope.
