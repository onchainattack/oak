# OAK-T9.004 — Access-Control Misconfiguration

**Parent Tactics:** OAK-T9 (Smart-Contract Exploit)
**Maturity:** stable
**Chains:** EVM, Solana, cross-chain
**First documented:** widespread; canonical modern cases 2022+
**Aliases:** "broken access control", "missing authorization check", "guardian-bypass", "logic-flaw extraction", "privilege-boundary violation"

## Description

A class of exploits in which a protocol contract permits an action that should have been gated by a privileged role, by a solvency check, or by another authorisation predicate — but the predicate is missing, evaluated incorrectly, or evaluated against the wrong subject. T9.004 is the broadest single class of smart-contract exploits by incident count: `[owaspscstop10]` (2025/2026) places access-control vulnerabilities at the top of the OWASP SC Top 10. From a defender's perspective, T9.004 is the residual category — the cases where audit, formal verification, or rigorous code review failed to catch a logic flaw before deployment.

OAK groups several sub-patterns under T9.004:

- **Missing authorisation modifier** — the canonical case, where a function lacks an `onlyOwner` / role-gate it should have (KiloEx 2025: external-callable `_setPrices`).
- **Misvalidated authorisation** — the function checks the *wrong* subject (e.g., compares against the caller address rather than the expected counterparty), or fails to validate a critical input (Wormhole 2022: missing guardian-account validation).
- **Inter-contract privilege-boundary violation** — privileged calls between contracts where the calling contract's identity is impersonable (Poly Network 2021: cross-chain manager could be re-targeted via crafted calldata).
- **Missing solvency / invariant check** — the contract permits an action that violates a stated invariant because the check was omitted at the call path that the attacker found (Euler 2023: `donateToReserves`).

## Observed indicators

- Single-transaction extraction events whose mechanism cannot be explained by oracle manipulation, governance acquisition, or operator-controlled outflow — i.e., the contract is being told to do something it should not have permitted.
- Use of an unusual call path or a deprecated function that bypasses the protocol's normal entry points.
- Bytecode patterns: function selectors corresponding to privileged operations (mint, withdraw, setPrice, setOwner, upgradeTo) whose dispatch path lacks a `CALLER` comparison or `SLOAD` of an access-control role mapping.
- Missing-modifier patterns in deployed bytecode: privileged function selector with no `JUMPI` branch on `caller == role-mapping[key]` near function entry.
- Misvalidated-input patterns: external-callable function whose payload encodes a contract / account address that is then passed to a privileged sub-call without validation.
- Post-event: a clean reproducible test case that demonstrates the missing check (audit-firm replay).

## Detection signals

- **Pre-deployment static analysis (OAK-M02):** unprotected privileged functions, missing `require` statements on critical inputs, deprecated function references, function selectors in deployed bytecode that lack expected role-gate dispatch sequences.
- **Symbolic execution / formal verification (OAK-M16):** stated invariants under all reachable call paths; particular attention to inter-contract privilege boundaries and upgradeable-contract initialisation paths.
- **Per-transaction monitoring** at deployed protocols: alert on calls to deprecated or rarely-used functions that are not part of normal user flows; alert on privileged-action calls (mint, withdraw, role-assignment) from EOAs or contracts not in the protocol's published authority set.
- **Post-upgrade smoke-test:** simulate the privileged function call from an unprivileged caller after every contract upgrade; expect failure, alert if the call succeeds.

## Real-world examples

- **Wormhole bridge — Ethereum ↔ Solana — 2022-02-02** — Loss \~\$325M (~120,000 wETH). The attacker exploited a deprecated signature-verification function on the Solana side of the bridge that did not validate guardian-account addresses, allowing the attacker to forge a validator action approval (VAA) calling `complete_wrapped` to mint approximately 120,000 wETH "out of thin air" on the Ethereum side. The protocol's missing-authorisation-check is the canonical T9.004 mechanistically; operationally the case is also a canonical OAK-T10.002 (Message-Verification Bypass) bridge incident and is cross-referenced there. Funds were replaced by Jump Crypto. See worked example at `examples/2022-02-wormhole.md`.
- **Poly Network — Ethereum / BSC / Polygon — 2021-08-10** — Loss \~\$611M (largely returned by the attacker post-event). The attacker exploited an inter-contract privilege-boundary violation: Poly Network's `EthCrossChainManager` contract had authority to invoke `EthCrossChainData.putCurEpochConPubKeyBytes()`, which set the public-key bytes of the cross-chain validator. By crafting a cross-chain message whose `_method` field re-targeted `EthCrossChainData` itself with the attacker's chosen pubkey, the attacker rotated the cross-chain validator's authority to themselves and authorised arbitrary withdrawals. The case is the textbook **inter-contract privilege-boundary** sub-pattern. Sources: `[chainalysispoly2021]`, `[kudelskipoly2021]`, `[elliptipoly2021]`.
- **Multichain — Multi-chain — 2023-07-06** — Loss \~\$126M. While the proximate cause was MPC-key loss (T10.001-shaped), the on-chain extraction surface was an access-control surface: the bridge's withdrawal function authorised any caller in possession of the threshold-signed message, and the off-chain MPC ceremony was the only access-control gate. The case is included as a **boundary case** between T10.001 (off-chain key compromise) and T9.004 (the on-chain consumption surface); OAK classifies it primarily under T10.001 with a T9.004 cross-reference. Sources: `[chainalysismultichain2023]`, `[halbornmultichain2023]`, `[dlnewsmultichain2023]`.
- **KiloEx — BSC / opBNB / Base — 2025-04-15** — Loss \~\$7M. The attacker called the externally-exposed `_setPrices()` function on KiloEx's `MinimalForwarder` contract directly; the function should have been internal or gated by a `onlyKeeper` modifier, but was reachable from any caller. The attacker set a manipulated oracle price, then opened and closed a leveraged position at the manipulated price for a clean extraction. The canonical **missing-modifier** sub-pattern. See worked example at `examples/2025-04-kiloex.md`. Sources: `[kiloexpostmortem2025]`, `[peckshieldkiloex2025]`, `[slowmistkiloex2025]`, `[blocksecsuikiloex2025]`.
- **Euler Finance — Ethereum — 2023-03-13** — Loss \~\$197M. The attacker exploited a missing solvency check in Euler's `donateToReserves` function: the function permitted donating collateral to the protocol's reserves without verifying that the donor's position remained solvent post-donation. The attacker used a flash loan (T9.002) to set up an artificial-debt scenario that became liquidatable after the donation, paying out at liquidation. The attacker subsequently returned the funds. See worked example at `examples/2023-03-euler-finance.md`. Sources: `[chainalysiseuler2023]`, `[halborneuler2023]`, `[blocksec2023euler]`, `[elliptipeuler2023]`, `[eulerlabs2023statement]`.

## Mitigations

- **Protocols (OAK-M16):** require pre-deployment audit by an independent firm; consider formal verification for critical invariants; remove deprecated functions; instrument runtime invariants that pause the contract on violation.
- **Protocols (OAK-M02):** static-analysis gate at CI for unprotected privileged functions, missing `require` on critical inputs, and inter-contract privilege-boundary patterns; require zero new findings before deploying upgrades.
- **Protocols (OAK-M11):** rate-limiting and per-block caps on privileged actions to bound exploit payoff even when authorisation is bypassed (defence-in-depth).
- **Risk teams:** treat audit-pending and audit-not-public protocols as elevated risk; check that audited contract bytecode matches deployed bytecode (OAK-M01); prefer protocols with active bug-bounty programs.

## Discussion

T9.004 is intentionally a broad category because the failure modes vary widely (missing role-gate vs missing input validation vs missing invariant). Sub-Techniques may be split out in v0.x updates as specific sub-patterns reach independent characterisation maturity:

- T9.004.001 — Missing-modifier (KiloEx) — single-line code defect, found by static analysis.
- T9.004.002 — Misvalidated-authorisation / guardian-bypass (Wormhole) — boundary with T10.002 for bridge cases.
- T9.004.003 — Inter-contract privilege-boundary violation (Poly Network) — design-level flaw, found by formal verification.
- T9.004.004 — Missing-invariant / solvency-check (Euler) — pair with T9.002 in chained exploits.

Attribution-strength notes: KiloEx and Euler (high — full forensic write-ups, reproducible test cases); Wormhole (high — Halborn / Chainalysis post-mortem, Jump remediation public); Poly Network (medium-high — attacker engaged publicly post-event, mechanics widely documented but attacker identity remains pseudonymous); Multichain (low — internal-vs-external attribution disputed).

Wormhole is included as the v0.1 canonical example because the missing-authorisation-check is unambiguous from public technical write-ups, the dollar loss is large, and the post-mortem published by Halborn and Chainalysis is sufficient for contributors to understand the failure mode without us reproducing exploit specifics here. The KiloEx case is included specifically because it post-dates the broad adoption of static-analysis CI in DeFi and demonstrates the surface persists when the static-analysis pass is run on the wrong artifact (the contract was deployed but the analysis gate did not block it).
