# OAK-T10.002 — Message-Verification Bypass

**Parent Tactics:** OAK-T10 (Bridge and Cross-Chain)
**Maturity:** stable
**Chains:** EVM, Solana, cross-chain
**First documented:** 2022 (Wormhole, Nomad canonical cases)
**Aliases:** "bridge proof bypass", "message-validation flaw", "VAA forgery" (Wormhole-specific)

## Description

A class of bridge incidents in which the bridge's contract-side message-verification logic permits messages whose proofs are invalid, missing, or trivially constructible by the attacker — without the attacker having to compromise any validator key (which would be the T10.001 path). The attack surface is the **on-chain contract logic** that is supposed to verify cross-chain messages before acting on them. From a defender's perspective, T10.002 is identifiable as a contract-correctness flaw whose detection signal lives at the audit / code-review layer pre-deployment, with limited useful runtime detection signal (by the time a forged-proof message has been processed, the bridge is already executing the attacker's intent).

OAK classifies T10.002 separately from T9.004 (Access-Control Misconfiguration) because the failure mode is bridge-specific: the contract is *supposed* to act on validated cross-chain messages, and the question is *how the validation predicate is constructed*. Wormhole (Feb 2022) is the boundary case — its failure was a missing-guardian-account-validation flaw that is generically T9.004 but operationally a bridge incident; OAK cross-references the case under both Techniques.

## Observed indicators

- Single-transaction "mint from nothing" or "release from locked side without lock" events that the bridge's normal message flow could not have produced.
- Post-upgrade behaviour changes in a bridge's verification function — particularly upgrades that introduce new initialisation logic for verification state.
- Replay-style copying of an exploit transaction by multiple distinct addresses in rapid succession (the Nomad "mob attack" signature).

## Detection signals

- Pre-deployment: audit + formal verification of the bridge's message-verification function under all reachable initialisation states. Particular attention to upgrade paths that re-initialise verification roots or proof predicates.
- Post-upgrade: simulate a randomly-chosen invalid message against the bridge's verification function immediately after every contract upgrade as a smoke-test.
- Runtime: monitor for repeated processing of structurally-identical bridge messages from distinct senders within a short window — the canonical "mob attack" signature observed during the Nomad incident.

## Real-world examples

- **Nomad Bridge — Ethereum / Moonbeam / Avalanche / Evmos — 2022-08-01** — Loss \~\$190M. A routine contract upgrade initialised the bridge's "trusted root" value to `0x00`, which matched the value used to indicate an untrusted root. The verification function consequently treated all incoming messages as proven. The first attacker's exploit transaction was rapidly copied by hundreds of distinct addresses (the "mob attack" signature characterised in the Mandiant write-up), turning a single-attacker incident into an opportunistic frenzy. \~\$22M was recovered post-event. Sources: `[mandiantnomad2022]`, [Halborn — Explained: The Nomad Hack (August 2022)](https://www.halborn.com/blog/post/explained-the-nomad-hack-august-2022), [TRM Labs — Key Suspect in $190M Nomad Bridge Exploit Extradited](https://www.trmlabs.com/resources/blog/key-suspect-in-190m-nomad-bridge-exploit-extradited-to-the-united-states). See worked example at `examples/2022-08-nomad-bridge.md`.
- **Wormhole — Ethereum ↔ Solana — 2022-02-02** — Loss \~\$325M. The attacker exploited a deprecated signature-verification function on the Solana side that did not validate the guardian-account address, allowing forgery of a validator action approval (VAA) calling `complete_wrapped` to mint approximately 120,000 wETH on the Ethereum side without a corresponding lock. Operationally a T10.002 case; mechanistically a T9.004 access-control flaw. Cross-referenced under both Techniques. See worked example at `examples/2022-02-wormhole.md` and related discussion under OAK-T9.004.

## Mitigations

- **Bridge operators:** require pre-deployment audit by an independent firm with explicit attention to verification-function correctness across upgrade paths; implement post-upgrade smoke-tests that simulate invalid messages; instrument runtime invariants that pause the bridge on verification-root anomalies.
- **Risk teams:** treat bridges that have undergone recent contract upgrades as elevated risk for a calibrated post-upgrade window.
- **Bridge architecture:** prefer designs that minimise the verification-function attack surface (e.g., light-client verification with minimal trusted-root initialisation) over designs that require operator-controlled verification roots.

## Discussion

The Nomad case is structurally important to OAK because the "mob attack" signature is unique to T10.002: when the verification flaw can be exploited with a templated transaction, the attack stops being attributable to a single actor and becomes opportunistic. This complicates downstream attribution (`actors/`) and laundering tracking, since the proceeds are distributed across hundreds of independent addresses with mixed motivations. Contributors writing future T10.002 examples should expect this multi-actor pattern when the underlying flaw is templable; single-actor T10.002 cases (Wormhole) are the exception rather than the rule.
