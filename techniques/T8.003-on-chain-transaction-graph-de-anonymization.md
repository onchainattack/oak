# OAK-T8.003 — On-Chain Transaction Graph De-Anonymization

**Parent Tactics:** OAK-T8 (Operational Reuse)
**Maturity:** observed
**Chains:** chain-agnostic (Bitcoin UTXO graph, Ethereum account graph, Solana account graph)
**First documented:** 2013–2015 (Chainalysis, Elliptic, CipherTrace founding era; academic literature on Bitcoin transaction graph clustering)
**Aliases:** "blockchain transaction clustering", "UTXO taint analysis", "address attribution", "on-chain deanonymization", "exchange-deposit clustering", "Chainalysis-style graph analysis"

## Description

The operational security failure in which an attacker (or defender) uses on-chain transaction graph analysis to de-anonymize blockchain addresses, linking pseudonymous addresses to real-world entities (exchanges, custodians, known services) or clustering addresses controlled by the same entity. T8.003 covers the *on-chain graph layer* as a distinct operational security surface — distinct from T8.001 (off-chain key management and funder-graph reuse across operations) and T8.002 (cross-chain operator continuity via cross-chain graph extension).

The technique leverages five canonical clustering heuristics:

- **Co-spend heuristic (multi-input clustering).** Multiple input addresses in a single transaction are controlled by the same entity, because only the holder of each input's private key can produce a valid signature. This is the foundational heuristic for UTXO-based chains (Bitcoin, Litecoin, Bitcoin Cash) and extends to account-model chains where a single transaction draws from multiple sender addresses.
- **Change-address detection.** In UTXO-model chains, the output address that receives the "change" back from a spend is controlled by the sender. Heuristics for identifying the change output include: the output that was not previously seen on-chain (fresh address), the output whose amount is not a round number, and the output whose address format matches the sender's wallet fingerprint.
- **Exchange-deposit address clustering.** Addresses that deposit to the same exchange deposit-address pattern (same exchange hot-wallet cluster, same deposit-address derivation structure) are likely controlled by the same user. This is the primary operational heuristic for exchange-facing compliance vendors (Chainalysis, TRM, Elliptic).
- **Cross-chain address correlation (behavioural fingerprinting).** Addresses that exhibit the same behavioural fingerprint across chains — same transaction timing patterns, same counterparty interaction graphs, same amount-rounding conventions — are jointly attributable to the same entity. This is the T8.002 cross-chain extension applied at the address-identity layer rather than the operator-cluster layer.
- **Off-chain data integration.** Exchange KYC data (when available to the tracing entity), public address labels (Etherscan, Solscan, block explorer tags), forum/social-media address disclosures, and law-enforcement subpoena returns are integrated with on-chain graph data to produce real-world entity attribution.

T8.003 is the structural underpinning of the blockchain forensics industry: Chainalysis, Elliptic, TRM Labs, and CipherTrace were all founded on the premise that on-chain transaction graph analysis can deanonymize blockchain addresses at scale. From a defender's perspective, T8.003 is a **dual-use surface** — the same techniques that compliance vendors use to attribute illicit activity are used by adversaries to identify and target high-value addresses (exchange hot wallets, protocol treasuries, whale addresses) by tracing on-chain flows to known entity labels.

## Observed indicators

- Multiple previously-unlinked addresses consolidated into a single transaction input set (co-spend heuristic trigger).
- Change output returning to a new address in the same wallet — the change address shares a wallet-fingerprint with the sender (address type, lock-script pattern on Bitcoin, wallet-version fingerprint on account-model chains).
- Repeated interactions with the same exchange deposit-address pattern across multiple source addresses — the source addresses share a common deposit-behaviour cluster.
- Temporal correlation of address activity with off-chain events (exchange deposit confirmation time, forum-post timestamp, social-media address disclosure).
- Address reuse across transactions (sending from the same address multiple times) — the most basic de-anonymization signal because it collapses the pseudonymous-address-per-transaction privacy model.

## Detection signals

- **UTXO clustering algorithms.** Chainalysis Reactor, Elliptic Navigator, and TRM Forensics implement production-grade multi-input clustering with change-detection heuristics tuned per-chain. The foundational algorithm is the multi-input heuristic (Meiklejohn et al. 2013) extended with change-address detection and exchange-deposit-pattern matching.
- **Deposit-address pattern matching.** Exchange deposit addresses follow predictable derivation patterns (HD-wallet derivation paths, address-type uniformity, amount-window conventions). Clustering deposit addresses by pattern and attributing common-deposit-pattern source addresses to the same entity is the canonical exchange-compliance workflow.
- **Cross-chain address fingerprinting.** Per-chain behavioural fingerprints (transaction timing distributions, counterparty-graph topology, amount-rounding conventions) are compared across chains; matching fingerprints produce cross-chain address attributions.
- **Temporal-behaviour correlation.** Address activity timing is correlated against off-chain event timing (exchange API deposit-confirmation latency, market-open/close windows, timezone-of-operation inference).
- **Defender-side: deanonymization-exposure monitoring.** Operational security teams should monitor whether their organizational addresses have been deanonymized by checking public address-label databases (Etherscan labels, Solscan tags, OXT.me entity tags, Arkham Intelligence entity labels). An address appearing in a public label database with a correct entity attribution confirms that the address has been deanonymized — the entity should rotate the address and investigate the attribution path.

## Mitigations

- **Operators (OAK-M10 — Operational Security Discipline):** address rotation — each transaction should use a fresh address; avoid address reuse across transactions; avoid consolidating multiple previously-unlinked addresses into a single transaction input set (defeats the co-spend heuristic); avoid sending change to an address whose format reveals wallet-fingerprint information.
- **Operators (OAK-M23 — CoinJoin / Mixer Usage):** UTXO-model chains: use CoinJoin (Wasabi, Samourai Whirlpool, JoinMarket) to break the co-spend heuristic by mixing inputs from multiple participants in a single transaction. Account-model chains: use privacy-pool protocols (Tornado Cash, Railgun) or privacy-preserving L2s (Aztec) to break the account-graph linkability. Note: mixer usage is itself a detection signal (mixer-interacting addresses are flagged by compliance vendors), so the operational security trade-off is between deanonymization risk and mixer-flagging risk.
- **Operators (OAK-M24 — Cross-Chain Privacy Routing):** route funds through cross-chain swap protocols (THORChain, LI.FI, Across) with intermediate privacy-preserving hops to break cross-chain behavioural fingerprinting.
- **Receiving venues (CEXes, custodians):** use unique deposit addresses per customer (industry standard); avoid exposing deposit-address derivation patterns that enable deposit-address clustering; monitor for address-label-database exposure of organizational addresses.
- **Defenders / risk teams:** treat addresses that appear in public label databases with correct entity attribution as compromised from an operational security perspective; force address rotation and investigate the attribution path. Maintain a watchlist of organizational addresses and regularly query public label databases for exposure.

## Discussion

T8.003 is the foundational technique class for the commercial blockchain forensics industry — the entire vendor category (Chainalysis, Elliptic, TRM, CipherTrace/Mastercard) exists because the T8.003 heuristics are operationally effective at scale. The dual-use nature of the technique class (defenders use it for compliance attribution; adversaries use it for target identification) makes it a genuine operational security surface: an entity whose addresses can be deanonymized by a compliance vendor can equally be deanonymized by an adversary with equivalent graph-analysis capability.

The structural relationship between T8.001, T8.002, and T8.003 is a layering model:
- **T8.001** — single-chain funder-graph clustering (operational reuse at the deployment-funding layer).
- **T8.002** — cross-chain operator continuity (joint attribution across chains via cross-chain graph extension).
- **T8.003** — on-chain transaction graph de-anonymization (address-to-entity attribution at the graph-analysis layer, independent of operator-reuse detection).

T8.003 is the most general of the three: T8.001 and T8.002 techniques rely on T8.003 heuristics (particularly multi-input clustering and change-address detection) as primitives, but T8.003 covers the broader deanonymization surface — any entity whose on-chain addresses can be linked to a real-world identity via graph analysis is within T8.003 scope, regardless of whether the entity is an operator cluster (T8.001) or a cross-chain operator profile (T8.002).

False-positive sources at the T8.003 layer: shared-custody addresses (multiple entities using the same CEX deposit address — rare but possible with legacy systems); coinjoin/mixer transactions where the co-spend heuristic produces false clusters (the multi-input assumption fails when inputs come from different entities); exchange hot-wallet consolidation transactions that aggregate deposits from multiple unrelated customers (the co-spend heuristic correctly identifies the exchange as the controlling entity but over-attributes customer addresses to each other). CoinJoin-aware clustering (excluding known CoinJoin transactions from the co-spend heuristic) is the standard false-positive mitigation at the vendor layer.
