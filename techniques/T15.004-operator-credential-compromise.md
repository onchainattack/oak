# OAK-T15.004 — Operator-Side Credential Compromise (SSO / Cloud / Registrar / DNS / Package Registry)

**Parent Tactics:** OAK-T15 (Off-chain Entry-Vector / Pre-Positioning)
**Maturity:** emerging
**Chains:** chain-agnostic (the credential compromise is off-chain; downstream on-chain manifestation is per-incident)
**First documented:** systematically 2022 onward (Curve DNS hijack as a canonical anchor)
**Aliases:** "registrar credential compromise", "DNS hijack", "SSO compromise", "cloud-account takeover", "package-publisher credential compromise", "domain-control compromise"

## Description

Compromise of operator-side credentials, sessions, or identity surfaces — Single Sign-On (SSO) / IDP, cloud-platform accounts (AWS / GCP / Azure / Cloudflare / Vercel), domain-registrar accounts, DNS providers, package-registry publisher credentials, or operator-controlled cloud-database / cookie-store / OAuth-token surfaces — that produce downstream on-chain authority via the operator's UI, signing flow, deployment pipeline, or auth surface. The defining property is that the compromise is at a **non-signing credential surface** whose downstream effect is to manipulate what the on-chain protocol or the operator's audience sees, signs, or trusts.

T15.004 is **distinct from T15.003** in target-surface: T15.003 is the developer / signer / employee endpoint whose machine state is compromised. T15.004 is the operator's **identity / cloud / registrar / DNS / publishing** credential surface — a credential that lives at the cloud / IDP / registrar layer rather than on a developer's machine. (Often a T15.003 endpoint compromise leads to a T15.004 credential compromise — session-cookie theft from a developer's browser produces an SSO session; but the two have different mitigation surfaces and the boundary is operationally useful.)

T15.004 is **distinct from T15.005** in surface-type: T15.005 is operator-controlled communication-channel takeover (Discord / X / Telegram operator account compromise distributing a malicious message to the audience). T15.004 is operator-controlled **infrastructure-credential** compromise (DNS / registrar / cloud / IDP / package-registry / SSO). Boundary cases exist (an X account compromise via cloud-credential reuse could plausibly map to T15.004 or T15.005); contributors should adjudicate based on **what the attacker did with the access**: if the access manipulated a public communication channel under the operator's brand, T15.005; if the access manipulated infrastructure (DNS, hosting, package-registry, cloud-resource), T15.004.

T15.004 covers four structurally adjacent sub-shapes:

- **Domain-registrar / DNS credential compromise.** Attacker takes control of the operator's domain at the registrar layer (or the DNS-hosting layer) and serves a different page from the legitimate domain. Canonical anchors: Curve DNS hijack (2022-08; registrar credential, cloned UI served from `curve.fi`); Galxe DNS (2023-12).
- **Operator cloud / SaaS account compromise.** Attacker compromises operator's account at a cloud / SaaS surface that produces downstream on-chain effect via the operator's distribution channels. Canonical adjacent: CoinStats Snap (2024-06; operator-controlled push-notification infrastructure compromised, distributing a malicious Snap to MetaMask users).
- **Package-publisher credential compromise.** Attacker compromises the credentials that authorise publication to a package registry, without compromising the developer's workstation. The Ledger Connect Kit npm compromise (2023-12) was credential-mediated; the boundary against T15.002 is whether the **shared upstream surface** is the load-bearing class (T15.002) or whether the credential-mediated **identity compromise** is the structural lesson (T15.004); cases often map to both.
- **SSO / IDP / session-cookie compromise that produces operational authority.** Attacker compromises an SSO session or OAuth token granting access to operator infrastructure (cloud admin console, GitHub-org admin, vendor platform admin). Cohort-level: 2022-2026 has documented cases against multiple non-crypto-but-crypto-adjacent targets (Mailchimp 2022, Twilio Aug 2022, Okta sessions 2023). Crypto-specific cases are increasing in frequency but underdocumented at v0.1.

## Observed indicators

- **Off-chain (registrar / DNS):** unscheduled DNS record change at the operator's domain; nameserver change; SSL-certificate issuance for an attacker-controlled IP; registrar account-activity anomaly (login from new region, account-recovery event, MFA-bypass event).
- **Off-chain (cloud / SaaS):** unscheduled IAM policy change; root / admin login from unfamiliar source; OAuth-app authorisation for a new third-party application; API-key creation event; cloud-resource creation in unexpected regions.
- **Off-chain (package registry):** unscheduled publication from a publisher account; new versions published in rapid succession inconsistent with normal cadence; install-side script changes between versions.
- **Off-chain (IDP):** SSO session anomaly; impossible-travel login pattern; MFA-fatigue / push-bombing event; session-token-replay anomaly.
- **On-chain (downstream):** drainer-class extraction from users who interacted with the operator's domain during the DNS / cloud / SaaS compromise window; signature-phishing activity from users who loaded the operator's UI during the window; admin-key activity inconsistent with the operator's normal cadence (when the SSO compromise produced cloud-admin access that gated key material).
- **Forensic record:** registrar / DNS / cloud / IDP audit log; package-registry version history; the SSL-certificate transparency log for the affected domain; cookie / session-token replay evidence where preserved.

## Detection signals

- **Continuous monitoring of operator-controlled DNS / registrar surfaces:** Certificate Transparency log subscription; DNS-record-change alerting; registrar-account-activity alerting; nameserver-change alerting.
- **Cloud / IDP audit-log SIEM integration:** continuous ingestion of cloud / SSO / IDP audit logs into a SIEM with alerting on policy changes, root logins, OAuth grants, MFA anomalies.
- **MFA enforcement at the registrar / DNS / cloud / package-registry / IDP layer:** specifically, hardware-key MFA (FIDO2 / WebAuthn) on the highest-leverage credentials; SMS / TOTP MFA is insufficient against the documented MFA-fatigue / push-bombing patterns.
- **Registrar-lock + transfer-lock at the registrar layer:** prevent registrar-level transfers without out-of-band confirmation; register lock SLA with the registrar.
- **Customer-side detection:** verify the SSL certificate of the operator's domain matches the expected operator-side certificate; reject UI loads with mismatched certificate fingerprint.
- **Cohort-level on-chain detection:** when a registrar / DNS / cloud / package-publisher disclosure lands, treat operator's user base for the disclosure window as elevated risk.

## Mitigations

- **Hardware-key MFA on registrar / DNS / cloud / IDP / package-registry credentials (highest-leverage):** FIDO2 / WebAuthn on every credential whose downstream effect is operator-infrastructure-control; SMS / TOTP MFA is insufficient against documented attack patterns.
- **Registrar-lock + DNSSEC:** prevent registrar-level transfers and DNS-record changes without out-of-band confirmation; register-lock SLA with the registrar.
- **Cloud / IDP audit-log SIEM integration with continuous alerting on privilege changes, OAuth grants, root logins.**
- **OAK-M22 (Rotate-on-disclosure):** when a credential surface is suspected compromised, rotate every dependent credential (API keys, deployer keys, OAuth grants) before evaluating downstream impact.
- **Customer-side defensive signal:** wallet vendors and dApp users should verify SSL certificate fingerprints against an expected reference; flag UI loads from operator domains whose certificate fingerprint differs from the registered reference.
- **Cohort-level CTI integration:** subscribe to registrar / cloud / SaaS provider security advisories; cross-reference with internal credential audit trail.

## Discussion

T15.004's structural lesson is that operator-side **credential compromise can produce downstream on-chain authority without ever touching the on-chain protocol or the signing host**. The Curve DNS-hijack case is canonical here: the Curve smart contracts were not compromised; the signing flow was not compromised; the only thing that was compromised was the registrar credential, which the attacker used to serve a different UI from the legitimate domain. The on-chain victims signed away authority because they trusted the domain.

T15.004 is also the sub-Technique that **most clearly demonstrates that credential surfaces have an asymmetric leverage profile**. A single compromised registrar credential gives the attacker control of every page served from the operator's domain for as long as the compromise persists; a single compromised cloud-admin credential gives the attacker control of every cloud-resource the operator depends on; a single compromised package-publisher credential gives the attacker control of every downstream consumer of the package. Defender investment in hardware-key MFA on these credentials specifically (rather than on per-employee email accounts in general) is the highest-leverage T15.004 mitigation.

The boundary against T15.005 is **what surface the access manipulates**: T15.004 manipulates infrastructure; T15.005 manipulates the operator's communication channel under the operator's brand. The 2025-02 Solana brand-X-account compromise cohort sits **on this boundary** — the access was credential-mediated (X-account session compromise), but what the attacker did was distribute malicious messages under the operator's brand, which is structurally T15.005. OAK maps that case primarily to T15.005; if a future case has a credential-compromise that manipulates infrastructure rather than communication, it maps to T15.004.

The boundary against T15.002 (supply-chain) is **whether the load-bearing structural lesson is the shared-upstream-surface propagation** (T15.002) **or the credential-identity compromise itself** (T15.004). Cases like the Ledger Connect Kit npm compromise sit on this boundary: the publisher credential was compromised (T15.004 surface), but the load-bearing structural lesson is that the malicious package propagated to every downstream consumer (T15.002). Such cases map primarily to T15.002 with T15.004 as the producer-side antecedent.

False-positive considerations: legitimate registrar / DNS / cloud / IDP changes are common; the detection signal is not "a change occurred" but "the change was unscheduled" or "the change pathway bypassed normal review." This is why MFA enforcement + continuous audit-log SIEM is the canonical detection posture rather than per-event alerting.

v0.x deferred items: clearer treatment of the **session-cookie-theft → SSO-bypass** sub-shape that increasingly serves as the entry vector for cloud-account compromise (the boundary against T15.003 endpoint state where the cookie was stolen is operational rather than structural); Web3 wallet-as-identity surface (where the operator's signer wallet is itself the SSO surface) as a distinct sub-shape; multi-stage credential-compromise chains where T15.001 → T15.003 → T15.004 → T11.x reads as a four-stage chain.
