# OAK-T6.003 — Audit-of-Different-Bytecode-Version

**Parent Tactics:** OAK-T6 (Defense Evasion); modifier on T1 (Token Genesis), T2 (Liquidity Establishment), T3 (Holder Capture), T9 (Smart-Contract Exploit) classes when paired
**Maturity:** stable
**Chains:** chain-agnostic (the failure mode is the audit-vs-deployed gap; the underlying Technique it modifies may be on any chain)
**First documented:** widespread from approximately 2021 onward
**Aliases:** "audit-deployed mismatch", "post-audit redeploy", "audit-version drift", "audit of fork"

## Description

A class of OAK-T6 evasion patterns in which a token, protocol, or service truthfully claims audit coverage by a recognised firm — but the audited bytecode differs from what is actually deployed at the cited contract address. The audit certificate is real and verifiable at the firm's registry; the audited code is not what is running on chain. From a defender's perspective, T6.003 defeats the user's reasonable trust that "audited by Firm X" implies professional review of the code that will execute on their funds, while leaving the audit-firm-side verification surface (registry lookup) intact and apparently confirmatory.

T6.003 is distinct from OAK-T6.001 (Source-Verification Mismatch) in that the *block-explorer source verification may match the deployed bytecode* (deterministic recompile of the verified Etherscan source produces the deployed bytecode within compiler-deterministic noise) — yet the *audit firm's audit report* references a different version of the code. T6.001 lives at the explorer-source-vs-deployed-bytecode layer; T6.003 lives at the audit-report-vs-deployed-bytecode layer. The two are independent: a contract can pass T6.001 detection (verified source matches deployed bytecode) and still fail T6.003 detection (audited bytecode does not match deployed bytecode).

T6.003 is distinct from OAK-T6.002 (Fake Audit-Claim) in that the audit firm did engage with the project and produced a real audit — it is published in the firm's registry, and the firm will confirm rather than deny the engagement on inquiry. The failure mode is not fabricated affiliation; it is *scope drift* between what was audited and what is deployed. T6.002 mitigations (per-firm registry verification) succeed against T6.003 only insofar as the registry exposes the audited bytecode hash; if the registry exposes only project name, scope, and date, T6.003 evades T6.002-style checks.

## Observed indicators

- A project that cites a real, registry-listed audit report whose audited contract version (commit hash, file set, or audited-bytecode hash) cannot be matched against the currently-deployed bytecode at the cited address.
- A project whose audit report covers a single contract but whose deployed system is multi-contract, with the unaudited contracts holding upgrade authority, treasury authority, or routing authority over the audited contract.
- A proxy contract that cites a prior implementation's audit while currently delegating to an unaudited or modified implementation.
- A redeploy event (CREATE2 redeploy after `selfdestruct`, or fresh deployment to a new address marketed under the same project name) post-dating the cited audit, with no follow-up audit on the redeployed bytecode.
- A forked / customised version of an audited reference codebase (e.g., a forked Uniswap V2 / Compound V2 / OpenZeppelin contract) deployed under marketing language that conflates the upstream audit with coverage of the local fork.

## Detection signals

- **Canonical signal (highest-leverage):** compare the audit-firm-published audited-bytecode hash (or audited source-commit hash recompiled deterministically) against the deployed bytecode at the cited contract address. Non-match (beyond compiler-deterministic noise) is the canonical T6.003 signal. This signal requires the audit firm to publish the audited-bytecode hash in machine-readable form; where the firm publishes only a PDF report referencing a commit hash, the user must recompile from that commit and compare to deployed bytecode.
- **Proxy-implementation review:** where the cited contract is a proxy, resolve the current implementation address and compare it to the implementation address audited in the cited report; T6.003 frequently presents as a proxy whose current implementation is not the audited implementation.
- **Multi-contract scope review:** map the deployed system's contract graph (factory, router, treasury, governor, upgrader) and confirm that each authority-holding contract is in scope of the cited audit; T6.003 frequently presents as a single-contract audit cited as coverage for a multi-contract system.
- **Redeploy detection:** monitor for redeploy events (CREATE2 redeploy, fresh deployment) post-dating the cited audit at the project's claimed address-of-record; treat any redeploy without a corresponding follow-up audit as a candidate T6.003 vector.
- **Fork-conflation detection:** where the project cites the audit of an upstream reference codebase (e.g., Uniswap V2 core audit), require deterministic recompile of the deployed bytecode from the upstream audited commit — any local-fork modifications surface as bytecode differences and resolve the conflation.

## Real-world examples

- **Cohort scale — multi-chain — 2021 onward** — T6.003 is most visible at the cohort level rather than at the per-incident level, because the underlying primary Technique (T1, T5, T9) is typically the more publicly-reportable framing once funds are lost. Industry retrospectives (`[chainalysis2025rug]`) cover the cohort-scale impact of audit-coverage-of-record claims that did not bound the deployed system; the audit-vs-deployed gap is a recurring modifier across rug-pull and exploit cohorts but is rarely the primary headline.
- **Adjacent — Swaprum, 2023** — \~\$3M rug pull on a CertiK-audited project. The audited contract was the deployed contract; the operator team performed a post-launch treasury-management exit (T5.005). This is *not* a T6.003 case — the audited bytecode and deployed bytecode matched — but it is cited as adjacent reference because public discourse around the case frequently conflates "audited contract was rugged" with "audit did not cover the deployed code", which is the T6.003 framing. Source: `[dlnewsswaprum2023]`.
- **Adjacent — fake-audit cohort, 2024** — `[certikfakeaudit]` documents the T6.002 (Fake Audit-Claim) cohort. T6.003 is the cleanly-distinct sibling failure mode: in T6.002 the audit does not exist; in T6.003 the audit exists but covers different bytecode.
- **Arbix Finance — Binance Smart Chain — 2022-01-04** — Loss \~\$10M aggregate across BEP-20 token positions deposited by users. Arbix Finance held a real CertiK audit on its ARBX token contract published in November 2021, but the user-deposit-routing contract (the depositor contract that received user-deposited stablecoins, BETH, CAKE, etc.) and the destination yield pools were *not* in the audit scope and were source-unverified on BscScan. The operator drained the unverified pools and dumped freshly-minted ARBX (eight `mint()` functions, \~10M ARBX minted to operator-controlled addresses), then routed proceeds via BSC -> ETH -> Tornado Cash. CertiK's public callout within hours of the drain cited the unverified-pool routing as the canonical T6.003 signal. Arbix is the canonical T6.003 worked-example anchor for the audit-scope-mismatch sub-shape: real, registry-verifiable audit on the headline contract; multi-contract deployed system with the audit-coverage-of-record covering one contract while the user-funds-at-risk contract was outside scope. Composing Techniques: T5.003 (hidden mint), T5.001 (hard LP drain), T7.001 (Tornado Cash routing). See [`examples/2022-01-arbix-finance.md`](../examples/2022-01-arbix-finance.md).
- **Per-incident named cases — pre-Arbix corpus state** — Prior to the Arbix anchor, OAK did not name a single canonical T6.003 incident with public forensic disclosure of the audit-vs-deployed-system gap. Audit firms have published advisories on the pattern at the class level (as part of brand-misuse and audit-coverage-scope discussions); additional per-incident write-ups that specifically document a bytecode-hash mismatch between audit report and deployed contract are encouraged via PR.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1 (T6.003 detection requires both a machine-readable audit-attestation source and a deployed-bytecode comparator; the audit-attestation side of the pipeline is the binding constraint).
- Vendor coverage: bytecode-comparison tooling at audit firms is **emerging** rather than mature at v0.1. Some firms publish audited-commit hashes in their PDF reports; few publish audited-bytecode hashes in machine-readable form. Sourcify-style cryptographic source-verification provides a partial substitute where the audited source is the same as the verified source, but does not generalise to the audit-attestation case.
- Adjacent: block-explorer "verified bytecode" infrastructure (Etherscan, Sourcify) provides the deployed-bytecode-hash side of the comparison; the audit-firm side is the gap.

## Mitigations

- **Audit firms (highest-leverage):** publish the audited-bytecode hash (or, where the audit is source-level, the deterministic-recompile bytecode hash from the audited source commit) in machine-readable form in the audit registry, alongside the project name, scope, and date. This converts T6.003 detection from a manual recompile-and-compare exercise into a registry-API lookup that wallet vendors and risk teams can integrate.
- **Standardisation (canonical solution):** machine-readable cryptographic audit attestations binding the audited bytecode hash to the audit certificate — analogous to Sourcify-style cryptographic source-verification but at the audit-attestation layer. An attestation format that wallets, explorers, and risk-team tooling can verify against the deployed bytecode would reduce T6.003 to a deterministic check rather than a manual review.
- **Wallet vendors / risk teams:** treat "audited by Firm X" project-self-claims as a starting point, not a conclusion; where the cited audit is registry-listed, perform the audited-bytecode-vs-deployed-bytecode comparison before surfacing an "audited" label at the user-warning layer; for proxy contracts, surface the audit-vs-current-implementation gap explicitly.
- **Block explorers:** where the verified-source claim and a cited audit reference different commits, surface the discrepancy in the contract page UI; the explorer is uniquely positioned to flag the audit-vs-source drift even before the audit-vs-bytecode comparison is automated.
- **Project teams:** publish the audited-commit hash and the deployed-bytecode hash side-by-side at the audit-claim surface (project documentation, marketing site, contract page); for multi-contract systems, publish the audited-scope graph (which contracts are in scope, which are not) explicitly rather than letting the user infer scope from the project name.

## Discussion

T6.003 closes the third corner of the OAK-T6 audit-trust failure-mode triangle: T6.001 (verified source does not match deployed bytecode), T6.002 (claimed audit does not exist), T6.003 (audit exists but covers different bytecode). The three Techniques together describe the failure modes of the user-facing assertion "this contract has been audited" and bound the corresponding mitigation surfaces: explorer-side bytecode verification (T6.001), audit-firm-side registry verification (T6.002), and audit-firm-side bytecode attestation (T6.003).

The choice to scope T6.003 around *bytecode-version drift* (rather than including audit-quality, audit-scope-narrowness, or audit-recency concerns) reflects the same cleanly-binary-verification standard applied to T6.001 and T6.002: the audited-bytecode-hash-vs-deployed-bytecode-hash comparison is deterministic and operationally actionable, given the audit-attestation source. Adjacent failure modes — shallow audits, narrow-scope audits where the unaudited adjacent contracts are the failure surface but were never claimed in scope, audits that have aged past meaningful relevance — are documented as adjacent observations but are not classified as T6.003 because the failure mode is qualitative rather than binary and the mitigation is editorial rather than mechanical.

A live discussion in the audit-firm community concerns whether audit firms should standardise on publishing audited-bytecode hashes in machine-readable form as a default deliverable. OAK does not opine but documents that the audit-attestation gap — the absence of machine-readable bytecode-bound audit certificates — is the primary defensive surface that T6.003 exploits, and that closing this gap is the canonical mitigation. Until that infrastructure matures, T6.003 detection at scale remains a per-project manual exercise, and audit-claim assertions at the user-decision layer should be treated as load-bearing only after the audited-vs-deployed bytecode comparison has been performed.
