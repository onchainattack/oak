# OAK-T15.005 — Operator-Communication-Channel Takeover (Discord / X / Telegram)

**Parent Tactics:** OAK-T15 (Off-chain Entry-Vector / Pre-Positioning)
**Adjacent on-chain Tactics:** T4 (Access Acquisition — operator-channel takeover typically chains into T4.005 / T4.002 on-chain extraction)
**Maturity:** emerging
**Chains:** chain-agnostic (the channel takeover is off-chain; downstream on-chain manifestation is per-incident)
**First documented:** systematically 2022 onward (the Bored Ape / Yuga / Ronin Discord wave is the foundational cohort)
**Aliases:** "Discord compromise", "X account compromise", "Telegram channel takeover", "operator-brand-channel compromise", "community-manager account compromise", "official-channel phishing"

## Description

Compromise of an operator-controlled communication channel — a Discord server / moderator account, a brand X (Twitter) account, a Telegram channel admin, an Instagram account, a project's community-management surface — used to distribute a malicious link, transaction-prompt, or message to the operator's audience under the credibly-held operator brand. The defining property is that the audience trusts the message because it arrived through a channel the operator legitimately controls; the trust-substrate is the **brand identity** of the channel, not the underlying smart-contract authority of the project.

T15.005 is **structurally distinct** from end-user phishing where the attacker impersonates the operator from outside the operator's channel. T15.005 requires the attacker to have actually compromised the operator's channel — the message comes from the legitimate handle, the legitimate Discord moderator role, the verified X account, the registered Telegram admin. Defenders evaluating phishing risk against an operator's audience can filter "external impersonation" with relatively low effort; T15.005 defeats that filter because the message originates from the channel.

T15.005 is **distinct from T15.004** in surface-type. T15.004 covers operator infrastructure-credential compromise (DNS, registrar, cloud, IDP, package-registry); T15.005 covers operator **communication-channel** compromise where the load-bearing surface is the **public channel under the operator's brand**. Boundary cases exist (an X-account compromise via cloud-credential reuse could plausibly map to either), and OAK adjudicates by **what the access was used for**: if the attacker manipulated public communications under the operator's brand, the case is T15.005; if the attacker manipulated infrastructure, T15.004.

T15.005 covers four structurally adjacent sub-shapes:

- **Discord moderator / admin account compromise.** Most common channel; the attacker compromises a Discord moderator's session or credentials and posts a malicious "official mint" / "claim airdrop" / "snapshot" link in the project's official Discord channel. Canonical anchors: Bored Ape / BAYC Discord wave (April 2022 onward); Ronin Discord follow-up (April 2022, post-bridge incident); Yuga Otherside community-manager Discord compromise (August 2022 onward).
- **X (Twitter) account compromise of a project's brand handle.** The attacker compromises the project's verified X account (typically via session-cookie theft, SIM-swap, OAuth-app authorisation, or direct credential compromise) and posts a malicious link / image / contract address. Canonical anchor: Solana brand-X-account compromise cohort (February 2025; Jupiter, Pump.fun, DogWifCoin and others compromised in a coordinated wave).
- **Telegram channel admin compromise.** The attacker compromises a Telegram channel admin and posts a malicious message; common in regional crypto-projects with Telegram-primary community surface.
- **Instagram / other social channel compromise.** Lower-frequency but documented; the BAYC Instagram compromise (April 2022) was canonical in the NFT cohort.

## Observed indicators

- **Off-chain (channel-side):** unscheduled message from an official channel containing a wallet-connect link, a contract-address claim, an "urgent" mint window, or a snapshot announcement; account-recovery / login-anomaly events at the channel platform; loss of access reported by the legitimate channel admin; channel-history modification (deletion of pre-compromise messages).
- **Off-chain (operator-side):** account-recovery activity at the platform's audit log (X / Discord / Telegram / Meta); MFA-bypass / session-token-replay forensic record where preserved; OAuth-app authorisation event for a third-party application granting send-message permission.
- **On-chain (downstream):** drainer-class extraction events from users who interacted with the malicious link during the compromise window; signature-phishing activity from users who loaded the linked dApp; cluster of victims with a common time window aligned with the channel-compromise event.
- **Forensic record:** the platform's account-recovery audit log; preserved screenshots of the malicious-message window; on-chain wallet cluster of the recipient address.

## Detection signals

- **Hardware-key MFA on every operator-side communication-channel admin account.** FIDO2 / WebAuthn where the platform supports it (X, Discord, Meta); for Telegram, hardware-key 2FA via the Telegram passcode + cloud-password layer.
- **Session / OAuth audit on operator-side admin accounts:** continuous review of active sessions, OAuth-authorised third-party apps, and connected devices on every official channel.
- **Multi-mod review for high-stakes announcements:** require >1 admin to confirm before posting any message containing a wallet-connect link, a contract-address claim, or a transaction prompt; cross-channel confirmation (Discord announcement → X confirmation → blog-post confirmation, with delays sufficient to detect a single-channel compromise).
- **Audience-side defensive signal:** treat any official-channel message that carries an actionable transaction-prompt, mint window, or claim link as elevated-risk; cross-confirm against the operator's other channels before signing.
- **Cohort-level pattern detection:** when a wave of operator-channel compromises is observed (the Solana 2025-02 cohort is canonical), watchlist the recipient addresses and warn users on adjacent operator channels.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T15.005 detection lives at the off-chain channel-platform layer. On-chain post-event signals (cluster of `setApprovalForAll` events to a single grantee with common time window) are detectable at the runtime layer but only after extraction has begun.
- Vendor coverage: ScamSniffer (active phishing-domain feed); SlowMist (incident response and channel-compromise reporting); Discord / X / Telegram platform-side account-recovery and audit telemetry where shared with operators.

## Mitigations

- **Hardware-key MFA on every operator-channel admin (highest-leverage):** FIDO2 / WebAuthn where supported. SIM-swap and SMS-MFA-bypass have been the dominant entry vectors in documented cases.
- **Multi-mod review and cross-channel cross-confirmation for high-stakes announcements:** any message with a transaction-prompt, mint link, or contract-address claim should require multi-mod approval and confirmation across at least two independent channels with a sufficient delay window.
- **OAK-M22 (Rotate-on-disclosure):** when an operator-channel admin is suspected compromised, rotate the credential, revoke OAuth grants, and audit the channel's recent message history before posting any subsequent operator messages.
- **Audience-side education:** treat any official-channel transaction prompt as elevated-risk; cross-confirm before signing; recognise that "verified handle" / "Discord moderator role" do not authenticate the message intent.
- **Cohort-level CTI integration:** subscribe to ScamSniffer / SlowMist phishing-feed advisories on active operator-channel compromises; cross-reference with on-chain wallet-cluster watchlists.

## Discussion

T15.005's structural lesson is that the **operator's communication channel is itself a trust-substrate**. The audience does not validate the message's content against the operator's smart-contract authority; the audience trusts the channel because the brand is on it. A compromise of the channel converts the operator's accumulated brand-trust into a single-event phishing accelerant against the audience.

The cohort is structurally distinct from end-user phishing because the **filter "is this from the operator?" returns true** — the message came from the legitimate channel. Defender investment in audience-side education has to recognise this; the canonical defensive heuristic ("only trust messages from the official channel") is exactly the heuristic T15.005 defeats. The mitigation has to live at the channel-side (hardware-key MFA, multi-mod review) and at the audience-side defensive heuristic ("treat any actionable transaction prompt as elevated-risk regardless of source").

T15.005 is the OAK Tactic-level home for the **operator-side credential-compromise antecedent** that earlier OAK drafts documented as a footnote on T4.005 / T4.002 worked-example pages. The 2022-04 BAYC Discord wave and the 2025-02 Solana X-account cohort were the two largest documented cohorts of this class at v0.1; both were mapped to T11 broadly construed in earlier OAK drafts because no Tactic existed for the channel-takeover surface specifically. T15.005 fixes that.

The boundary against T15.004 is what-the-access-was-used-for. The boundary against T4 is the layer where the defender signal lives: T15.005's signal is at the channel platform's account-recovery audit log; T4's signal is at the on-chain authority artefact. A complete chain reads T15.005 (channel-takeover) → T4.005 / T4.002 (on-chain extraction).

False-positive considerations: legitimate operator messages are routinely actionable (mint windows, airdrop claims, snapshot announcements, treasury operations). The detection signal is not "the message is actionable" but "the message arrived without the operator's documented multi-channel confirmation pattern" or "the channel's account-recovery / session-anomaly signal fired in the relevant window." This is one of the reasons multi-mod review + cross-channel cross-confirmation is the canonical mitigation — it converts a single channel-compromise into a coordinated multi-channel-compromise problem.

v0.x deferred items: cleaner treatment of cases where the **channel was always operator-controlled but the operator themselves is acting maliciously** (rug-pull operator using their legitimate channel as the rug-pull vehicle — structurally T2.x / T5.x, not T15.005); cohort-level metrics on time-from-compromise-to-takedown across X / Discord / Telegram (X has been the slowest-responding platform in documented cases); the role of the platform's verification-tier signal (X blue-checkmark / Discord verification-badge) in audience deference, given that those signals are now decoupled from operator-credibility in 2024-2026.
