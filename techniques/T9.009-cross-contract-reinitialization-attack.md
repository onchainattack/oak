# OAK-T9.009 — Cross-Contract Reinitialization Attack

**Parent Tactics:** OAK-T9 (Smart-Contract Exploit)
**Maturity:** emerging
**Chains:** EVM (primary); any chain whose execution model allows cross-contract calls during initialisation (upgradeable proxy patterns are EVM-native but the reinitialisation surface generalises)
**First documented:** 2023 (class characterised in audit-firm literature post-UUPS/transparent-proxy proliferation; specific named exploits from 2023 onward)
**Aliases:** "reinitialization attack", "double-init attack", "cross-contract init callback", "proxy reinitialization", "init-state injection"

## Description

A smart-contract exploit in which the attacker triggers the initialisation function of a target contract via a callback from a peer contract, exploiting the absence of reinitialisation guards across the cross-contract boundary. The target contract (typically an upgradeable proxy implementing OpenZeppelin's UUPS or Transparent Proxy pattern) relies on an `initializer` modifier that sets a storage flag after the first successful initialisation — but the flag is set at the end of the initialiser's execution, not before external calls made during initialisation. If the initialiser makes an external call to a peer contract (Contract B), and Contract B's callback re-enters the target contract's not-yet-flagged initialisation path, the attacker can execute the initialisation logic twice — setting privileged roles, minting tokens, or configuring parameters that should only be settable once. The reinitialisation guard fires only after the outer initialiser returns, and the inner (re-)initialisation has already completed by then.

T9.009 is structurally distinct from T9.005 (Reentrancy): T9.005 exploits external calls during state-mutating operations whose state updates are deferred until after the external call returns; T9.009 exploits external calls during the **initialisation** phase — the window where the contract's state is supposed to be transitioned from "uninitialised" to "initialised" exactly once. The reinitialisation guard (OpenZeppelin's `initializer` modifier, inherited from `Initializable`) is the structural parallel to `ReentrancyGuard`: it protects against calling `initialize` twice from outside the contract, but it does not protect against a cross-contract callback during the initialiser's external call because the guard flag has not been set yet. In T9.005, the defence is checks-effects-interactions; in T9.009, the defence is initialise-then-interact — performing all external calls after the initialisation state flag has been set.

The attack is a specific failure of the UUPS/Transparent proxy initialisation discipline, but the class is broader: any contract that (a) has a once-only initialisation path, (b) makes an external call during that path, and (c) sets the "already initialised" flag after the external call is vulnerable. Non-proxy contracts with constructor-replacement patterns (EIP-1167 minimal proxies, Create2-deployed clones with `init` entrypoints) exhibit the same surface.

## Observed indicators

- The target contract's initialisation function makes an external call (to a peer contract, a token, or a user-supplied address) before setting the `initialized` storage flag — the external call site is the load-bearing indicator.
- The peer contract called during initialisation has a code path that calls back into the target contract — and that callback path reaches the initialisation entrypoint rather than being gated by a separate access-control check.
- Post-exploit: the target contract's privileged roles (owner, admin, minter, fee-recipient) are held by an attacker-supplied address that was set during the inner (re-)initialisation, not during the outer initialisation.
- The transaction trace shows the target's `initialize` function selector appearing twice in the same call-tree depth, with the inner call originating from a peer contract invoked during the outer initialisation's external call.
- Pre-deployment: an upgradeable proxy contract whose `initialize` function makes external calls (including token transfers, oracle reads, or peer-contract registrations) before the `__gap` / storage-layout finalisation — the external call site is an audit-flag surface independent of the reinitialisation guard.

## Detection signals

- Pre-deployment static analysis: flag any `initialize` function that performs an external call before the `initializer` modifier's guard flag write — i.e., the external call precedes the `onlyInitializing` check on re-entry. This is a structural audit signal.
- Pre-deployment proxy audit: for every upgradeable proxy contract, enumerate all external calls made during the `initialize` path; verify that every called contract's code is known and cannot call back into the target's initialisation path; treat user-supplied addresses as a reinitialisation surface by default.
- Per-transaction monitoring at deployed protocols: alert on transactions whose internal call trace shows a contract's `initialize` function selector emitted more than once within a single transaction, where the inner call originates from a contract called during the outer initialisation.
- Post-deployment proxy state audit: for each deployed proxy, verify that the `initialized` flag version matches the expected version and that privileged roles (owner, admin) were set exactly once by the canonical deployer address; a privileged role set to a non-canonical address is an indicator of potential reinitialisation.

## Real-world examples

- **Paraspace reinitialisation / BlockSec whitehat rescue — Ethereum — 2023-03.** The ParaSpace NFT lending protocol deployed an upgradeable proxy whose initialisation path lacked a cross-contract reinitialisation guard; a whitehat (BlockSec) demonstrated the reinitialisation vector, calling `initialize` a second time via a callback from a peer contract invoked during the initialiser's external call. The whitehat used the vector to rescue funds from a separate vulnerability path; the reinitialisation surface was disclosed and patched. The case is the canonical OAK v0.1 anchor for T9.009 because it combines the cross-contract callback path with a publicly documented initialisation flow.
- **Audit-firm literature — UUPS/Transparent proxy reinitialisation class — 2023–2024.** Trail of Bits, OpenZeppelin, and Spearbit have each published advisories flagging the pattern: an `initialize` function that performs external calls before the `initializer` modifier's guard takes effect (i.e., before the storage flag write commits). The class is well-documented in pre-deployment audit findings but the public named-exploit record is sparse relative to T9.005 — most instances are caught at the audit layer. The class is listed here as emerging rather than stable because the surface is growing with proxy-pattern adoption.
- **Cohort context.** The reinitialisation class is structurally adjacent to the UUPS selfdestruct / storage-collision class (T9.004 — access-control misconfiguration) and the proxy-upgrade-malicious-switching class (T6.005). The OAK T9.009 distinction is that the reinitialisation occurs through a **legitimate initialisation entrypoint** exploited via cross-contract callback timing, not through a misconfigured access-control check.

## Mitigations

- **Protocols (initialise-then-interact discipline):** perform all external calls after the initialisation state flag has been committed — the analogue of checks-effects-interactions applied to the initialisation phase. Set up roles, parameters, and storage state first; interact with external contracts second.
- **Protocols (OAK-M01 — audit discipline for proxy contracts):** require audit-firm review of every upgradeable proxy's initialisation path, with specific attention to the ordering of external calls relative to the `initializer` guard flag.
- **Protocols (cross-contract reinitialisation guard):** deploy a per-contract reinitialisation lock that spans the proxy and all peer contracts called during initialisation — a cross-contract `initializer` that gates not only direct calls but also indirect calls received during the initialisation window.
- **Risk teams:** maintain per-protocol inventories of (a) which contracts are upgradeable proxies, (b) which proxies' initialisers make external calls, and (c) which peer contracts are called during initialisation. Treat proxies whose initialisers call user-supplied addresses as elevated T9.009 risk independent of existing audit posture.

## Discussion

T9.009 is structurally the initialisation analogue of T9.005's checks-effects-interactions failure: in T9.005, the contract mutates state after an external call; in T9.009, the contract performs its one-time state setup (initialisation) that includes an external call before committing the "already initialised" flag. The canonical defences are parallel — checks-effects-interactions for T9.005, initialise-then-interact for T9.009. The OpenZeppelin `Initializable` contract provides the `initializer` modifier as a guard, but that guard protects against direct re-calls of `initialize` from outside the contract; it does not protect against a peer contract's callback during the initialisation external-call window. The attacker does not bypass the guard — the guard has not yet been set.

The class is marked emerging rather than stable because (a) proxy-pattern adoption is still growing, (b) the public named-exploit record is sparse relative to the audit-finding record (most instances are caught pre-deployment), and (c) the UUPS/Transparent proxy pattern is evolving (OpenZeppelin v5 introduced structural changes to the `Initializable` contract). Contributors writing future T9.009 entries should anchor any new named exploit against the public forensic record and cross-reference the specific proxy pattern (UUPS vs. Transparent vs. Beacon) in the worked example.

False-positive considerations: a legitimate proxy upgrade or reinitialisation (e.g., a protocol governance-approved upgrade that calls a new initialiser version with a new version number, as supported by OpenZeppelin's `reinitializer` modifier) is not T9.009. The distinguishing signal is the cross-contract callback during the initialisation external-call window — the `initialize` function appears twice in the call trace, the inner call originates from a peer contract invoked during the outer initialisation, and the privileged-role assignments from the inner call differ from the outer call. The reinitialisation guard's version-number mechanism (OpenZeppelin v4.5+, `reinitializer(n)`) partially mitigates the surface by requiring the inner call to use a different version number, but the guard still does not protect against a peer contract that calls the *same* initialisation version from a callback — the `onlyInitializing` modifier only checks that the contract is currently in an initialisation context, not that the callback origin is the expected initiator.

v0.x deferred items: a systematic survey of UUPS/Transparent/Beacon proxy deployments enumerating which proxies' initialisers make external calls and to which peer contracts; an OpenZeppelin `Initializable` contract improvement that allows an initialiser to specify a set of approved caller addresses for the initialisation context (so that a peer contract's callback is rejected unless the peer's address is in the approved set); a distinction between the reinitialisation class and the metamorphic-contract reinitialisation class (Create2 redeploy + reinit, which is a separate surface currently under-documented at v0.1).

T9.009 also is structurally adjacent to T9.004 (Access-Control Misconfiguration): in T9.004, the attacker exploits a missing or misconfigured access-control check on a privileged function; in T9.009, the attacker exploits the timing gap between the initialisation logic and the reinitialisation guard's state-flag commitment — the access-control check exists (the `initializer` modifier) but is not yet in effect at the moment of the callback. The mitigation lesson is that the initialisation guard should be set before any external call, not after all initialisation logic completes — the initialise-then-interact discipline applied at the modifier level.

Attribution-strength considerations: T9.009 attribution is typically at the `pseudonymous-attacker` strength level for operational exploits — the attacker address that submits the outer transaction is attributable at the on-chain identity layer, and the callback-to-initialize trace path is reproducible from the transaction trace. Pre-deployment audit findings are attributable to the protocol team (the deployer) rather than to an attacker; the distinction is important for OAK's attribution taxonomy.

T9.009 composes with T9.004 (Access-Control Misconfiguration) when the reinitialisation sets privileged roles that the protocol's access-control design did not intend to be settable by a reinitialisation path — the reinitialisation creates the privilege, and the missing access-control check permits its subsequent use. The two techniques are distinct in the exploit phase (T9.009 sets the privilege during init; T9.004 permits its subsequent abuse) but compose into a single exploit chain. The OAK convention classifies the load-bearing surface: if the initialisation path was re-entered to set the privilege, T9.009 is the primary technique; if the privilege was set legitimately but the privileged function lacked access control, T9.004 is the primary technique.
