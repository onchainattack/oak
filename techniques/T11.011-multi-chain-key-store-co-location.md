# OAK-T11.011 — Multi-chain Key-store Co-location

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure)
**Maturity:** emerging
**Chains:** chain-agnostic (the anti-pattern is an operator-side architectural decision, not a chain-specific vulnerability)
**First documented:** 2023-11-10 (Poloniex hot-wallet drain — simultaneous multi-chain extraction across ETH, TRX, BTTC, and others)
**Aliases:** "multi-chain key co-location", "shared signing-infrastructure compromise", "cross-chain hot-wallet co-location", "single-point-of-compromise multi-chain extraction"

## Description

An operator-side custody architectural anti-pattern in which an exchange, custodian, or protocol maintains **signing key material for multiple chains in a co-located infrastructure** — shared hardware, shared HSM/MPC context, shared key-management database, or shared signing-execution environment — such that a single compromise of that infrastructure produces simultaneous extraction across all supported chains. The attacker does not compromise multiple per-chain key stores; the attacker compromises one infrastructure component whose blast radius spans all chains the operator supports.

The structural feature is **key-store co-location**: the operator's custody architecture treats "multi-chain support" as a UI/application-layer feature while the signing keys for all chains share a single infrastructure layer. The attacker's extraction pattern — simultaneous or near-simultaneous outflows across multiple chains from the same operator — is itself the diagnostic signal that key material was co-located rather than per-chain isolated.

Distinct from sibling Techniques:

- **T11.001 (Third-Party Signing-Vendor Compromise)** — the vendor surface is breached. T11.011's compromise is of the operator's own co-located infrastructure; the vendor relationship is not the load-bearing variable.
- **T11.002 (Wallet-Software Distribution Compromise)** — the distribution channel is the attack surface. T11.011's surface is the operator's internal key-management architecture.
- **T4.001 (Private Key / Seed Phrase Compromise)** — the key material is compromised. T11.011 names the **architectural decision** that determines the blast radius of that compromise: co-location vs. per-chain isolation.
- **T15.003 (Operator Endpoint / Infrastructure Compromise)** — the entry vector. T11.011 is what the attacker finds once inside.

The pattern recurs across the OAK-G01 TraderTraitor cluster: Poloniex (November 2023, $120M), HTX (November 2023, $110M), Indodax (September 2024, $22M), Phemex (January 2025, $73M) — all share the simultaneous-multi-chain extraction signature indicating co-located key infrastructure.

## Observed indicators

- **Simultaneous multi-chain extraction:** the attacker drains assets across ETH, TRX, BSC, BTTC, and other chains within the same block window or within minutes — a pattern incompatible with per-chain isolated compromises, which would require sequential per-chain intrusions.
- **Single operator-side disclosure covering all affected chains:** the operator announces a breach affecting "hot wallets across multiple chains" — the disclosure language itself encodes the co-location architecture.
- **Laundering-cluster commonality across chains:** the extracted assets from different chains converge onto the same laundering infrastructure (same mixer addresses, same cross-chain bridge routes, same CEX deposit addresses) — indicating a single operator-controlled extraction, not multiple independent per-chain attackers.
- **No per-chain user-facing phishing or approval prompts:** the extraction occurs entirely at the operator's infrastructure layer; users do not receive phishing messages, approval requests, or any per-chain interaction. The absence of user-facing signals is itself a co-location diagnostic.

## Detection signals

- **Simultaneous-extraction cross-chain correlation (highest-leverage):** monitor for large outflows from exchange/custodian-labelled wallets across multiple chains within the same short time window (<1 hour). Multi-chain simultaneous extraction is a high-confidence indicator of co-located key infrastructure and a single-point compromise.
- **Operator-side key-isolation audit:** for every exchange, custodian, or protocol holding assets on multiple chains, audit whether signing key material for different chains is stored, managed, and accessed through per-chain isolated infrastructure (separate HSMs, separate MPC contexts, separate access-control policies) or through a shared infrastructure layer. Co-location is the T11.011 exposure surface.
- **Per-chain signing-infrastructure segmentation verification:** verify that the operator's signing infrastructure enforces per-chain segmentation at the cryptographic-architecture layer (separate key shares, separate signer sets, separate access policies), not merely at the application layer (UI that displays "Ethereum wallet" vs "Tron wallet" while both draw from the same key database).
- **Laundering-path cross-chain convergence monitoring:** when a multi-chain extraction occurs, monitor whether the extracted assets from different chains converge onto the same laundering infrastructure — this confirms single-operator extraction and strengthens attribution.

## Mitigations

- **Per-chain key isolation via cryptographic architecture (highest-leverage).** Exchange/custodian operators should enforce per-chain segmentation at the cryptographic-architecture layer: separate HSM partitions, separate MPC key shares, separate signer sets, and separate access-control policies for each supported chain. The goal is that compromising one chain's signing infrastructure does not grant access to any other chain's key material.
- **Per-chain access-control segmentation.** Even within a shared HSM/MPC infrastructure, access-control policies should be per-chain scoped: an operator credential that can authorise an Ethereum withdrawal should not also authorise a Tron or BSC withdrawal without an independent per-chain authorisation step.
- **Cross-chain extraction-correlation monitoring.** Third-party risk-monitoring services should maintain per-operator cross-chain extraction monitors that trigger alerts when multiple chains show simultaneous large outflows from the same operator's labelled wallets.
- **Operator-side architectural audit disclosure.** Exchanges and custodians should disclose, at minimum to their security auditors and ideally publicly, whether their signing infrastructure enforces per-chain key isolation or uses a co-located architecture. The disclosure enables user-side custody-risk assessment.

## Discussion

T11.011 names a custody architectural anti-pattern that has been present in exchange-infrastructure design since the multi-chain exchange era began but was not articulated as a discrete OAK Technique at v0.1. The pattern is structurally distinct from per-chain key compromise because the **operator's architectural decision** — to co-locate or isolate per-chain key material — determines the blast radius of any given infrastructure compromise, independent of the compromise vector.

The Poloniex November 2023 case is the canonical anchor because the simultaneous multi-chain extraction pattern is unambiguously documented across four independent forensic providers (Chainalysis, SlowMist, PeckShield, ZachXBT), the loss magnitude ($120M) is material, the attribution (OAK-G01 at `inferred-strong`) is robust, and the structural pattern — co-located hot-wallet infrastructure producing simultaneous multi-chain extraction — recurs across HTX, Indodax, and Phemex in the same 15-month cluster.

The technique is especially relevant to the OAK-G01 TraderTraitor cluster because the DPRK's exchange-compromise operational template appears to actively target co-located custody infrastructure: the simultaneous multi-chain extraction pattern is present across the majority of 2023–2025 G01-attributed exchange compromises, suggesting that G01 operators either (a) specifically target exchanges known to have co-located infrastructure, or (b) the co-location pattern is sufficiently widespread that any exchange compromise is likely to encounter it.

Promotion to `observed` requires at least one additional independent worked-example anchor beyond Poloniex — HTX (November 2023) is the closest pipeline target. Promotion to `stable` requires cross-operator pattern confirmation across at least three independent exchange operators, which the Poloniex/HTX/Indodax/Phemex cluster provides once per-incident worked examples are written up.
