# OAK-T11.008 — Embedded-Wallet Identity-Provider Compromise

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure)
**Maturity:** emerging
**Chains:** EVM (Polygon-resident Polymarket / Magic Labs canonical at v0.1; cross-chain analogues across Privy / Web3Auth / Dynamic deployments on Ethereum, Base, Arbitrum, Solana, others)
**First documented:** 2024-09 (Polymarket Magic-Labs takeover); cohort scale-out 2024–2026 across Polymarket-class platforms whose user onboarding runs through third-party email-auth / OAuth / MPC-social-login providers
**Aliases:** "Magic Labs takeover", "Privy / Web3Auth / Dynamic compromise", "embedded-wallet auth-provider compromise", "email-magic-link wallet hijack", "social-login wallet drain"

## Description

Compromise of the identity-provider layer of an *embedded wallet* — a wallet whose key derivation runs through a third-party authentication service (email magic-link, OAuth federation, MPC-backed social login). The on-chain wallet appears non-custodial from the protocol perspective; from the user's perspective, the access path is "log in with your email" or "log in with Google / Apple / X." The effective custody surface is the **auth provider's security posture**, not the user's hardware-wallet hygiene or the contract's correctness.

Vendor cohort at v0.1: Magic Labs, Privy, Web3Auth (now `tKey` / `Auth` / Torus), Dynamic, Particle Network, and adjacent embedded-wallet-as-a-service offerings. The on-chain transaction is correctly signed by the provider's key; the upstream compromise occurs at the email-auth, OAuth, or MPC-share layer.

T11.008 is **structurally distinct** from adjacent T11 sub-Techniques and access-acquisition Techniques:

- **T11.001** (Third-Party Signing-Vendor Compromise) — covers vendor-side UI / signing-flow compromise with organisational-customer victim shape (Bybit / Safe{Wallet}). T11.008's victims are *individual end-users* of the embedded-wallet platform, not the platform-operator.
- **T11.002** (Wallet-Software Distribution Compromise) — substitutes the wallet binary on a legitimate device. T11.008 does not require any wallet-software compromise; the wallet derivation runs entirely through the auth provider's hosted infrastructure.
- **T11.006** (Cold-storage Seed-phrase Exfiltration at Rest) — third-party storage-service-at-rest failure surface (LastPass / iCloud-MetaMask). T11.008 does not require a stored seed phrase; the user typically *has no seed phrase* in the embedded-wallet model.
- **T4 phishing** — phishing campaigns that exploit embedded-wallet auth flows are upstream of T11.008 (e.g., the [`examples/2025-11-polymarket-comment-section-phishing.md`](../examples/2025-11-polymarket-comment-section-phishing.md) campaign captures email-magic-link credentials to exploit the Magic Labs auth path). T11.008 is the *custody surface* the phishing chains into; the phishing itself is T4.

The structural feature is **the widespread "non-custodial" claim collapses back to "as custodial as the auth provider's security posture."** The user's mental model of self-custody is constructed around the on-chain wallet's contract-level non-custodial properties; the operational reality is that the access path is mediated through a third-party identity provider whose compromise produces wallet access without any on-chain action required.

## Observed indicators

- **User-onboarding flow:** the platform's signup is "log in with email" / "log in with Google / Apple / X" / "log in with phone number" rather than "connect wallet." A non-custodial wallet is auto-derived as part of the auth flow.
- **No user-controlled seed phrase:** the user is never shown a seed phrase / mnemonic / private key for the auto-derived wallet; key derivation runs through the auth provider's hosted infrastructure.
- **Auth-provider attribution:** post-incident, the platform attributes the compromise to "abuse of a third-party authentication tool" / "third-party login tool" / "third-party auth service" — the load-bearing platform-side framing.
- **On-chain transaction signature:** the on-chain action is a *legitimate signed transaction* by the auth-provider-managed key; on-chain forensics surface no signing-flow anomaly because the auth provider's signature is valid.
- **Cohort fingerprint:** affected users share the embedded-wallet platform but have no other commonality (no on-chain phishing signature, no compromised wallet binary, no SIM-swap chain).

## Detection signals

- **Auth-provider-side telemetry:** unusual login patterns at the email-magic-link / OAuth layer (geographic anomalies, device anomalies, rate-limit triggers); session-token anomalies; multi-device session reconciliation failures. The detection surface lives at the auth provider's infrastructure, not at the on-chain runtime.
- **Platform-side audit:** integration-time review of the auth provider's security posture (SOC-2 / equivalent attestations; hardware-backed multi-factor authentication offering; recovery-flow design that limits social-engineering surface).
- **Cross-platform wave detection:** when multiple embedded-wallet platforms share the same auth provider, a compromise wave at one platform is a leading indicator for the others. The Polymarket September 2024 + December 2025 incidents (both attributed to "third-party login tool") plus the broader 2024–2026 cohort of Magic Labs / Privy / Web3Auth-class compromises form the cohort-detection surface.
- **User-side awareness:** users with material holdings on embedded-wallet platforms should treat the email-auth credential as the load-bearing security primitive — strong unique passwords, hardware-backed two-factor authentication, awareness of phishing flows that solicit the magic-link verification code.

## Real-world examples

- **Polymarket account takeover via third-party auth — Polygon — 2024-09 (canonical T11.008 case)** — Loss undisclosed; Polymarket characterised the affected set as "a small number of users." Attack vector: abuse of Magic Labs (the third-party email-auth provider used to derive non-custodial wallets for Polymarket users). Funds held on-chain in a smart-contract wallet but access mediated through a Magic-Labs-managed key derived from the email-auth flow. Attacker called `proxy()` on the Magic-derived wallets, transferring USDC to attacker-controlled addresses. Polymarket publicly attributed the incident to abuse of a third-party authentication tool. Attribution: `inferred-strong`; specific operator attribution not published. The September 2024 incident shares structural pattern with the December 2025 Polymarket account-breach wave (also attributed to third-party auth) and with the broader 2024–2026 cohort of Magic Labs / Privy / Web3Auth-class compromises. See [`examples/2024-09-polymarket-magic-labs-takeover.md`](../examples/2024-09-polymarket-magic-labs-takeover.md).
- **Polymarket comment-section phishing campaign — Polygon — 2025-11 (T11.008-chained T4 case)** — ≥ $500,000 cumulative loss across affected users. The attack chain: T4 phishing (comment-pinning on engagement-weighted platform; cloned login page captures email + magic-link verification code) → T11.008 (attacker authenticates to Magic Labs as the user; gains access to the user's Polymarket-derived non-custodial wallet; transfers USDC). The case is the canonical illustration of T4 phishing chaining into T11.008 — the embedded-wallet identity-provider is the custody surface that the phishing exploits. See [`examples/2025-11-polymarket-comment-section-phishing.md`](../examples/2025-11-polymarket-comment-section-phishing.md).

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T11.008's primary detection surface is at the auth provider's infrastructure (login patterns, session-token anomalies, recovery-flow integrity), not the on-chain runtime risk-engine surface MG covers.
- Vendor coverage outside MG: auth-provider-side telemetry (Magic Labs / Privy / Web3Auth / Dynamic internal monitoring); platform-side post-incident audit (Polymarket's third-party-auth attribution and remediation guidance); industry-press cohort framing.

## Mitigations

- **Platform-side (highest-leverage):** integration-time security review of the auth provider; SOC-2 / equivalent attestation requirement; hardware-backed multi-factor authentication offering as a configurable option for end-users with material holdings; recovery-flow design that limits social-engineering surface (avoiding email-only recovery; requiring multi-channel verification for sensitive actions).
- **Auth-provider-side:** unusual-login-pattern detection at the email-magic-link / OAuth layer; session-token rotation and anomaly detection; recovery-flow hardening; bug-bounty programme for auth-provider-specific compromise paths.
- **User-side:** treat the email-auth credential as the load-bearing security primitive — strong unique passwords for the email account underpinning the magic-link flow, hardware-backed two-factor authentication on the email account, awareness of phishing flows that solicit the magic-link verification code.
- **Disclosure invariant:** platforms using embedded-wallet identity-providers should disclose the dependency to users and explicitly state that the access path's security is bounded by the auth provider's posture. The "non-custodial" claim should be qualified with the auth-provider dependency rather than left ambiguous.
- **Cross-platform coordination:** when an auth-provider compromise wave is observed at one platform, dependent platforms should treat it as a leading indicator and run pre-emptive session invalidation / multi-factor enrollment campaigns across their user base.

OAK Mitigation cross-references at v0.1 are a **gap** — no existing OAK-M* entry directly maps to the embedded-wallet auth-provider security surface. v0.x candidate Mitigation entries: "OAK-M-x embedded-wallet auth-provider integration audit"; "OAK-M-x embedded-wallet hardware-MFA opt-in"; "OAK-M-x platform-side disclosure of auth-provider dependency".

## Discussion

T11.008 closes a structural gap in OAK v0.1's T11 coverage. The dominant v0.1 T11 worked examples assume an on-chain signing key under the user's direct control; T11.008 introduces a custody substrate where the on-chain signing key is *managed by a third-party identity provider* whose compromise produces wallet access without any on-chain action required.

The structural OAK lesson is that **the "non-custodial" claim is a compositional property, not an absolute one**. A wallet whose contract-level properties are non-custodial may have an access path mediated through a custodial-equivalent identity provider; the effective custody surface is the union of (a) the on-chain contract's properties and (b) the auth provider's security posture. Platforms using embedded-wallet identity-providers should disclose this composition to users explicitly.

The cohort-level pattern at v0.1 is dominated by Polymarket / Magic Labs cases. The structural pattern generalises beyond Polymarket: any platform whose user onboarding runs through Magic Labs / Privy / Web3Auth / Dynamic / Particle Network is exposed to the same auth-provider-compromise surface. The 2024-09 and 2025-12 Polymarket incidents are bracketed by a broader 2024–2026 cohort of embedded-wallet-platform compromises; pipeline anchor cases for v0.x include other Polymarket-class platforms experiencing auth-provider-compromise events.

The T4-T11.008 attack chain is structurally important. T4 phishing campaigns that target embedded-wallet platforms (e.g., the [`examples/2025-11-polymarket-comment-section-phishing.md`](../examples/2025-11-polymarket-comment-section-phishing.md) campaign) capture the email-magic-link credentials at the T4 layer and exploit them at the T11.008 layer. Detection at the T4 layer is necessary but not sufficient; defender investments must also harden the T11.008 layer (auth-provider posture, recovery-flow integrity, session-anomaly detection).

The auth-provider concentration risk is structural. A small number of embedded-wallet-as-a-service vendors (Magic Labs, Privy, Web3Auth, Dynamic) provide identity-provider infrastructure to a large fraction of the embedded-wallet ecosystem. A compromise wave at one provider exposes every dependent platform's user base simultaneously — an OAK-T8.002-adjacent operator-continuity surface at the auth-provider layer rather than at the on-chain operator layer.

Promotion to `stable` will follow once additional auth-provider-cohort cases land (Privy / Web3Auth / Dynamic incidents with public forensic disclosure) and once the cross-platform cohort beyond Polymarket is anchored.
