# OAK-T11.012 — Server-side Raw Private-Key Storage (Custodial Trading-Bot Anti-pattern)

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure)
**Maturity:** emerging
**Chains:** chain-agnostic (the anti-pattern is an operator-side architectural decision, not a chain-specific vulnerability)
**First documented:** 2024-09 (DEXX trading-bot platform cohort; the pattern predates DEXX but DEXX is the first cleanly-documented OAK worked example)
**Aliases:** "custodial-private-key-storage anti-pattern", "server-side raw-key storage", "trading-bot key-storage compromise", "raw-private-key-holding platform"

## Description

An operator-side architectural anti-pattern in which a trading-bot platform, automated-execution service, or copy-trading platform stores users' **raw (unencrypted) private keys server-side** in order to execute transactions on the user's behalf without requiring per-trade user-side signing. When the operator's server-side infrastructure is compromised — via database breach, API key leak, cloud-credential compromise, or insider access — the attacker gains direct access to the raw signing keys of every active user and can drain wallets without any user-facing interaction.

The structural failure is not the infrastructure compromise (which is the entry vector, covered by T15.003) but the **architectural decision to hold raw private keys server-side**. That decision converts an operator-infrastructure intrusion into a mass-user-asset-loss event: instead of compromising one user at a time via per-user phishing (T4), the attacker compromises one operator database and exfiltrates every active user's private key in a single operation.

Distinct from sibling Techniques:

- **T11.001 (Third-Party Signing-Vendor Compromise)** — the vendor's signing flow is breached. T11.012's operator is not a signing vendor; it is a trading-bot platform that chose to hold raw keys for operational convenience.
- **T11.002 (Wallet-Software Distribution Compromise)** — the wallet binary is substituted. T11.012 involves no wallet software distribution; the user voluntarily provided their private key to the platform.
- **T4.001 (Private Key / Seed Phrase Compromise)** — the key material is compromised. T11.012 names the **architectural decision** that makes mass compromise possible: server-side raw-key storage vs. non-custodial signing architectures.
- **T11.011 (Multi-chain Key-store Co-location)** — the operator's own infrastructure keys are co-located. T11.012's co-location is of **user** keys, not operator keys, and the compromise surface is the user-facing platform database rather than the operator's treasury infrastructure.

The user-side failure is also structural: users who provide raw private keys to a trading-bot platform have opted out of the self-custody security model without receiving the custodial-regulatory protections that a licensed custodian would provide. The platform is "custodial in practice, unregulated in law" — the worst of both security models.

## Observed indicators

- **Server-side raw-key storage disclosure (or absence of denial):** the platform's documentation does not explicitly state that user keys are held in non-custodial form (e.g., encrypted client-side, never transmitted to server). The absence of a clear non-custodial architecture statement is itself a signal.
- **No per-trade user-side signing requirement:** the platform executes trades without requiring the user to sign each transaction — the platform holds sufficient key material to sign autonomously. This is the user-observable fingerprint of server-side raw-key storage.
- **Mass simultaneous user-wallet draining:** a single platform security incident produces simultaneous draining across many user wallets, with no per-user phishing or approval interaction. The mass-simultaneous pattern is incompatible with per-user compromise and indicates a centralised key store.
- **Platform's incident disclosure frames the loss as "user wallet drains" rather than "platform treasury drain":** the operator's own funds may be unaffected; the compromised database contained user keys, not operator keys. The disclosure language distinguishes T11.012 from T11.001/T11.011.

## Detection signals

- **Pre-incident: platform architectural audit.** Security auditors and due-diligence providers should verify whether a trading-bot or automated-execution platform holds user private keys server-side. Platforms that do should be flagged as T11.012-exposed regardless of their current security posture — the architectural decision is the vulnerability.
- **Pre-incident: user-side "test withdrawal" invariant.** Users considering a trading-bot platform should verify whether the platform requires per-trade signing or holds their raw key. Any platform that holds raw private keys server-side should be treated as a custodial counterparty, not a tool — the user is exposed to the platform's entire infrastructure security posture.
- **At-incident: mass wallet-drain correlation.** When many user wallets associated with the same platform drain simultaneously without per-user phishing signals, the incident is a high-confidence T11.012 event. The mass-simultaneous pattern distinguishes T11.012 from per-user T4 phishing cohorts.
- **Post-incident: forensic confirmation of centralised key storage.** Post-incident forensic analysis should confirm whether the platform held raw private keys in a centralised database. The forensic finding is the definitive confirmation signal.

## Real-world examples

- **DEXX trading-bot platform — Solana + EVM — 2024-09 to 2024-11 (canonical T11.012 anchor).** DEXX held users' raw private keys server-side to execute automated trades without per-trade user-side signing. A series of operator-side database compromises across September–November 2024 exfiltrated the raw signing keys of all active users, producing cumulative drains of ~$2.5M+ across many user accounts. DEXX publicly disclosed the breach progression, partially reimbursed verified affected users, and committed to migrating to a non-custodial signing architecture. See [`examples/2024-09-dexx-cohort.md`](../examples/2024-09-dexx-cohort.md).

### Additional incidents (pipeline anchors)

- **Banana Gun — September 2024.** Trading-bot platform information-leak incident with a different mechanism (information leakage rather than raw-key database compromise) but the same structural surface: a trading-bot platform whose user-data storage architecture converted an operational incident into user-asset losses. Pipeline anchor for the broader trading-bot-platform-as-attack-surface class.
- **Unibot — October 2023.** Trading-bot platform exploit via contract vulnerability (T9) rather than key-storage compromise, but the same user-base and platform category. Demonstrates the broader trading-bot attack surface beyond the T11.012 server-side-key-storage subclass.

## Mitigations

- **Non-custodial signing architecture (highest-leverage).** Trading-bot and automated-execution platforms should use non-custodial signing architectures where the user's private key never leaves the user's client: per-trade user-side signing, session-key delegation with time-and-scope bounds (T13.003), or MPC-based distributed signing where the platform holds only a partial key share.
- **Platform architectural disclosure requirement.** Trading-bot platforms should be required to disclose, in user-facing language, whether they hold user private keys server-side. The disclosure should be prominent, not buried in ToS — the user's decision to provide a raw private key is a custody decision that should be made with full information.
- **User-side "never share your raw private key" hygiene.** User education should explicitly name trading-bot platforms as a category that should not receive raw private keys. The rule "never share your seed phrase" should be extended to "never share your raw private key with a trading-bot platform that holds it server-side."
- **Regulatory classification of raw-key-holding platforms as custodians.** Jurisdictions should classify platforms that hold user raw private keys server-side as custodial financial intermediaries subject to the same regulatory capital, segregation, audit, and cybersecurity requirements as licensed custodians.

OAK Mitigation cross-references at v0.1: gap. T11.012's mitigation surface is at the platform architectural design and regulatory-classification layers. No existing OAK-M* entry directly maps to non-custodial signing architecture requirements for trading-bot platforms. v0.x candidate Mitigation entry.

## Discussion

T11.012 names a structural anti-pattern that sits at the intersection of custody architecture and user security hygiene. The trading-bot platform category — Telegram bots, web-based trading terminals, copy-trading platforms — grew rapidly in the 2023–2025 retail-trading cycle, and many platforms adopted server-side raw-key storage as the simplest path to automated execution. The architectural convenience of server-side key storage (no per-trade user friction, no session-key infrastructure to build, no MPC to coordinate) directly produces the T11.012 mass-compromise surface.

The user-side dimension is equally structural. Users who provide raw private keys to a trading-bot platform have made a custody decision — they have delegated signing authority to a third party — often without the information or framing that this is a custody decision. The platform presents itself as a "tool" or "bot" (low-trust terminology) while operating as a custodian (high-trust function). The terminology gap — "bot" vs "custodian" — is the user-side deception surface that T11.012 surfaces.

The DEXX case is the canonical anchor because it is the cleanest documented instance of the pattern: the platform explicitly stored raw private keys server-side, the compromise was of the operator's database infrastructure, the mass-simultaneous draining pattern confirmed centralised key storage, and the platform's post-incident commitment to non-custodial architecture is the explicit acknowledgment that the server-side raw-key model was the root cause.

Promotion to `observed` requires at least one additional independent worked-example anchor beyond DEXX. The pipeline is thinner than for T11.011 because trading-bot platforms with server-side raw-key storage are a smaller and newer category than multi-chain exchanges; additional anchors are expected as the category matures and as forensic analysis of trading-bot incidents explicitly attributes loss to server-side key storage architecture.
