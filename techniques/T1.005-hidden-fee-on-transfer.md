# OAK-T1.005 — Hidden Fee-on-Transfer

**Parent Tactics:** OAK-T1 (Token Genesis), OAK-T6 (Defense Evasion)
**Maturity:** stable
**Chains:** EVM (primary), Solana (SPL via Token-2022 transfer-fee extension; secondary)
**First documented:** widespread from approximately 2020 onward
**Aliases:** "sell tax", "anti-bot tax", "asymmetric fee", "router-only fee", "honeypot-lite", "conditional fee-on-transfer"

## Description

A class of OAK-T1 patterns in which a token contract embeds a conditionally-triggered fee in its transfer logic at deployment — the fee is *not* changed post-launch, but is non-uniformly applied based on caller, counterparty, transfer size, or block / timestamp predicates. From a defender's perspective, T1.005 is structurally distinct from T1.001 (Modifiable Tax Function) along two axes: (a) the fee parameter is fixed in immutable bytecode (so naive "is the tax modifiable?" checks pass), and (b) the failure mode is the *predicate* gating the fee path rather than the fee value itself. T1.005 is also distinct from T1.004 (Blacklist / Pausable Transfer Weaponization) because the failure mode is *gradient* (extractive cost on a successful transfer) rather than *binary* (transfer reverts).

Common sub-patterns observed at the cohort scale:

- **Sell-only fees** — transfers whose `to` address is an AMM pool / router contract incur a fee; transfers between EOAs do not. Naive simulation on EOA-to-EOA transfers shows zero fee and the contract appears benign.
- **Threshold-triggered fees** — transfers above N% of supply (or above an absolute token amount) incur a fee; small simulation amounts pass through cleanly.
- **Time-windowed fees** — fees apply only within the first M blocks / seconds after deployment ("anti-bot"), or, inversely, fees activate after a launch grace window during which the contract appears clean.
- **Caller-specific exemptions** — a hard-coded allowlist (often the deployer cluster and an internal market-maker address) is exempted from the fee, allowing privileged extraction without paying the same cost imposed on retail holders.

Because the fee is static code, T1.005 is detectable at deployment time via static analysis of the transfer-logic predicates plus dynamic simulation across a synthesised matrix of caller / counterparty / size / time conditions. A single-condition simulation (the default in most commodity vendor scanners at v0.1) is insufficient.

## Observed indicators

- A `_transfer` (or equivalent) function whose fee branch is gated by a predicate involving (`to == pair`), (`from == pair`), (`amount > threshold`), (`block.number < launchBlock + N`), or membership in a hard-coded `_isExcludedFromFee` mapping initialised at construction.
- An `_isExcludedFromFee` (or analogously-named) mapping populated in the constructor with the deployer EOA, the deployer cluster, and / or unattributed addresses, with no public function to add others.
- Asymmetric fee values across branches (e.g. buy-tax = 0%, sell-tax = 10–25%) embedded as immutable constants.
- Cross-token: deployer cluster previously deployed contracts with the same conditional-fee predicate structure (frequent overlap with T1.001 and T1.004 across the same operator's deployments).

## Detection signals

- Static (pre-trade): enumerate every fee-branch predicate on the transfer-logic path; flag any predicate keyed on caller / counterparty address, transfer size, or block / timestamp; flag any constructor-populated exemption mapping.
- Dynamic (pre-trade): simulate transfers across a matrix of (small / medium / large) amounts × (EOA-to-EOA / EOA-to-pool / pool-to-EOA) directions × (early / late) block conditions; flag any simulation cell where the executed fee deviates from the advertised nominal fee.
- Cross-token: cluster contracts by deployer fingerprint; if a deployer's prior tokens combined T1.005 with subsequent T5.001 (Hard LP Drain) or T5.006 (Vesting-Cliff Dump) extraction events, treat new deployments from the same deployer as elevated risk regardless of source-verification claims.
- Cross-pattern: T1.005 frequently co-occurs with T6.001 (Source Verification Mismatch) — verified source is a clean ERC-20, deployed bytecode contains the conditional-fee branch.

## Real-world examples

- **Cohort scale — multi-chain — 2020 onward** — T1.005 is most visible at the cohort level rather than at the per-incident level. Per token-risk-scanner cohorts (Token Sniffer, GoPlus, RugCheck, Honeypot.is), the conditional / asymmetric fee pattern recurs across thousands of memecoin-class deployments per quarter, and is one of the highest-recall single signals these scanners surface ("Sell tax: X%", "Buyer-only fee", "Honeypot-like behavior detected"). Industry retrospectives (`[chainalysis2025rug]`, `[slowmist2024report]`) cover the cohort-scale impact without naming individual T1.005 cases as canonical because the publicly-reportable framing is typically the downstream T5 extraction event rather than the upstream fee primitive.
- **Per-incident named cases** — OAK does not at v0.1 name a single canonical T1.005 incident. The pattern is most often surfaced as an artefact of post-mortem token-contract review on rug-pull-class events whose primary OAK Technique is T5.001 (Hard LP Drain) or T1.001 (Modifiable Tax). Contributors with public forensic write-ups of specific named incidents whose primary failure mode is T1.005 (rather than T5 / T1.001 with T1.005 as a contributing primitive) are encouraged to submit them via PR.

- `mg-detectors-rs` — coverage **partial** at v0.1 (single-condition fee detection in scope; multi-axis simulation matrix and constructor-exemption-mapping enumeration in progress).
- Vendor coverage: Token Sniffer, GoPlus, RugCheck, Honeypot.is (per-token scam scanners surface sell-tax / asymmetric-fee detections; matrix-coverage and exemption-mapping enumeration vary by vendor).

## Mitigations

- **Traders / risk teams:** require pre-trade simulation across a *matrix* of caller / counterparty / size / time conditions rather than a single-cell simulation; flag any token whose executed fee differs across the matrix as elevated risk regardless of advertised nominal fee.
- **Static-analysis tooling:** enumerate all fee-branch predicates on the transfer-logic path; enumerate all constructor-populated exemption mappings; surface both as machine-readable risk attributes (not free-text warnings).
- **Token vendors / launchpads:** require either (a) immutable transfer logic with a single, uniform fee path (no caller / counterparty / size / time gating), or (b) explicit declaration of all fee branches and exemption-mapping members as a listing precondition, with on-chain verifiability.
- **Wallet-vendor pre-trade UX:** surface "asymmetric fee" / "exemption mapping present" / "size-gated fee" presence in pre-trade risk warnings; do not collapse to a single nominal-fee number.

**Canonical mitigations:** OAK-M01, OAK-M02, OAK-M23, OAK-M25.

## Discussion

T1.005 is a commodity-grade pattern at the cohort level (high prevalence in token-risk-scanner per-token output) but underweighted relative to T1.001 and T1.004 in industry retrospectives, because its failure mode is gradient rather than binary and is therefore frequently absorbed into the downstream extraction-event framing (T5.001, T5.006) rather than reported as a distinct primitive.

Open questions for v0.2:

- Should OAK distinguish "anti-bot" time-windowed fees with a hard-coded short window (e.g. < N blocks, fee permanently zero thereafter) from extractive time-windowed fees? The static signature is identical; the operator intent is not. A `Maturity: anti-bot-window` sub-classification (analogous to the `governance-gated` proposal under T1.001) may be warranted.
- Should constructor-populated exemption mappings be promoted to a distinct sub-Technique (T1.005.a)? They are the operator-extraction primitive that makes the otherwise-symmetric-looking fee asymmetric in practice, and may deserve standalone visibility.
- How should T1.005 in Token-2022 (Solana SPL `TransferFeeConfig` extension) be modelled? The extension exposes a fee-config authority and a withdraw-withheld authority; the EVM-style "static code, conditionally applied" framing maps imperfectly. Cross-reference with T1.002 (Token-2022 Permanent Delegate) tactic-overlap deferred to v0.2.
