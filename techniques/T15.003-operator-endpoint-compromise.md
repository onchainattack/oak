# OAK-T15.003 — Operator-Endpoint Compromise (Developer Workstation / Signing Machine)

**Parent Tactics:** OAK-T15 (Off-chain Entry-Vector / Pre-Positioning)
**Maturity:** emerging
**Chains:** chain-agnostic (the endpoint compromise is off-chain; downstream on-chain manifestation is per-incident)
**First documented:** systematically 2017 onward (Bithumb employee-laptop with customer-data DB as the canonical pre-2020 anchor); the developer-workstation / signing-host sub-shape stabilised across the 2024-2025 OAK-G01 wave
**Aliases:** "developer workstation compromise", "signing-host compromise", "MITM on signing host", "INLETDRIFT-class macOS implant", "signing-machine takeover", "employee endpoint compromise"

## Description

Direct compromise of an operator-controlled endpoint — a developer workstation, a signing machine (the host that holds private-key material or relays signed transactions to the broadcast channel), an internal admin console, or an employee laptop with privileged access to customer data. T15.003 is the **endpoint-state** sub-Technique: the proximate enabler of a downstream T11.x / T9.x / T10.x event is that the operator's machine is in a state where its signing flow, transaction-broadcast pipeline, deployment artefacts, or data-at-rest can be manipulated by the attacker.

T15.003 is **distinct from T15.001 (the social-engineering vector)** even though the two are typically chained. T15.001 is the cause; T15.003 is the resulting endpoint state. They are split because the mitigation surface differs:

- T15.001 mitigation lives at the **anti-phishing / awareness / pretext-recognition** layer (OAK-M21).
- T15.003 mitigation lives at the **endpoint-hardening / signing-host-architecture / EDR / MDM / air-gap** layer (OAK-M19, EDR / MDM controls, signing-host segmentation).

A complete chain typically reads T15.001 → T15.003 → T11.x / T9.x / T10.x. But T15.003 can also originate without T15.001 — opportunistic infostealer infections, drive-by exploits, physical access — and a worked example may be primarily T15.003 even when no clear T15.001 pretext exists.

T15.003 covers three structurally adjacent endpoint sub-shapes:

- **Developer workstation in a state that compromises future signing or deployment.** The classic case: implant established (INLETDRIFT, BeaverTail-stage-2, RustBucket, KandyKorn), MITM layer between the wallet UI and the broadcast channel, or browser extensions that exfiltrate sessions. Canonical anchors: Radiant Capital (October 2024 — INLETDRIFT macOS implant on multiple developer machines, MITM modification of transaction payload between Tenderly-simulation review and broadcast); Concentric Finance (January 2024 — multisig-signer dev-machine compromise harvesting private-key material).
- **Signing-machine compromise** in the narrower sense — a host whose specific function is to hold key material or relay signed transactions. The Bybit / Safe{Wallet} case is anchored at this sub-shape: the Safe{Wallet} developer's workstation (the host that built the signing UI Bybit's signers loaded) was the compromised endpoint.
- **Employee laptop with access to non-signing privileged data.** Customer-data databases, internal admin consoles, IDP / SSO sessions. Canonical anchor: Bithumb (June 2017 — employee laptop with customer-data DB extracted, used downstream to phish customers in the second-stage operation). The class extends to operator-side credential / cookie / session-token theft on endpoints whose proximate function isn't signing but whose state controls a signing or auth surface.

## Observed indicators

- **Off-chain (endpoint forensics):** EDR / MDM detection of unsolicited code execution; persistence implants in standard macOS / Windows / Linux footholds; MITM proxy / man-in-the-browser layer between wallet UI and broadcast endpoint; browser extensions exfiltrating session cookies / OAuth tokens; second-stage tooling whose teardown is anti-forensic (canonical: Radiant's three-minute teardown of second-stage tooling post-extraction).
- **Off-chain (operator-side):** post-event endpoint forensic engagement (Mandiant, Stroz Friedberg, ZeroShadow, SEAL 911) confirming the implant chain; implant-family attribution (INLETDRIFT, BeaverTail / InvisibleFerret, KandyKorn, RustBucket, ObjCShellz, SwiftLoader).
- **On-chain (downstream):** the on-chain transaction signed by the operator differs from what the operator's UI / simulation / signing personnel report having approved (Bybit / Radiant signature); deployment of a malicious upgrade against contracts whose admin key lived on a compromised machine (Concentric); admin / deployer key activity inconsistent with the operator's normal cadence (Ankr Dec 2022, deployer-key compromise via host).
- **Forensic record:** the implant binary / sample (where shared with researchers); the on-host persistence path; the MITM layer's network footprint; any preserved second-stage payload.

## Detection signals

- **EDR / MDM on operator-privileged endpoints** — mandatory for any host that holds key material, deploys contracts, signs transactions, or has access to multisig signing UIs.
- **Signing-host segmentation (OAK-M19)** — air-gap signing for cold-wallet / treasury / multisig operations; the host displaying the Tenderly simulation must not be the same host whose RPC connection is used to broadcast.
- **Independent destination verification at sign time (OAK-M18)** — read the destination address from a hardware-wallet display, not from the wallet UI; this is the canonical mitigation when the host is in a T15.003 state.
- **Implant-family detection** — subscribe to Mandiant / Microsoft / SentinelOne / Jamf / Elastic / Unit 42 advisories on active DPRK and other state-aligned implant families targeting crypto operators; EDR signatures + MITRE ATT&CK technique mapping.
- **Out-of-band confirmation of signed transactions** — for high-value operations, require multi-channel confirmation that the broadcast transaction matches what the signer reviewed, with the confirmation channel independent of the signing host.
- **Anti-forensic posture** — assume successful T15.003 attackers will tear down second-stage tooling; preserve EDR / MDM telemetry continuously rather than rely on on-host artefacts being available post-event.

## Real-world examples

- **Bybit / Safe{Wallet} — Ethereum — 2025-02-21** — Loss \~\$1.46B. The proximate enabler of the on-chain extraction was a compromised Safe{Wallet} developer workstation: malicious JS planted in the signing UI altered the destination address at sign time. The cryptographic-signing layer was intact; the endpoint state at the vendor's developer was the load-bearing failure. See [`examples/2025-02-bybit.md`](../examples/2025-02-bybit.md).
- **Radiant Capital — Arbitrum / BNB Chain — 2024-10-16** — Loss \~\$50M. Multiple Radiant developer macOS workstations in a compromised state via INLETDRIFT implant; MITM layer modified the transaction payload between the signer's wallet-UI / Tenderly-simulation review and the broadcast channel. The Mandiant on-device forensic engagement was the load-bearing step in establishing the developer-laptop sub-vector. The canonical T15.003 anchor for the developer-workstation-with-MITM-on-signing sub-shape. See [`examples/2024-10-radiant-capital.md`](../examples/2024-10-radiant-capital.md).
- **Concentric Finance — Arbitrum — 2024-01-22** — Loss \~\$1.7M. Multisig signer's developer machine compromised; payload harvested private-key material from the local wallet-software environment, converting the social-engineering entry into private-key extraction. The canonical T15.003 anchor for the small-multisig dev-machine sub-shape. See [`examples/2024-01-concentric.md`](../examples/2024-01-concentric.md).
- **Bithumb — multi-asset — 2017-06-29** — Loss \~\$1M+ direct (with broader downstream cohort impact). Employee laptop with access to customer-data database compromised; data exfiltration enabled second-stage targeted phishing of customers. The canonical T15.003 anchor for the **employee-laptop-with-non-signing-data sub-shape** and the earliest T15.003 anchor in OAK at v0.1. See [`examples/2017-06-bithumb.md`](../examples/2017-06-bithumb.md).
- **Liquid Global / QUOINE PTE — multi-chain — 2021-08-19** — Loss \~\$91M. The public forensic record at v0.1 places the case inside the broader DPRK / TraderTraitor 2021 cohort but does not pin the exact off-chain entry vector to a single sub-Technique — Chainalysis and Elliptic published cluster-level wallet attribution; the FBI / CISA / Treasury joint advisory (April 2022) consolidates DPRK-cluster attribution at the cohort level without naming Liquid specifically. The case anchors the *T15 cohort-class mapping convention* OAK uses when the operator-cluster attribution is strong but the per-incident sub-Technique is not pinned in the public record. ## Mitigations

- **OAK-M19 (Air-gap cold-wallet signing):** highest-leverage layer for treasury / cold-wallet operations — the host that signs must be independent of the host that displays the simulation and the host that broadcasts. Defeats MITM-on-signing-host (the Radiant pattern).
- **OAK-M18 (Out-of-band destination verification):** read the destination address from a hardware-wallet display, not from the signing-host UI, before approving high-value transactions. The Bybit / Radiant lesson restated.
- **EDR / MDM on operator-privileged endpoints:** mandatory baseline; without endpoint telemetry, post-event forensic reconstruction of T15.003 is significantly harder (Radiant's anti-forensic teardown demonstrates the importance).
- **Signing-host segmentation:** the host that holds key material or signs production transactions should not be the developer's day-to-day workstation; physical / virtual segmentation between developer environments and signing environments closes the most common T15.003 → T11.x chain.
- **OAK-M22 (Rotate-on-disclosure):** when an endpoint is suspected compromised, treat all credentials, sessions, and signing material on that endpoint as compromised; rotate before evaluating impact, not after.
- **Continuous EDR / MDM telemetry preservation:** collect and externalise EDR data continuously; do not rely on on-host log preservation (assume successful attackers will tear down second-stage tooling).

## Discussion

T15.003's structural lesson is that the **defender signal lives in endpoint telemetry, not at the on-chain layer**. The Radiant case is canonical here: the signers followed standard procedure (Tenderly simulation, hardware-wallet signing, payload-data verification) and the breach succeeded in spite of those procedures, not because they were skipped — because the host displaying the simulation was the same host whose RPC connection was compromised. The mitigation surface is structurally different from on-chain detection: by the time the malicious transaction hit the RPC endpoint, the attacker had already won.

T15.003 is also the sub-Technique that most clearly **separates supply-chain attacks from end-user phishing**. End-user phishing (T4) compromises a user's signing flow at the user's wallet UI; T15.003 compromises an operator's signing flow at the operator's signing host, and the dollar impact distributes across the operator's customer base. Defenders evaluating supply-chain risk in 2024 generally enumerated "third-party signing platform" (Bybit pattern) and "third-party wallet-software vendor" (DMM / Ginco) but rarely enumerated "our own developers' laptops" — the Radiant case closed that gap in the public record.

The boundary against T15.001 is causal-direction (vector → endpoint state). The boundary against T15.002 is the **propagation-shape**: T15.003's effect is bounded to what the compromised endpoint can do (sign, broadcast, deploy, exfiltrate); T15.002's effect multiplies across every consumer of the shared upstream surface. A T15.003 endpoint at a vendor that pushes a malicious commit produces both T15.003 (the endpoint) and T15.002 (the publication); the worked example should map both stages.

False-positive considerations: legitimate developer-machine activity routinely includes signing transactions, deploying contracts, running unsolicited code in evaluation environments. The detection signal is not "the endpoint signed something" but "the endpoint signed something that diverges from what the user-facing UI / simulation displayed" or "the endpoint deployed a contract whose admin-key surface diverges from what was reviewed." This is one of the reasons the canonical mitigation is OAK-M18 / M19 (out-of-band verification + air-gap), not on-host detection alone.

v0.x deferred items: hardware-wallet host-bypass cases where the hardware-wallet itself is the implicit-trust anchor but the host between the user and the hardware-wallet is what has been compromised (boundary-case against the user-side T11 sub-classes); treatment of operator-side mobile-device compromise as a distinct sub-shape from desktop-workstation compromise; insider-action cases where the endpoint state was deliberately compromised by an authorised insider (which structurally overlaps with T15.003 but has a different mitigation surface).
