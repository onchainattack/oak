# OAK-T1.004 — Blacklist / Pausable Transfer Weaponization

**Parent Tactics:** OAK-T1 (Token Genesis), OAK-T6 (Defense Evasion)
**Maturity:** stable
**Chains:** EVM (primary)
**First documented:** widespread from approximately 2020 onward
**Aliases:** "blacklist scam", "pausable token", "anti-sell pausable", "selective-block transfer", "freeze-on-buy"

## Description

A class of OAK-T1 patterns in which a token contract retains administrative authority to (a) selectively block specific addresses from transferring (blacklist), (b) globally pause all transfers (pausable), or both — typically used post-launch to trap holders who attempt to sell. From a defender's perspective, T1.004 is structurally distinct from T1.001 (Modifiable Tax Function) because the failure mode is *binary* (transfer either succeeds or reverts) rather than *quantitative* (transfer succeeds with an attacker-favourable fee). Detection signals are accordingly different: T1.004 is detectable at deployment time via static analysis of the transfer-logic gate predicates, with dynamic confirmation via simulation against varying caller addresses.

T1.004 is among the most-reused patterns in commodity scam-token deployment; per QuillAudits / Token Sniffer / GoPlus per-token risk scoring at the cohort scale, it appears as a recurring sub-pattern alongside T1.001 and T1.003 across multiple deployer-cluster footprints.

Distinction from legitimate pausable: regulated stablecoins (USDC, USDT) and compliance-bearing tokens carry analogous controls by design and are not within OAK-T1.004's hostile-classification scope. The discriminator is *who* holds the pause / blacklist authority, *what* governance / disclosure binds its use, and *whether* the authority is bundled with deployer-cluster control of LP and supply.

## Observed indicators

- A `_transfer` (or `transferFrom`) function whose first statements include `require(!_blacklist[from] && !_blacklist[to], ...)` or `require(!paused(), ...)` — the predicate gates the transfer-logic path.
- A public setter on the blacklist mapping (e.g. `setBlacklist(address,bool)`, `addToBlacklist(address[])`, `blockWallet`) gated by `onlyOwner` / `Ownable` / a custom modifier with no time-lock and no governance gate.
- Pause-state setter (e.g. `pause()`, `setPaused(bool)`, OZ `Pausable._pause`) callable by the owner with no time-lock; combined with sell-only pause (pause checked only on transfers to the AMM pool) is a particularly hostile sub-pattern.
- Storage layout: `mapping(address => bool) private _blacklist` and / or `bool private _paused` — verify via `eth_getStorageAt` against the resolved storage slot, especially for proxy-pattern contracts where source-level types may have been altered.
- Event signatures: `Blacklisted(address indexed)`, `Paused(address)`, `Unpaused(address)` — index these and alert on their emission post-launch.
- Cross-token cluster: deployer cluster previously deployed contracts with the same blacklist / pausable pattern, frequently combined with T1.001 modifiable-tax across the same operator's deployments (M04: funder-graph clustering).
- Combined with T1.003: the pause / blacklist authority survives a surface-level Ownable renouncement via an EIP-1967 admin slot or a separate `PAUSER_ROLE` AccessControl role.
- Sell-side gating sub-pattern: blacklist or pause check applied only when `to == pair` (the AMM-pool address), allowing buys to continue while sells revert — produces the canonical "buyers can enter but not exit" honeypot.

## Detection signals

- **Static, pre-trade — bytecode disassembly:** enumerate every conditional `JUMP` on the transfer-logic path; flag any branch keyed on a state variable that has a public setter not gated by a known timelock or governance contract. Slither's `pausable` and `blacklist` detectors are direct analogues for source-level analysis; bytecode-level analysis (Mythril / Etherscan-decompiled IR) extends coverage to T6.001-combined cases.
- **Dynamic, simulation matrix:** simulate `transfer` calls from a synthesised set of buyer addresses (recently-funded EOA, randomly-sampled holder) under both `paused == false` and a state-fast-forwarded `paused == true`; if any caller address can be selectively blocked while others succeed, or if a fast-forwarded pause-state simulation reverts the sell-side leg, T1.004 is confirmed. Honeypot.is implements this matrix at vendor scale.
- **Indexer query (Dune / Goldsky / Allium):** subscribe to `Blacklisted(address)` and `Paused(address)` events on the token contract; alert on first emission. For pre-listing scoring, count distinct blacklist setters in deployer's prior deployments.
- **Cross-token cluster (M04):** cluster contracts by deployer fingerprint; if a deployer's prior tokens combined T1.004 with subsequent T5.001 (Hard LP Drain) extraction events, treat new deployments from the same deployer as elevated risk regardless of source-verification claims.
- **Authority-graph (M05):** enumerate `PAUSER_ROLE` and `BLACKLISTER_ROLE` AccessControl members; combine with EIP-1967 admin slot read to defeat the T1.003-combined sub-pattern.

## Mitigations

- **OAK-M02 (Static-Analysis Pre-Deployment)** — required: flag any token whose transfer logic is gated by a settable blacklist or pausable predicate without time-lock.
- **OAK-M05 (Authority-Graph Enumeration)** — required: enumerate every role that can set blacklist / pause state, including AccessControl roles and proxy admin.
- **OAK-M01 (Source-Bytecode Verification)** — paired: ensures verified source matches deployed runtime; T1.004 + T6.001 combined cases hide the predicate in deployed bytecode.
- **OAK-M03 (Continuous Bytecode-Diff Monitoring)** — for proxy-pattern tokens, monitor implementation upgrades for newly-introduced blacklist / pause predicates.
- **OAK-M17 (Time-Locked Governance and Multi-Block Quorum)** — accept governance-gated pause authority only when timelock is non-trivial and quorum is multi-block. This is the standard discriminator that legitimately allows compliance-bearing tokens to ship pause / freeze without being classified T1.004.
- **OAK-M25 (Listing-Time Source-Verification + Audit-Status Gate)** — venue / launchpad gate.
- **OAK-M16 (Pre-Deployment Audit and Formal Verification)** — required for tokens claiming "no admin pause" — formal verification of the absence of a pause / blacklist predicate.

Operational: risk teams flag any token whose transfer logic is gated by a settable blacklist or pausable predicate without time-lock; treat such tokens as elevated risk regardless of advertised renouncement (combine with T1.003 for renounced-but-not-really detection). Token vendors / launchpads require either (a) immutable transfer logic without admin-controlled gates, or (b) explicit time-locked governance gating, as a listing precondition. Wallet-vendor pre-trade UX surfaces blacklist / pausable presence in pre-trade risk warnings.

## Discussion

T1.004 is one of the most-mature commodity scam-token patterns in OAK's coverage at v0.1; the pattern is widely recognised, the detection methodology is straightforward, and the mitigation (immutable transfer logic or time-locked governance) is well-understood. Contributors writing future T1.004 examples should focus on novel sub-patterns (per-block selective blocking, time-window-restricted transfer windows, sell-side-only pause via AMM-pool-gated predicate, EIP-1967-hidden pauser role) rather than re-documenting the canonical blacklist / pausable variants which are well-covered in commodity vendor tooling.

Calibration gap at v0.1: distinguishing hostile T1.004 from legitimate compliance-bearing pause / blacklist (USDC, USDT, regulated securities) requires authority-disclosure context that is not always machine-parseable. The pragmatic v0.1 resolution is a vetted-issuer allowlist (analogous to the timelock allowlist for T1.001 and the delegate allowlist for T1.002), with anything off-allowlist flagged. v0.x should formalise the allowlist schema.

Attribution-strength caveat: per-incident T1.004 attribution is rarely the headline framing in public post-mortems — the publicly-reportable narrative is typically the downstream T5.001 / T1.001 event. Defenders should not over-weight per-incident named-case scarcity as evidence of low prevalence; cohort-scale prevalence per vendor scanner output is the more reliable indicator.

Sub-pattern split deferred to v0.x: per-block selective blocking, time-window-restricted transfer windows, and sell-side-only pause (predicate gated on `to == pair`) may warrant their own T1.004.NN sub-Techniques. The static-rule shape differs enough across sub-patterns that detection-tooling operators will eventually want sub-Technique granularity.
