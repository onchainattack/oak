# OAK-T1.006 — Honeypot-by-Design

**Parent Tactics:** OAK-T1 (Token Genesis)
**Maturity:** emerging
**Chains:** EVM (BNB Chain dominant by deployment count; Base, Ethereum); cross-chain
**First documented:** academic foundational reference 2019 (`[torres2019]`); industrial-scale cohort observable from 2023 onward; canonical 2024-Q4 cross-chain prevalence
**Aliases:** "honeypot token", "buy-only token", "cannot-sell token", "sell-blocking smart contract", "asymmetric-fee honeypot"

## Description

Smart-contract token deployments whose immutable bytecode at deployment encodes hostile transfer-logic that permits purchases but blocks sales. The structural feature is that the trap is hard-coded into the bytecode at deployment — distinguishing T1.006 from authority-mutable variants — and detection is therefore decidable at deployment time via static analysis plus simulation matrix coverage.

Four sub-patterns recur across the cohort:

- **Sell-side-gated revert predicate** in `_transfer`: e.g., `require(to != pair || isExcluded[from], "...")` where the only addresses excluded are the deployer cluster.
- **Malicious `_beforeTokenTransfer` hook** in OZ-derived contracts that reverts on conditions designed to block all retail sells while permitting deployer-cluster operations.
- **Asymmetric extreme fee-on-transfer**: sell-tax set to 99% in the immutable constructor, achieving quantitative sell-prevention indistinguishable from a binary revert. (T1.005 frequently co-occurs as the contributing primitive.)
- **Malicious `approve` handler** that rewrites allowance state to redirect spender authority to a deployer-controlled address rather than the requested spender.

T1.006 is **structurally distinct** from adjacent token-genesis-class Techniques:

- **T1.001 (Modifiable Tax)** — fee parameter is mutable; failure mode is post-launch authority abuse; detection requires authority-graph enumeration. T1.006's fee gate (when via the asymmetric-fee sub-pattern) is set immutably at deployment.
- **T1.004 (Blacklist / Pausable Weaponization)** — gate predicate state is mutable via authority. T1.006's gate is hard-coded at deployment; no authority mutation is required for the trap to fire.
- **T1.005 (Hidden Fee-on-Transfer)** — frequently co-occurs as a contributing primitive in the asymmetric-fee sub-pattern but is not by itself sufficient: T1.005 covers the legitimate-use-overlap fee mechanism, while T1.006 covers the hostile asymmetric structure that uses the same primitive.
- **T4.001 (Permit2 Authority Misuse)** — the trap is in the user's signing surface. T1.006's trap is in the contract itself.

The legitimate-use overlap is **structurally null**: an immutably-deployed contract that permits purchases but blocks sales has no legitimate token-economic purpose. This contrasts with T1.001 / T1.002 / T1.004 (where governance-gated tax / pause / delegate authority on regulated stablecoins is a legitimate use of the same primitive). Flagged contracts under T1.006 are *always* malicious or grossly misconfigured.

## Observed indicators

- Static analysis surfaces sell-side-gated revert predicates in `_transfer` or analogous transfer entry-points; exemption mappings whose membership is restricted to deployer-cluster addresses.
- Dynamic simulation across the matrix `(amount-size × counterparty × direction × time)` produces successful buys and reverted sells against the same pool / token pair.
- Asymmetric tax structure — buy-tax vs. sell-tax differential where sell-tax is at-or-near 100%.
- On-chain transaction-pair signature: successful swaps into the pool followed by reverted sell attempts produce a characteristic explorer signature.
- Per-token-volume-versus-holder-count divergence: high cumulative buy volume, low-or-zero cumulative sell volume relative to expected token-economic distribution.
- Funder-graph fingerprint: cohort-scale deployments funded from a single or small number of operator EOAs (canonical: 2023 BNB Chain 979-EOA fan-out from a single funder).

## Detection signals

- **Pre-deployment / deployment-time:** Honeypot.is matrix-simulation methodology — synthesised matrix of `(small / medium / large) × (EOA-to-EOA / EOA-to-pool / pool-to-EOA) × (early / late) block conditions`. GoPlus token-security API exposes `is_honeypot`, `cannot_sell_all`, `slippage_modifiable` as discrete risk attributes. Token Sniffer / CertiK Skynet / RugCheck (Solana) surface T1.006-class detections under labels like "Cannot sell," "Sell tax 99%," "Honeypot-like behavior detected."
- **Static analysis:** flag any `_transfer` / `_beforeTokenTransfer` predicate that branches on counterparty identity in a way that grants asymmetric privileges to deployer-cluster addresses; flag asymmetric tax constants where sell-side fee exceeds typical legitimate ranges (>10–15%).
- **Wallet pre-trade simulation:** MetaMask / Rabby / Phantom / OKX Wallet increasingly surface honeypot warnings inline with the swap-confirmation prompt. The detection-to-enforcement chain at the wallet UX layer is the binding constraint, not detection signal availability.
- **Aggregator-side route blocking:** 1inch, Matcha, Jupiter route-blocking against detected honeypot pools.
- **Venue / launchpad listing-time gates:** centralised-venue listing review and launchpad listing-time honeypot scanning are the cohort-level prevention surfaces.

## Mitigations

- **Vendor-tooling:** matrix-simulation methodology against newly-deployed tokens; static-analysis enumeration of fee-branch predicates and exemption mappings; static-analysis flagging of asymmetric tax constants.
- **Wallet UX:** pre-trade simulation surfacing honeypot warnings inline with the swap-confirmation prompt, with high-friction confirmation flow when the simulation surface fails.
- **Aggregator-side:** route-blocking against detected honeypot pools.
- **Venue / launchpad:** listing-time honeypot-scanning gate; refusal to list tokens whose deployment-time simulation matrix surfaces sell-blocking predicates.
- **Cross-chain coverage:** the chain-distribution gradient (Base 54k / BNB 12k / Ethereum 1.2k in Q4 2024) reflects gas-cost economics. Per-chain tooling investment yields diminishing returns once the operator can cross-deploy. Defender investments should prioritise cross-chain detection coverage.

## Discussion

T1.006 is the **highest-volume malicious-token class by deployment count** at v0.1. The detection-tooling cohort is mature; the binding constraint at the cohort scale is the **discovery-surface latency** rather than the detection-signal availability. Malicious mints are detected within minutes-to-hours of deployment by commodity scanners (Honeypot.is, Token Sniffer, GoPlus); retail buyers reach them through Telegram channels, X memecoin trader accounts, and DEX-aggregator listings within seconds-to-minutes. The cohort persists not because detection is unsolved but because the *time-to-buy* is shorter than the *time-to-detect-at-the-user-decision-layer*.

The detection asymmetry across T1.001 / T1.004 / T1.006 is the structural argument for separating T1.006 from its siblings. T1.001 / T1.004 require *post-launch behavioural confirmation* (the authority-mutation event must be observed); T1.006 is decidable at deployment time with sufficient simulation matrix coverage. The vendor-tooling surface reflects this — Honeypot.is's matrix-simulation methodology is calibrated specifically for the T1.006 detection surface; Token Sniffer / GoPlus / CertiK Skynet surface T1.006-class detections under labels that cohort across T1.001 / T1.004 / T1.006 in vendor practice but separate cleanly in OAK's framing.

Templated-redeployment is the operator's leverage. The CertiK February 2024 cohort and the August–October 2023 BNB Chain 979-EOA fan-out are the canonical funder-graph (M04-class) attribution cases. Future T1.006 worked examples should record the funder-graph fingerprint where the public record permits.

The cohort scales with gas-cost economics. The Base 2024-Q4 spike (54,000 honeypot tokens) reflects per-deployment cost economics, not detection-tooling gaps. Cross-chain detection coverage is the binding defender-investment direction.

The legitimate-use overlap is structurally null. This simplifies the false-positive analysis dramatically vs. T1.001 / T1.002 / T1.004 and operationalises the listing-time gate: flagged contracts can be refused listing without manual triage in the great majority of cases.

Promotion to `stable` will follow once the per-incident worked-example anchor set expands beyond the current single cross-chain cohort file (Snibbb-as-standalone, CertiK-2024-cohort-as-standalone, BNB-979-EOA-fan-out-as-standalone). The current 2024-Q4 cohort file is sufficient to anchor T1.006 at `emerging` per the v0.1 promotion criteria but is structurally a single composite anchor rather than the multi-anchor independent-confirmation pattern that warrants `stable`.
