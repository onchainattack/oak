# OAK-T9.005 — Reentrancy

**Parent Tactics:** OAK-T9 (Smart-Contract Exploit)
**Maturity:** stable
**Chains:** EVM (primary); EVM-compatible L2s; conceptually applicable to any chain whose execution model allows external calls before state finalisation
**First documented:** 2016 (The DAO); modern hook-based and cross-protocol variants 2020+
**Aliases:** "recursive call attack", "callback reentry", "cross-function reentrancy", "cross-protocol reentrancy", "read-only reentrancy", "ERC-777 hook reentrancy", "ERC-721/1155 receive-hook reentrancy"

## Description

A class of smart-contract exploits in which the target contract performs an external call to an attacker-controlled address before its own internal state has been brought into a consistent post-action state, and the attacker's contract uses control of the execution flow during that external call to re-enter the target — calling back into the same function, a sibling function on the same contract, or a related function on a peer protocol — while the target's storage still reflects the pre-action snapshot. The attacker thereby exercises a state transition (typically a withdrawal, a debt-accounting update, or a balance read) against stale state, repeatedly, and extracts more value than the protocol intends to release. From a defender's perspective, T9.005 is identifiable through the call-graph shape: a single external transaction whose internal call trace contains repeated re-entries into the same function selector (or into a tightly-coupled set of selectors on the same or related contracts) before the originating call returns. `[zhou2023sok]` and `[owaspscstop10]` both place reentrancy among the canonical smart-contract vulnerability classes; OWASP SC Top 10 lists Reentrancy explicitly as a top-tier category through 2025/2026.

OAK groups several sub-patterns under T9.005:

- **Single-function (canonical) reentrancy** — the function makes an external call before updating its own storage; the attacker re-enters the same function. This is the historical DAO pattern and is largely closed by the now-standard checks-effects-interactions discipline and OpenZeppelin's `ReentrancyGuard`.
- **Cross-function reentrancy** — function A makes the external call; the attacker re-enters function B, which reads or mutates state that A had not yet finalised. A per-function `nonReentrant` modifier does not block this if B is not gated by the same lock.
- **Cross-protocol (cross-contract) reentrancy** — the attacker re-enters a *different* protocol that reads stale state from the target (e.g., reads a balance, share-price, or LP-token accounting that the target has not yet updated). This is the dominant modern variant and is frequently chained with T9.002 (Flash-Loan-Enabled Exploit) to acquire the capital required to make the inconsistent intermediate state economically meaningful.
- **Hook-based reentrancy** — the external call is implicit in a token-standard hook: ERC-777 `tokensReceived`, ERC-721 `onERC721Received`, ERC-1155 `onERC1155Received` / `onERC1155BatchReceived`. The target may not realise it has surrendered execution flow, because the call site looks like a token transfer rather than an explicit `call`.
- **Read-only reentrancy** — the attacker re-enters a `view` function on the target during a state-mid-update window; a third protocol that reads that view-function output makes a decision against stale data. No state on the target is mutated by the re-entry itself; the loss is realised on the third protocol.

## Observed indicators

- Single-transaction extraction whose internal call trace shows repeated invocations of the same function selector on the same contract, or a fan-out of calls into related selectors on the same contract before the outer call returns.
- Token-transfer call sites (ERC-777, ERC-721, ERC-1155) that hand control to a user-supplied receiver address before the sender contract has finalised its accounting.
- Third-protocol settlement, liquidation, or pricing decision taken against a view-function reading from a contract that is mid-execution in the same transaction.
- Post-event: a clean reproducible test case that demonstrates the reentry path (audit-firm replay against a forked block).

## Detection signals

- Pre-deployment: static analysis for external calls preceding storage writes (Slither's `reentrancy-eth`, `reentrancy-no-eth`, `reentrancy-benign` detectors; MythX, Securify); flagging of any `call` / `transfer` / token-hook-triggering operation that is not preceded by the relevant state update.
- Pre-deployment: explicit per-function review of cross-function and cross-contract state dependencies; absence of `ReentrancyGuard` on functions that share a state surface.
- Pre-deployment: formal verification of "no external call observes pre-finalised state" as a stated invariant under all reachable call paths.
- Per-transaction monitoring at deployed protocols: alert on transactions whose internal call trace shows the target contract being re-entered (same `to` address appearing multiple times within a single outer call) before the outer call returns.
- Cross-protocol monitoring: alert on view-function reads from a contract whose storage is mid-update in the same transaction (read-only reentrancy signature).
- **Read-only reentrancy runtime signal:** cross-contract `view` / `staticcall` reads against a contract that is currently mid-execution (the call frame for that contract has not yet returned), where the read returns data inconsistent with the post-write state the contract will commit on frame exit. Canonical worked example: the Curve July 2023 incident family and the Market.xyz consumer that priced LP-token reads off Curve pools whose storage was mid-update during the malformed-`@nonreentrant`-lock window — the `view` function returned a transient inconsistent value to a third protocol that took an economic action against it. ## Real-world examples

- **The DAO — Ethereum — 2016-06-17** — Loss \~\$60M (~3.6M ETH at then-prevailing prices). The DAO's `splitDAO` / withdrawal path performed an external call to the recipient before zeroing the recipient's internal balance; the recipient contract re-entered the same path, draining the pool against repeatedly-stale state. The incident is the historical origin of the reentrancy class and prompted the contentious Ethereum hard fork that produced the Ethereum / Ethereum Classic split. Cited as the canonical reference incident for T9.005.
- **Lendf.Me — Ethereum — 2020-04-19** — Loss \~\$25M. The protocol accepted imBTC, an ERC-777-style token whose `tokensReceived` hook handed execution to the depositor before Lendf.Me finalised its internal accounting; the attacker used the hook to re-enter the deposit / borrow path against stale balance state. The case is the canonical hook-based reentrancy reference and is widely cited as the first large-loss ERC-777-hook incident.
- **Cream Finance (Iron Bank) — Ethereum — 2021-10-27** — Loss \~\$130M+. Cross-protocol reentrancy combined with a flash loan (T9.002): the attacker used flash-loaned capital to interact with Cream's collateral / borrow accounting through an external integration whose callback re-entered Cream while its internal state was mid-update. The case is the canonical modern reference for cross-protocol reentrancy chained with flash loans, and is the principal reason simple per-function `ReentrancyGuard` is no longer considered sufficient on its own.
- **Cohort context** — `[owaspscstop10]` (Reentrancy category) and `[zhou2023sok]` taxonomy both document that reentrancy continues to surface as an incident class through 2022–2024, predominantly in cross-function, cross-protocol, and hook-based forms rather than the canonical 2016-style single-function pattern.

## Mitigations

- **Audit firms:** require checks-effects-interactions discipline as a structural review item; flag any external call preceding state finalisation; flag absence of `ReentrancyGuard` on functions sharing a state surface; treat ERC-777 / ERC-721 / ERC-1155 hook surfaces as implicit external calls and audit them as such.
- **Protocols:** adopt OpenZeppelin `ReentrancyGuard` (or equivalent) as a baseline; apply the lock across the *set* of functions that share state, not per-function; finalise state writes before external calls (checks-effects-interactions); avoid integrating tokens with reentrancy-relevant hooks (ERC-777, certain ERC-721/1155) in lending-style accounting paths without explicit hook-aware design; treat view functions read by third protocols as part of the reentrancy surface (read-only reentrancy mitigation).
- **Risk teams:** maintain per-protocol inventories of (a) which contracts use `ReentrancyGuard` and at what granularity, (b) which protocols accept ERC-777-style hooked tokens in collateral or accounting paths, (c) which view functions are consumed by third protocols. Treat protocols whose accounting paths integrate hooked tokens without explicit hook-aware design as elevated risk; treat any cross-protocol integration whose price / balance reads happen mid-transaction as an elevated read-only-reentrancy surface.

## Discussion

T9.005 is marked Maturity = stable rather than historical because the underlying failure mode — surrendering execution flow before finalising state — recurs in modern variants that the canonical mitigation does not fully address. OpenZeppelin's `ReentrancyGuard` (widely adopted post-2019) substantially closes the 2016-style single-function pattern, and pre-deployment static analysis routinely catches it; consequently, post-2021 reentrancy incidents are dominated by (i) cross-function reentrancy where the per-function lock did not span the shared-state surface, (ii) cross-protocol reentrancy where the attacker re-enters a *peer* protocol that reads stale state, frequently chained with T9.002 to acquire flash-loaned capital, (iii) hook-based reentrancy where the external call is implicit in a token-standard receive hook the protocol did not treat as an external call, and (iv) read-only reentrancy where no state mutation occurs in the re-entry but a third protocol's view-function read is the loss surface.

The Cream Finance October 2021 case is the v0.1 canonical example of the modern variant because it combines the cross-protocol pattern with flash-loan-enabled capital, making the T9.002 cross-reference unambiguous; The DAO is retained as the historical reference for the class origin and to ground the Ethereum-Classic fork context that newer contributors may not have lived through. Sub-Techniques may be split out in v0.x updates if the cross-protocol and read-only sub-patterns reach independent characterisation maturity (notably, read-only reentrancy is increasingly cited as a distinct sub-class in 2023–2025 audit-firm literature and may warrant its own technique ID).
