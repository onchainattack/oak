# OAK-T4.012 — Clipboard-Hijacker (Clipper) Address Substitution

**Parent Tactics:** OAK-T4 (Access Acquisition), OAK-T6 (Defense Evasion)
**Maturity:** emerging
**Chains:** cross-chain (clippers are coin-agnostic — Bitcoin, Ethereum, Litecoin, Monero, Dogecoin, Tron, XRP, Solana, Cardano, Zcash, and others are matched by per-format address regex)
**First documented:** 2017 (CryptoShuffler clipboard trojan); commodity scale 2018 onward (ESET clipboard-hijacker family); resurgence 2023 (trojanised Tor Browser clipper, ~$400K) and 2026 (Microsoft `CryptoBandits` USB-worm clipper; Check Point Rust-clipper reputation-laundering campaign)
**Aliases:** "clipper", "crypto clipper", "clipboard hijacker", "ClipBanker", "address-swap malware", "clipboard wallet-address replacement"

## Description

Host-resident malware that monitors the operating-system clipboard and, when it detects a string matching a cryptocurrency address format, silently overwrites the copied value with an **attacker-controlled address of the same coin** before the victim pastes it into a send field. The victim authorises a transfer to what they believe is the address they copied; the funds go to the attacker. The same clipboard-monitoring primitive is routinely extended into a **secret-harvesting arm** — the malware also watches the clipboard (and captures screenshots) for **BIP39 seed phrases (12–24 words)** and **private keys (raw hex / Bitcoin WIF)** and exfiltrates them, converting a recipient-substitution tool into a full wallet-compromise tool.

T4.012 is **structurally distinct from T4.003 (Address Poisoning)**, even though both end with the victim sending to an attacker address they believed was correct. T4.003 plants look-alike transactions in the victim's *on-chain wallet history* and relies on the victim copy-pasting from that history — there is no malware on the host, and the manipulated substrate is the wallet UI's history display. T4.012's compromise is a **host-side endpoint infection that tampers with the clipboard in real time**; it works regardless of where the victim copied the address from (a website, a message, their own address book). T4.012 is also distinct from the Bybit-class **sign-time UI swap**, where the destination is altered inside compromised signing infrastructure (a software supply-chain compromise of the operator's signing stack) rather than on a commodity-infected end-user endpoint.

Like T4.003, the load-bearing weakness is **truncated / unverified destination display**: clippers frequently select a substitute address that shares leading and/or trailing characters with a plausible target so the victim's habitual "first-and-last-six-characters" check passes. Modern clippers ship **large embedded attacker-address lists** keyed by coin and address format (the 2026 Check Point Rust clipper carried over 15,500 addresses) so that a same-format replacement is always available. T4.012 requires **no on-chain authority and no protocol interaction** — only the victim's own copy → paste → send action on a compromised device — which is why it produces no on-chain authorisation event at compromise time and is invisible to allowance/permit/signature detection.

The delivery vector is malware-class-typical and varies independently of the clipper logic: **removable-media / USB `.lnk` worms** (Microsoft `CryptoBandits`), **reputation-laundered fake-software distribution** (fake reviews, AI-narrated tutorial videos, manipulated VirusTotal votes, decoy GitHub/SourceForge repos — Check Point's 2026 Rust-clipper campaign), **trojanised installers** (2023 Tor Browser clipper), cracked-software bundles, and drainer/loader payloads. The crypto-relevant primitive OAK classifies here is the clipboard substitution/harvesting itself; the surrounding delivery, persistence, anonymising C2 (commonly Tor SOCKS to `.onion` hidden services), and remote-code-execution facets are general-malware capabilities that ride alongside it.

## Observed indicators

- A resident process polling the clipboard at high frequency (commodity clippers poll every ~100–500 ms) and writing the clipboard immediately after a crypto-address-shaped string appears.
- Pasted destination address differs from the copied address while sharing leading/trailing characters (same coin, same format) — the canonical clipper signature.
- Per-format address regex sets in the binary (Bitcoin bech32 / legacy / P2SH, Ethereum `0x`-hex, Litecoin, Monero, Dogecoin, Tron, XRP, Solana, Cardano, Zcash) paired with embedded attacker-address tables.
- Clipboard/screen monitoring for BIP39 mnemonic word-lists and WIF/hex private-key patterns; periodic screenshot capture for context.
- Anonymised egress: a bundled or co-installed Tor client and outbound SOCKS connections to `localhost:9050` / Tor relay ports, with C2 over HTTP POST to `.onion` hidden services.
- Persistence + propagation artefacts: scheduled tasks or Startup/LaunchAgent entries running script payloads; creation of malicious `.lnk` shortcuts on inserted removable drives (worm sub-shape); macOS Gatekeeper-bypass helper scripts.

## Detection signals

- **Wallet-side / pre-send (highest leverage, OAK-M29 / OAK-M18):** the wallet (or signing device) renders the **full** destination address and re-derives it from a trusted source at sign time rather than trusting the pasted string; checksum/full-address confirmation defeats the same-prefix substitution. Address-book entries preferred over paste.
- **Endpoint:** detect a process that reads and writes the clipboard within a tight time window whenever clipboard content matches an address regex; flag clipboard writes not originating from a user gesture.
- **Network:** alert on unexpected local Tor SOCKS proxy activity (`localhost:9050`) and outbound Tor connections from non-browser processes; hunt script-interpreter-to-network chains.
- **Removable-media (worm sub-shape):** alert on `.lnk` files whose target invokes a script interpreter, and on shortcut creation against files on newly-mounted drives; disable AutoRun/AutoPlay.
- **Distribution-side (reputation-laundering sub-shape):** treat coordinated five-star reviews, AI-narrated "how to install" videos, inflated download counters, and benign VirusTotal votes/comments on freshly-published "trading-bot" / "predictor" tools as a distribution-fraud signal rather than reputation.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T4.012's actionable detection lives at the endpoint (clipboard-monitoring behaviour, EDR) and at the wallet-UX layer (full-address verification at sign time), not at the on-chain runtime risk-engine surface MG operates on.
- Vendor coverage: endpoint AV/EDR clipper detections (Microsoft Defender `Trojan:Win32/CryptoBandits.*` / `ClipBanker.*`; ESET, Kaspersky clipboard-hijacker families); wallet-side full-address verification (Trezor Suite, Rabby, MetaMask address checks); address-book / out-of-band confirmation flows.

## Mitigations

- **OAK-M29 (Full-Address Verification and Lookalike Detection):** highest-leverage wallet-side control — render and verify the **complete** destination address (with checksum) at sign time; never trust a same-prefix paste; prefer address-book entries over pasted values.
- **OAK-M18 (Out-of-Band Destination Verification):** confirm the full destination through a second channel before sending, especially for high-value transfers; the clipper cannot tamper with an address read aloud or scanned from a trusted out-of-band source.
- **OAK-M19 (Air-Gap Cold-Wallet Signing):** air-gapped signers that display the full destination with checksum defeat clipboard substitution for treasury-scale transfers.
- **OAK-M22 (Rotate-on-Disclosure Discipline):** any seed phrase or private key that touched the clipboard of an infected host must be treated as compromised — migrate funds to freshly-generated keys on a clean device.
- **OAK-M30 (Per-dApp Domain and App-Store-Package Allowlist) / OAK-M40 (Supply-Chain Package Integrity):** counter the fake-software distribution sub-shape — install tooling only from verified first-party sources; treat manipulated reviews/votes and decoy repos as adversarial.
- **OAK-M21 (Anti-Phishing Training for Privileged Staff):** removable-media hygiene (no unknown USB devices; disable AutoRun; block `.lnk` execution from removable drives) addresses the worm-propagated sub-shape.

**Canonical mitigations:** OAK-M29, OAK-M18, OAK-M19, OAK-M22.

## Discussion

T4.012 fills a long-standing gap in OAK: clipboard hijacking is one of the oldest and most prevalent crypto-theft malware classes in the wild, yet the matrix previously had no home for it — T4.003 (Address Poisoning) explicitly scopes itself to *on-chain* history manipulation with no host malware, and its own discussion carves out host-side and sign-time address swaps as out-of-class. The defining feature of T4.012 is that the substitution happens **on the victim's endpoint, in the clipboard, at copy time** — upstream of any wallet, any chain, and any signing surface — so it is invisible to every on-chain detection layer and is defeated only at the endpoint (EDR / clipboard-behaviour detection) and at the wallet-UX layer (full-address verification at sign time). This makes the highest-leverage takeaway identical in spirit to T4.003's — *verify the full destination address, never the truncated form* — even though the mechanism and the defender's instrumentation point are different.

The class is marked `emerging` rather than `stable` because per-campaign operator attribution is typically `pseudonymous` (commodity clipper builders sold on forums; reputation-laundering personas such as `@JoseCmanXD`) and the wallet-side full-address-verification surface is not uniformly deployed. The two 2026 anchors capture two independent **distribution** sub-shapes that are tracked as forward candidates pending a second anchor each: **T4.012.001** (removable-media / worm-propagated clipper — Microsoft `CryptoBandits`) and **T4.012.002** (reputation-laundered fake-software distribution — Check Point Rust clipper). The general-malware facets that ride alongside the clipper — USB-worm propagation, Tor-anonymised C2, scheduled-task persistence, and `EVAL`-style remote code execution — are deliberately *not* minted as crypto-attack techniques here; OAK classifies the crypto-relevant primitive (clipboard substitution and secret harvesting) and treats the surrounding malware capabilities as delivery/operational context, consistent with how the framework scopes endpoint-malware incidents elsewhere.
