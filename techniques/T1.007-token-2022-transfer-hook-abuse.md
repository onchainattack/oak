# OAK-T1.007 — Token-2022 Transfer-Hook Abuse

**Parent Tactics:** OAK-T1 (Token Genesis); modifier on OAK-T6 (Defense Evasion) where the hook program is independently upgradable
**Maturity:** emerging
**Chains:** Solana (SPL Token-2022); cross-standard analogues on EVM (ERC-777 `tokensReceived`, ERC-1363 transfer-and-call, ERC-4626 vault hooks) covered separately at OAK-T9.005
**First documented:** Halborn pre-production audit of Token-2022 (November 2022); class-level developer-side documentation 2023–2025; April 2025 ZK-ElGamal proof zero-day disclosure-and-patch cycle. Per-incident externally-attributed-exploit anchor remains empty at v0.1 freeze.
**Aliases:** "transfer-hook callback abuse", "Token-2022 hook reentrancy", "SPL transfer-hook attack"

## Description

SPL Token-2022 mints with the `TransferHook` extension configured to invoke an attacker-controlled program on every `transfer_checked` invocation. Reintroduces the **callback-based vulnerability class** — reentrancy, mid-transfer state mutation, attacker-controlled CPI — into Solana's runtime. The class is structurally novel for Solana because the runtime had previously precluded callback-based attack surfaces by design; the hook extension intentionally adds a callback surface for legitimate business reasons (royalty enforcement, transfer-restriction policies, on-transfer accounting).

Three properties make the surface structurally significant:

1. **The hook program is specified by the token creator, not the caller.** When `transfer_checked` is invoked on a mint configured with a `transfer_hook_program_id`, the Token-2022 program makes an additional CPI call to the hook program. The caller has no say in which program is invoked; the mint config is binding.
2. **Solana's runtime invariant — "a program can only modify accounts it owns" — is preserved at the protection-domain layer, but the callback-based attack surface is reintroduced.** The hook program can read transfer details, can perform arbitrary CPI calls of its own, and can return execution to the original Token-2022 program in a state mutated by the hook's actions.
3. **The hook program is itself upgradable via `BPFLoaderUpgradeable`.** A benign hook at deployment can be upgraded to a malicious hook post-deployment by the program's upgrade authority. This creates a residual T6.005-style mutable-implementation sub-surface within the T1.007 class.

T1.007 is **structurally distinct** from adjacent Token-2022-extension-based Techniques:

- **T1.002 (Permanent Delegate)** — the extension grants standing burn / transfer authority over holder balances. Detection is binary at the mint-config layer; UX-layer enforcement is the binding constraint. T1.007's failure mode is hook-program-controlled execution rather than standing authority.
- **T1.005 (Hidden Fee-on-Transfer / TransferFee extension)** — the hook's behaviour is not bounded by the fee-arithmetic primitive that constrains T1.005. Detection-tool maturity for the transfer-hook class lags T1.002 / T1.005 because reasoning about the hook's behaviour requires program-level static analysis or simulation rather than mint-config binary checks.
- **T9.005 (Reentrancy)** — the cross-standard analogue. ERC-777 reentrancy (imBTC 2020, Lendf.Me 2020 ~\$25M) is the closest cross-standard reference; T1.007 is the Solana-runtime-import variant of the same threat class.

## Observed indicators

- **Mint-config:** the `TransferHook` extension is configured on the mint with a non-zero `transfer_hook_program_id`. This is binary-detectable via standard Token-2022 extension TLV parsing.
- **Hook-program behaviour:** the hook performs CPI calls of its own; mutates state observable by the calling program; conditionally reverts based on caller / counterparty identity. Detecting *malicious* behaviour from the hook program requires program-level static analysis or simulation.
- **Hook-program upgrade authority:** the hook program is deployed via `BPFLoaderUpgradeable` with a non-zero / non-burned upgrade authority, indicating runtime-mutability of the hook implementation.
- **Cross-program-flow signature:** transactions invoking `transfer_checked` against a hook-configured mint exhibit elevated CPI depth; legitimate hooks typically perform bounded, well-defined logic, while malicious hooks may perform arbitrary CPI fan-out.

## Detection signals

- **Pre-deployment / mint-config layer:** binary detection of *whether* a transfer hook is configured (`is_transfer_hook_configured`); enumeration of the hook program ID; authority-graph enumeration of the hook program's upgrade authority. Mature at v0.1.
- **Hook program static analysis:** Anchor IDL inspection where the hook program publishes one; raw Solana BPF static analysis otherwise. Detection-tool maturity for the hook *behaviour* analysis lags the mint-config binary checks; vendor coverage at v0.1 is partial.
- **Simulation:** hook-program execution simulation against synthesised `(buyer × seller × amount × pool)` matrices, paralleling the T1.006 simulation-matrix methodology adapted for the Solana runtime.
- **Runtime monitoring:** for protocols that integrate Token-2022 mints, alert on `transfer_checked` invocations that exhibit unexpected mid-transfer state mutations or CPI fan-out beyond expected bounds.
- **Cross-standard Ethereum analogues:** the detection-signal structure parallels T9.005's hook-based reentrancy detectors (Slither's hook-aware detectors; runtime call-trace monitoring for re-entry into the calling program during a Token-2022 transfer-hook CPI).

## Mitigations

- **Pre-deployment (token issuer side):** issuer-side adoption of conservative hook patterns (no CPI fan-out; bounded state mutations; no caller / counterparty branching that could constitute asymmetric privilege); hook-program upgrade-authority burn or multisig timelock to bound the residual T6.005 surface.
- **Pre-integration (protocol side):** for protocols accepting Token-2022 mints in collateral / accounting / vault paths, enumerate the hook program's upgrade authority and behaviour at integration time; refuse integration of mints whose hook program is not explicitly trusted (analogous to the ERC-777 / hooked-token integration discipline at T9.005).
- **Runtime (program side):** apply checks-effects-interactions discipline ported from EVM practice — finalise state writes before CPI invocations into Token-2022 transfers; do not re-enter the calling program from the hook in a way that observes pre-finalised state.
- **Ecosystem-side:** Solana program-static-analysis tooling investment to bring hook-behaviour analysis to parity with mint-config binary checks; vendor coverage of the hook-program upgrade-authority surface as a first-class authority enumeration target.

## Discussion

T1.007 is the canonical 2024–2025 candidate for the **`Maturity: emerging`** classification at v0.1: the threat-class characterisation is mature across multiple defender-side and developer-side sources, but the field-confirmed externally-attributed-exploit anchor is empty. The class is structurally the highest-leverage Token-2022 extension class for cohort-scale exploitation: PermanentDelegate (T1.002) requires the operator to manage the delegate authority off-chain; TransferFee (T1.005-Solana-side analogue) is bounded by the fee-arithmetic primitive; TransferHook permits arbitrary attacker-controlled code execution on every transfer.

The detection-surface generalisation is structurally important: T1.007 detection is **closer to EVM honeypot detection (T1.006)** than to mint-config binary checks (T1.002 / T1.005). The hook program ID is observable at the mint-config layer, but reasoning about the hook program's *behaviour* requires program-level static analysis or simulation. The defender-tooling cohort (Anchor static checks, Sec3, OtterSec audit practice) is calibrating to the new threat surface in 2024–2026.

The hook-program-upgrade-authority sub-surface is a residual T6.005 risk within the T1.007 class. A benign hook at mint-deployment can be upgraded to a malicious hook post-deployment via standard `BPFLoaderUpgradeable` mechanics. T1.007 detection must enumerate the hook program's upgrade authority as a first-class authority surface (M05-class enumeration applied at the Solana program layer, not the EVM contract layer).

The class generalises beyond Solana. Any token standard that admits attacker-controlled callbacks on transfer is exposed to T1.007-equivalent exploitation. ERC-777 (Ethereum, with documented reentrancy exploits including imBTC 2020 and Lendf.Me 2020); ERC-1363 (transfer-and-call); ERC-4626 (vault hooks). The Solana Token-2022 transfer-hook is the latest entry in this class; the Lendf.Me 2020 \$25M ERC-777 reentrancy case is the closest cross-standard reference for cohort-scale extraction shape. Contributors writing future T1.007 worked examples should treat the cross-standard analogues as adjacent reference and ensure the framing generalises.

The Halborn pre-production audit pattern + Solana Foundation ZK-ElGamal April 2025 disclosure-and-patch cycle establish a positive defender-tooling baseline that T1.007 contributors should track. The disclosure-and-patch cycle is a positive defender-tooling capability and a useful operational signal for the broader Token-2022 surface.

Promotion to `observed` will follow the first per-incident exploit with public on-chain forensic disclosure. Until then, T1.007 remains at `emerging` with the class-level anchor file documenting the threat-class characterisation maturity. Contributors with such disclosures should submit them via PR.
