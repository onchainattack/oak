# OAK-T15.002 — Supply-Chain / Vendor-Pipeline Compromise

**Parent Tactics:** OAK-T15 (Off-chain Entry-Vector / Pre-Positioning)
**Maturity:** emerging
**Chains:** chain-agnostic (the supply-chain compromise is off-chain; downstream on-chain manifestation is per-incident)
**First documented:** systematically 2023 onward (Atomic Wallet, Ledger Connect Kit), with earlier antecedents in npm / package-registry compromise outside crypto
**Aliases:** "supply-chain attack", "build-pipeline injection", "npm package compromise", "CI/CD compromise", "post-install backdoor", "vendor-pipeline compromise"

## Description

Compromise of the **build, distribution, packaging, signing, or authentication infrastructure** of an operator OR of an upstream vendor whose code, packages, signed artefacts, or auth flows feed into the operator. The defining property is that the compromise lives at a **shared upstream surface** — once the malicious artefact is published, every downstream consumer (every dApp embedding the package, every customer of the wallet vendor, every operator running the toolchain) is exposed for as long as the malicious version is live.

T15.002 covers four structurally adjacent sub-shapes that share the same shared-upstream-surface property:

- **Wallet-vendor build-pipeline compromise.** The vendor's CI / release pipeline emits a binary or update package containing attacker-injected code. Canonical anchor: Atomic Wallet (2023-06; \~\$100M); the malicious update propagated to the vendor's user base via the vendor's own update channel.
- **Package-registry compromise (npm / PyPI / GitHub-org).** A package that downstream operators or dApps depend on is republished with a malicious version, OR a typosquat is positioned to be installed via tooling-error. Canonical anchors: Ledger Connect Kit npm compromise (2023-12; three malicious versions 1.1.5/1.1.6/1.1.7 published during the compromise window); Solana web3.js npm compromise (2024-12); 3CX 2023; Bittensor PyPI 2024; Polymarket trader-tooling supply chain (2026-01; npm + GitHub-org).
- **Third-party signing-platform / vendor build-pipeline compromise.** A vendor's signing-flow UI or build pipeline is compromised such that subsequent legitimate-customer signing operations carry attacker-substituted payloads. Canonical anchor: Bybit / Safe{Wallet} (2025-02; the proximate compromise was at the Safe{Wallet} build pipeline that produced the signing UI Bybit's signers loaded).
- **Vendor-internal-communications / shared-tooling compromise.** Vendor-internal Slack / Telegram / ticketing surfaces compromised such that the vendor's customer-facing requests to the operator are manipulable. Canonical adjacent: DMM Bitcoin / Ginco wallet-software vendor (2024-05); the compromise was at the vendor's internal communications surface, not at the binary-distribution surface, but the supply-chain class is shared.

T15.002 is **distinct from T15.003** even though both can chain together. T15.002 is the **shared-upstream-surface** compromise — the malicious artefact published from the vendor's pipeline propagates to every consumer. T15.003 is the **endpoint-state** compromise — the developer's workstation is in a state where its outbound transactions / commits / deployment artefacts can be manipulated. A T15.002 publication often originates from a T15.003 endpoint state at the vendor (the developer whose machine was compromised pushed the malicious commit), but the downstream impact differs: T15.002 multiplies across every consumer of the shared upstream surface; T15.003 is bounded to what that endpoint can do.

T15.002 is **distinct from T11.002 (Wallet-Software Distribution Compromise)**. T11.002 covers the on-chain-extraction event from the user-base of a compromised wallet vendor — the loss observable on-chain from victims who installed the malicious wallet binary. T15.002 covers the upstream supply-chain compromise that made T11.002's distribution malicious in the first place. They are paired but separated because the mitigation surface differs (T11.002: customer-side wallet-vendor allowlisting and update-rollback; T15.002: vendor-side build-pipeline integrity, package-registry hardening, signed-artefact verification).

## Observed indicators

- **Off-chain (vendor-side):** vendor disclosure of CI / release-pipeline compromise; package-registry version-history showing republication of a previously-released version OR rapid sequence of new versions inconsistent with the project's normal cadence; signed-artefact verification mismatch against expected reference; build-artefact hash drift between vendor reference and what's distributed.
- **Off-chain (registry-side):** package-registry advisory / takedown event; npm `Advisory` or `Vulnerability` listing; GitHub-org compromise notice; PyPI / Cargo / RubyGems package-yank event with associated security advisory.
- **On-chain (downstream):** drainer-class extraction events from the user base of a wallet vendor or dApp embedding the compromised package, in a tight time window after the malicious version was published; cluster of victims with no obvious shared phishing lure beyond "they all updated the same wallet / loaded the same dApp during window X."
- **Forensic record:** the compromised package version on the registry; the diff between the malicious version and the prior legitimate version; the build-pipeline log if the vendor publishes one; the IDP / SSO log showing the developer-credential / publisher-credential compromise that authorised the malicious publication.

## Detection signals

- **Vendor-side build-pipeline integrity** — runtime integrity verification of build artefacts against a signed reference; mandatory two-person review for production publication; hardware-key-protected publisher credentials; EDR / MDM on machines that hold publisher credentials.
- **Customer-side dependency hash-pinning** — pin to a specific package version's hash, not just version range; flag any version-bump that introduces a new install-side script.
- **Registry-advisory monitoring** — subscribe to npm / PyPI / Cargo security advisories; cross-reference dependency manifests against advisories on a continuous schedule.
- **Update-channel anomaly detection** — vendor-side: detect new releases that did not pass the standard CI / review path; customer-side: detect updates that change the binary's signing surface.
- **Cohort-level on-chain detection** — when a wallet vendor publishes an unscheduled update OR a widely-embedded npm package is republished, treat the vendor's user base as elevated-risk and watchlist their wallets for outflow patterns inconsistent with their baseline.

## Real-world examples

- **Bybit / Safe{Wallet} — Ethereum — 2025-02-21** — Loss \~\$1.46B. The compromise was at the Safe{Wallet} build pipeline that produced the signing UI deployed for Bybit's signers; the compromised UI substituted the destination address at sign time. Canonical T15.002 → T11.001 chain at the largest dollar scale on the public record. See [`examples/2025-02-bybit.md`](../examples/2025-02-bybit.md).
- **Atomic Wallet — multi-chain — 2023-06-03** — Loss \~\$100M. The Atomic Wallet vendor's build-and-distribution pipeline was compromised; the malicious update propagated via the vendor's normal update channel and produced cohort-scale extraction across the user base. Canonical T15.002 → T11.002 chain anchor for the wallet-vendor sub-shape. See [`examples/2023-06-atomic-wallet.md`](../examples/2023-06-atomic-wallet.md).
- **Ledger Connect Kit — Ethereum (multi-dApp) — 2023-12-14** — npm compromise of `@ledgerhq/connect-kit`; three malicious versions (1.1.5, 1.1.6, 1.1.7) were published during the compromise window. Every dApp embedding the library presented a wallet-drainer overlay to end-users connecting their wallets. Canonical T15.002 → T11.002 / T1.003 chain for the npm-package sub-shape. See [`examples/2023-12-ledger-connect-kit.md`](../examples/2023-12-ledger-connect-kit.md).
- **Solana web3.js npm — Solana (multi-dApp) — 2024-12** — Compromise of the Solana web3.js npm package; cohort-level downstream impact across dApps that pulled the malicious version during the compromise window. See [`examples/2024-12-solana-web3js-npm-supply-chain.md`](../examples/2024-12-solana-web3js-npm-supply-chain.md).
- **Polymarket trader-tooling — Polygon — 2026-01** — Loss attributable to the trader-tooling supply-chain compromise; the malicious npm package + GitHub-org publication targeted bot-builder / quant developer environments specifically. The case is also OAK-T11.009's canonical anchor. See [`examples/2026-01-polymarket-trader-tooling-supply-chain.md`](../examples/2026-01-polymarket-trader-tooling-supply-chain.md).

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T15.002 detection lives at the off-chain build-pipeline / package-registry / vendor-disclosure layer, not at the on-chain runtime risk-engine layer. Cohort-level on-chain indicators (cluster of vendor-user-base outflows immediately after a vendor publication) are detectable at the on-chain layer but only after extraction has begun.
- Vendor coverage: GitHub Advanced Security / dependabot; Snyk; Socket.dev; npm `audit`; PyPI / Cargo registry advisories; SLSA build-provenance frameworks; Sigstore signing infrastructure.

## Mitigations

- **Vendor-side build-pipeline integrity (highest-leverage):** runtime integrity verification of build artefacts against a signed reference; mandatory two-person review for production publication; hardware-key-protected publisher credentials; EDR / MDM on publisher-credentialed machines; reproducible builds where feasible; SLSA-class build-provenance attestation.
- **Customer-side hash pinning + update-channel verification:** pin dependency versions by hash, not version range; verify wallet-binary signatures on update; verify npm package install-side scripts before merge.
- **OAK-M20 (Vendor breach-notification SLA):** require contractual breach-notification SLA from upstream vendors so customers can trigger their incident-response within hours rather than days.
- **OAK-M22 (Rotate-on-disclosure discipline):** when an upstream vendor discloses a build-pipeline compromise, treat any artefact pulled during the compromise window as compromised and rotate the affected surface.
- **Cohort-level CTI integration:** subscribe to vendor-disclosure feeds; cross-reference disclosed compromise windows with internal dependency-update history; treat customer organisations of an affected vendor as elevated risk for a calibrated post-disclosure window.

## Discussion

T15.002's structural lesson is that the build / distribution / signing pipeline is a **shared-upstream-surface attack class** — the dollar-loss-prevented metric is sensitive not to per-victim mitigation but to vendor-side build-pipeline integrity, because every successful T15.002 multiplies across the vendor's customer base. The Atomic Wallet and Ledger Connect Kit cases demonstrate this asymmetry: a single compromise produced cohort-scale downstream loss with no per-victim mitigation surface available between publication and on-chain extraction.

T15.002 is also the **most actively-evolving sub-Technique** in T15. The npm / PyPI / GitHub-org sub-shape (Ledger Connect Kit, Solana web3.js, Bittensor PyPI, 3CX, Polymarket) has produced multiple high-impact incidents per year since 2023 and shows no sign of slowing; the wallet-vendor build-pipeline sub-shape is a smaller cohort but each individual incident is dollar-significant; the third-party signing-platform sub-shape (Safe{Wallet} at Bybit) demonstrated the largest single-incident loss in T15 history. Contributors writing future T15.002 examples should preserve the sub-shape distinction (wallet-vendor / package-registry / signing-platform / vendor-internal-comms) because the mitigation surface differs across them.

The boundary against T15.001 is causal-direction: T15.001 is the social-engineering vector that produced the developer-credential or publisher-credential compromise; T15.002 is what the attacker did with that credential once obtained. A complete chain typically reads T15.001 → T15.003 (developer endpoint) → T15.002 (publication) → T11.001 / T11.002 (downstream extraction). Worked examples may map to multiple stages of this chain.

False-positive considerations: legitimate package republications (security patches, urgent bugfixes, version-bump cycles) are common; the detection signal is not "a republication occurred" but "the republication path bypassed normal review" or "the new version's install-side surface differs structurally from prior versions." Customer-side detection benefits significantly from registry-advisory subscriptions because most T15.002 incidents are publicly disclosed by the vendor or registry within hours-to-days.

v0.x deferred items: a cleaner boundary between T15.002 and **dependency-confusion / typosquat** patterns (where the malicious package was never the intended dependency but was installed via tooling-error); cohort-level metrics on time-from-publication-to-takedown (the Ledger Connect Kit window was hours; the Atomic Wallet window was days; the asymmetry has direct dollar-loss-prevented implications); explicit treatment of the "the npm publisher credential was compromised but the developer's workstation was not" sub-case which may need its own sub-Technique.
