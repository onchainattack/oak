# OAK-T13.004 — EIP-7702 Delegation Abuse

**Parent Tactics:** OAK-T13 (Account Abstraction Attacks)
**Maturity:** emerging
**Chains:** EVM (Ethereum mainnet primary; cross-chain replay surface across all EIP-7702-activated chains)
**First documented:** Pectra hard-fork activation 2025-05-07; cohort onset within first weeks; canonical Wintermute "CrimeEnjoyor" tag 2025-06-02
**Aliases:** "EIP-7702 phishing", "set-code transaction abuse", "CrimeEnjoyor delegation", "persistent-execution-authority abuse", "delegator sweeper"

## Description

Persistent-execution-authority delegation under the EIP-7702 set-code transaction type, shipped with Ethereum's Pectra hard fork on 2025-05-07. The primitive permits a vanilla externally-owned account (EOA) to sign a single authorisation tuple that registers a delegator contract address whose bytecode is then treated as the EOA's own code for purposes of execution. For every subsequent transaction the EOA participates in, the EVM routes execution through the delegator's bytecode rather than through the EOA's traditional opcode-empty execution model — including, critically, on inbound transactions the victim does not initiate (gifts, airdrops, staking-reward distributions, returned deposits).

Within weeks of Pectra activation, telemetry from Wintermute and GoPlus Security showed >90% of EIP-7702 delegations linked to malicious contracts, dominated by a single sweeper-template family ("CrimeEnjoyor" — Wintermute's tag for a ~120-byte contract whose bytecode is copy-pasted across the majority of malicious delegations).

T13.004 covers four sub-patterns:

- **(a) Malicious-delegator phishing (CrimeEnjoyor cluster)** — fake-frontend lures and other signed-authority phishing patterns produce a single signed authorisation tuple delegating execution authority to an attacker-controlled implementation. Wintermute / GoPlus telemetry: >90% of EIP-7702 delegations linked to malicious contracts; ~79,000 delegations registered for ~2.88 ETH against typically near-zero residual EOAs.
- **(b) Batch-transaction signing-prompt abuse** — drainer-as-a-service infrastructure (Inferno Drainer, Pink Drainer affiliate ecosystem) leverages EIP-7702 batch transactions to execute multiple token transfers in a single user-authorised invocation. Canonical: 2025-05-24 Inferno Drainer ~\$146K MetaMask-delegated-EOA drain; 2025-08-24 \$1.54M single-victim wstETH/cbBTC drain.
- **(c) Chain-agnostic-replay amplification** — EIP-7702 authorisation tuples signed without chain-id binding can be replayed against the same EOA on any EIP-7702-activated chain that shares the EOA's key derivation. A user signing a delegation for a single-chain use-case can find the same delegation active on every EVM chain whose key derivation matches.
- **(d) Future-deposit-trapping optionality** — the CrimeEnjoyor cohort's economic shape is option-portfolio acquisition: by registering a delegation while gas is cheap, the operator captures any *future* deposit to the compromised EOA. The realised cohort loss (~\$12M+ across 15,000+ wallets through 2025-Q4) substantially under-states the addressable surface (450,000+ delegated wallets observed by Wintermute / GoPlus).

T13.004 is **structurally distinct** from adjacent Techniques:

- **T4.001 (Permit2 Authority Misuse)** — the closest existing-Technique analogue. Both involve a victim signing a single approval-equivalent artefact whose downstream effect exceeds the victim's intent. The structural difference is at the artefact level: Permit2 delegates *spending authority over specific tokens* within the EOA's static-code execution model; EIP-7702 *replaces the EOA's execution model entirely* with attacker-supplied bytecode. The blast-radius scaling is qualitative, not merely quantitative.
- **T13.001 (Paymaster Compromise)** / **T13.002 (Bundler MEV)** / **T13.003 (Session-Key Hijacking)** — none capture the EOA-resident delegation primitive that EIP-7702 introduces. T13.001 / T13.002 / T13.003 are ERC-4337 sub-Techniques (UserOperation, EntryPoint, bundler, paymaster, smart-account-issued session keys); T13.004 is the EIP-7702 EOA-conversion sub-Technique. Mechanically distinct, structurally united under the broad T13 (Account Abstraction Attacks) Tactic because the load-bearing primitive is the conversion of an EOA's static-code execution model into a smart-account-equivalent dynamic-code execution model under a signed-authority artefact.
- **T13.003 (Session-Key Hijacking)** — session keys are smart-account-issued ephemeral signers; EIP-7702 delegations are EOA-issued persistent execution-context replacements.

The primitive is option-shaped: the realised loss is bounded by what was held at compromise time and by future inbound transactions to the compromised EOA, not by an upfront authorisation amount. Defender practice should treat *delegated-EOA-set size* as a first-class metric alongside realised-extraction sum.

## Observed indicators

- **Pre-event:** EIP-7702 set-code authorisation prompt delivered to the user via fake-frontend signing-prompt UI; user is asked to sign an authorisation tuple registering a delegator contract address. Distribution surface: fake-frontend phishing UIs (typosquat / clone DEX UIs); compromised legitimate-frontend signing prompts; in-app prompts via trojanised wallet extensions.
- **At-event:** the EIP-7702 set-code transaction is broadcast and the EOA's code is set to point at the delegator's bytecode; subsequent transaction execution against the EOA routes through the delegator. CrimeEnjoyor-family bytecode signature is detectable at the on-chain level via the Wintermute / community detection regex.
- **Post-event:** every subsequent inbound deposit to the compromised EOA is auto-routed by the delegator's bytecode to attacker-controlled receiving infrastructure (Relay Protocol cross-chain hops to consolidate proceeds on Ethereum mainnet). The future-deposit-trapping primitive remains active for the lifetime of the delegation.
- **Cohort signature:** addresses with active EIP-7702 delegations to a small number of bytecode-identical sweeper templates; >90% of all observed delegations clustered into the malicious-delegator set per Wintermute / GoPlus telemetry.

## Detection signals

- **Wallet UX (pre-signing):** surfacing of `SetCodeTransactionType` authorisations as a structurally distinct prompt class, *not* as a generic "approve this transaction" prompt. MetaMask, Trust Wallet, Rabby, Frame have shipped pre-signing delegation-warning UX from 2025-Q4 onward as a distinct prompt class.
- **On-chain detection:** Wintermute Dune dashboard for the CrimeEnjoyor cohort; GoPlus delegation-state API; community-published on-chain detection regex matching the CrimeEnjoyor bytecode template; bytecode-signature clustering on EIP-7702 delegators.
- **Cross-chain delegation-state monitoring:** for any EOA flagged on one chain, enumerate delegation state on every EIP-7702-activated chain whose key derivation matches.
- **Cohort-level metrics:** delegated-EOA-set size as a first-class metric alongside realised-extraction sum; future-deposit-trapping optionality as a class-level harm dimension.
- **Cluster-attribution forensic:** Inferno Drainer infrastructure overlap (per SlowMist 2025-05-24 disclosure); cross-cluster correlation with prior 2024 wallet-drainer affiliate-marketplace infrastructure (per DEV-community on-chain analyst writeup).

## Mitigations

- **Wallet UX (highest-leverage):** surfacing of `SetCodeTransactionType` authorisations as a structurally distinct prompt class, with explicit communication that the signature replaces the EOA's execution model with attacker-supplied bytecode. The cognitive load required to evaluate "what does this delegator's bytecode do" is structurally higher than "what does this allowance amount mean"; presenting the delegation prompt as a generic "approve transaction" surface is operationally inadequate.
- **Wallet vendors:** chain-agnostic-replay detection — flag delegations signed without chain-id binding as elevated risk; cross-chain delegation-state enumeration for any flagged EOA.
- **Cohort-level monitoring:** Wintermute Dune dashboard, GoPlus delegation-state API, community-published detection regex as the cohort-tracking surface; integrate into wallet / infrastructure / exchange-side risk-monitoring layers.
- **Re-delegation surface:** the EIP-7702 re-delegation primitive (re-sign a new authorization tuple to a benign delegator address or to a "no delegate" zero-address sentinel) is structurally available but is gated by the same wallet-UX surface that originally authorised the malicious delegation. Defender practice should accept the permanent-recovery posture as the realistic ceiling and concentrate effort on prevention (wallet-UX), detection (cohort dashboards), and exposure-mitigation (re-delegation UX) rather than on post-event recovery.
- **High-balance custody-side:** segregation of high-balance EOAs from interactive-signing surfaces; air-gap signing for high-value EOAs; refusal to participate in EIP-7702 set-code transactions from interactive-signing wallets.

OAK Mitigation cross-references at v0.1: gap. v0.x candidates: "OAK-M-x EIP-7702-distinct-prompt wallet-UX surfacing"; "OAK-M-x cross-chain delegation-state monitoring"; "OAK-M-x set-code-transaction-class refusal at high-balance custody surface".

## Discussion

T13.004 introduces a **persistent-execution-authority artefact whose blast radius substantially exceeds the EOA-era signed-authority artefacts** (Permit2, ERC-20 approvals, NFT setApprovalForAll). Pre-EIP-7702, every signed-authority artefact in the EVM ecosystem was bounded — bounded in scope (which token? which spender? which collection?), bounded in time (expiry, nonce), bounded in target (which counterparty contract?). EIP-7702 delegation removes all four bounds simultaneously: the delegator's bytecode can execute arbitrary logic, persistently, against any future state of the EOA.

Defender-mental-model literature should articulate this as a **category-level shift, not as an incremental risk increase**. The case argues that the OAK Mitigations layer should add a cross-cutting Mitigation entry covering wallet-UX-surfacing-of-execution-authority-class artefacts, with the EIP-7702 cohort as the leading 2025–2026 worked example.

The CrimeEnjoyor optionality model **decouples cohort scale from realised loss**. The operator(s) paid ~2.88 ETH to acquire ~79,000 future-deposit-trapping options; the addressable cohort exceeds 450,000 delegated EOAs across the broader malicious-delegator ecosystem; the realised extraction (\$12M+) reflects only the small subset of delegated EOAs holding material balances at activation time. Traditional realised-loss aggregation is a substantial under-estimate of the cohort's actual harm shape. OAK worked-example methodology should record both quantities explicitly when the artefact is option-shaped (T13.004 EIP-7702 delegations, T4.001 Permit2 large-allowance approvals, T11.002 supply-chain-installed wallet-extension malware) and should not conflate realised loss with cohort harm.

The Inferno Drainer pivot to EIP-7702 (2025-05-24) is **operational continuity, not novel capability**. Same affiliate marketplace, same fake-frontend lure infrastructure, same proceeds-laundering rails — only the on-chain extraction primitive (delegation + batched execute() call) is new. This is the OAK corpus's broadest pattern across drainer-class incidents: new on-chain primitives are deployment surfaces for *existing* operator clusters, not evidence of new entrant cohorts. Contributors writing future T13.004 worked examples should treat the cluster-substrate-continuity pattern as the operational baseline and require cluster-level forensic evidence before treating a case as a new entrant cohort.

The defender-response timeline for T13.004 is **one of the fastest-coordinated in the OAK 2024–2026 corpus**. Wintermute's CrimeEnjoyor tagging (2025-06-02, less than a month after Pectra activation), GoPlus delegation-state API (2025-Q2), MetaMask pre-signing warning UX (2025-Q4 onward), Yan et al. arXiv academic paper (2025-12), DEV-community detection regex (2025-Q3) — together this constitutes a coordinated defender-response shape that compares favourably with the multi-year vendor-warning lag on T4.001 Permit2 phishing. The structural difference is the substantial pre-fork warning literature (Cyfrin / ThreeSigma / OpenZeppelin pre-Pectra advisories) that primed wallet-vendor coordination from before mass adoption. Future high-blast-radius protocol-level primitives should default to this coordination shape as the operational baseline.

The realistic recovery posture for T13.004 incidents is **permanent**. Compromised EOAs whose authority is delegated to attacker-controlled implementation logic remain sweepable for the lifetime of the delegation. There is no on-chain primitive analogous to the WalletConnect session-revocation surface; there is no equivalent of the Permit2 `lockdown` function for EIP-7702 delegations. Defender effort should concentrate on prevention, detection, and exposure-mitigation rather than on post-event recovery.

Promotion to `stable` will follow once T13.004 carries multiple per-incident worked-example anchors beyond the current single cross-cohort file (which itself anchors three sub-incidents: CrimeEnjoyor cohort, Inferno Drainer 2025-05-24, single-victim 2025-08-24). The 2026-04 Quant-network protocol-counterparty incident is a candidate for a standalone per-incident write-up at v0.x, alongside any subsequent high-net-worth single-victim cases. The class is structurally clean and the field-confirmed-anchor cohort is large enough to support `stable` once the per-incident write-up cohort matures.
