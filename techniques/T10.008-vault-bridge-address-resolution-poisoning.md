# OAK-T10.008 — Vault/Bridge Address Resolution Poisoning

**Parent Tactics:** OAK-T10 (Bridge and Cross-Chain)
**Maturity:** emerging
**Chains:** Cross-chain (any bridge or cross-chain protocol where an off-chain consensus/relayer system resolves a destination address that controls on-chain custody — THORChain Bifrost, Chainlink CCIP, LayerZero relayer, Wormhole Guardian set, Axelar, Multichain MPC, any custody-migration bridge)
**First documented:** 2026-05-15 (THORChain Router exploit — vault churn address poisoning via `MIGRATE:26146440`)
**Aliases:** "vault churn poisoning", "bridge address hijack", "migration address spoofing", "Bifrost address poisoning", "relayer resolution poisoning", "vanity-vault attack"

## Description

A class of bridge-infrastructure attacks where the attacker poisons the address resolution mechanism during a vault migration (churn, rotation, upgrade) to receive on-chain custody of assets meant for the legitimate successor entity. The defining structural feature is the gap between off-chain consensus address resolution and on-chain validation: the consensus/relayer layer directs allowances, approvals, or custody to an address that has NOT been cryptographically proven to belong to the legitimate successor (TSS group, multisig, governance contract).

The attack exploits the absence of on-chain cryptographic identity binding during migration events. The bridge contract correctly enforces access control (only the address holding the allowance can withdraw), but the address that receives the allowance is supplied from off-chain consensus with no verification that it corresponds to the TSS public key or validator set.

T10.008 is structurally distinct from T10.001 (Validator-Signer Key Compromise): no signing key material is compromised. The consensus layer is tricked — through address poisoning, vanity addresses, or relayer manipulation — into directing custody to the wrong address. The TSS keys remain secure; the resolution mechanism is the vulnerability.

T10.008 is structurally distinct from T10.002 (Message Verification Bypass): the on-chain message is cryptographically valid (correctly signed by the old vault). The problem is semantic — the `newVault` address in the message resolves to an attacker-controlled address, not to the legitimate TSS-controlled vault.

Sub-cases observed and anticipated:

- **(a) Vault churn address poisoning.** During vault rotation (churn), the attacker generates a vanity address matching the byte-pattern of legitimate vaults, poisons transaction history with dust from lookalike addresses, and the L1 consensus accepts the attacker's address as the legitimate successor vault. The attacker receives allowances via `transferAllowance()` and later drains all ERC20 tokens. (THORChain 2026-05-15)
- **(b) Relayer address resolution manipulation.** The attacker compromises or feeds malicious data to the relayer/bridge-oracle that resolves destination addresses, causing it to return an attacker-controlled address for a legitimate migration event.
- **(c) Governance address substitution.** During a bridge governance upgrade, the attacker substitutes the new governance/owner address in the migration transaction, gaining control of the bridge's administrative functions.
- **(d) Cross-chain address derivation mismatch.** The attacker exploits differences in address derivation between chains (e.g., Ethereum vs Bitcoin vs Solana) — the TSS pubkey derives to different address formats on each chain, and the attacker inserts their address on one chain where validation is weaker.

## Observed indicators

- A vault migration (`transferAllowance`, `migrate`, `churn`) event where the `newVault` address has no prior history as a vault signer or operator.
- The `newVault` address was funded from a non-protocol-controlled address shortly before the migration.
- Multiple dust transactions from lookalike addresses (sharing byte-pattern suffixes with legitimate vaults) to the `newVault` address in the days before the migration.
- The `newVault` address holds ERC20 allowances idle (no outbound transfers) while processing only ETH/native-currency outbounds that bypass allowance checks via `msg.value`.
- Cross-chain inconsistency: the `newVault` on one chain does not match the vault address derived from the same TSS public key on another chain.

## Detection signals

- **Per-migration address validation:** For each `transferAllowance` or equivalent migration event, verify that `newVault` is derivable from the bridge's TSS public key (or is in the set of pre-announced vault addresses). Alert when `newVault` is not cryptographically bound to the TSS group.
- **Pre-migration address poisoning campaign:** Detect clusters of dust transactions (sub-cent USD) from addresses sharing byte-pattern suffixes with legitimate vaults, all targeting the same recipient, in the window before a vault migration.
- **Post-migration allowance-usage anomaly:** Monitor addresses that receive large allowances during migration. Alert when an address processes only ETH outbounds (bypassing allowance checks) while holding ERC20 allowances untouched for an extended period — the stealth positioning pattern.
- **Cross-chain vault address consistency:** For bridges operating on multiple chains, verify that all `newVault` addresses across chains derive from the same TSS public key material. A mismatch on one chain indicates address poisoning.
- **Relayer address resolution integrity:** Monitor Bifrost-equivalent relayer events. For each `MIGRATE` memo, compare the resolved destination address against the cryptographically expected vault address. Alert on divergence.

## Real-world examples

- **THORChain Router — Ethereum/BSC/Base/Bitcoin — 2026-05-15.** Loss ~$7.4M+ ($3.7M ERC20s on Ethereum, remainder on BSC/Base/BTC). The attacker generated a vanity address (`0x82a5...bDA`) matching the byte-pattern of THORChain vault addresses, sent dust from 4+ lookalike addresses to poison transaction history, and was accepted by THORChain L1 consensus as the legitimate successor vault during `MIGRATE:26146440` at Ethereum block 25,080,575. The old vault called `transferAllowance(router, attacker, USDC, ~798K)`, granting the attacker custody of all ERC20 allowances. The attacker then operated stealthily for ~48 hours (processing 166 legitimate ETH outbound transactions via `transferOut`, which uses `msg.value` and bypasses allowance checks), before draining all ERC20s on May 15 at 08:12 UTC. The vulnerability was the absence of on-chain TSS-pubkey-to-address derivation verification in the `transferAllowance` flow. Full proof: [`investigations/2026-05-15-thorchain-router-exploit.md`](../investigations/2026-05-15-thorchain-router-exploit.md).

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1 (T10.008 detection requires cross-chain vault-address cryptographic verification and pre-migration address-poisoning campaign detection, which is a cross-chain relayer-integrity and address-derivation modelling problem out of MG's single-chain runtime scope).
- Vendor/research coverage: BlockSec Phalcon (post-hoc attack tracing), MetaSleuth (fund flow), ZachXBT (investigative), PeckShield (on-chain monitoring). No widely-deployed runtime detector specifically validates vault-migration address cryptographic binding as of v0.1.

## Mitigations

- **Bridge architects (OAK-M09 — cryptographic vault identity binding):** `transferAllowance` and equivalent migration functions MUST verify that `newVault` is derivable from the bridge's TSS public key. The verification is on-chain: `require(deriveAddress(tssPubkey) == newVault, "not TSS-controlled vault")`.
- **Bridge architects (OAK-M08 — migration timelock):** Mandatory delay (e.g., 24h) between `transferAllowance` and the new vault's ability to call `transferOut`, giving node operators time to detect and revert bad migrations.
- **Bridge operators (OAK-M02 — dual-consensus validation):** Require both L1 consensus AND an on-chain signature from the TSS group confirming the `newVault` address before executing the migration.
- **Bridge operators (vanity address detection):** Deploy pre-migration monitoring that flags addresses receiving dust from lookalike addresses in the window before scheduled churn events.
- **Risk teams:** Before integrating any cross-chain bridge: require (a) documentation of how vault addresses are resolved during migration, (b) evidence that on-chain TSS-pubkey-to-address verification exists, (c) migration timelock duration. Treat bridges without on-chain cryptographic vault identity binding as standing T10.008 surfaces.

## Discussion

T10.008 is the first OAK technique that targets the "address resolution gap" — the architectural seam where an off-chain consensus system (THORChain L1, relayer network, bridge oracle) hands an address to an on-chain contract, and the on-chain contract trusts the address without independent cryptographic verification. This seam exists in virtually every bridge architecture that separates consensus from execution, making T10.008 a broadly applicable class.

The attack has an important temporal signature: the attacker does NOT drain immediately upon receiving allowances. Instead, they operate stealthily — processing legitimate native-currency transactions while holding ERC20 allowances — to build confidence before the drain. This stealth period (48 hours in the THORChain case) is a detection opportunity: an address that processes only ETH outbounds while holding ERC20 allowances idle is anomalous for a legitimate vault.

T10.008 composes with T4.003 (Address Poisoning): the attacker uses user-level address poisoning techniques (dust from lookalike addresses) as a *setup primitive* for the protocol-level address resolution attack. The dust transactions are not the exploit — they are camouflage for the vanity address to blend into the transaction history of legitimate vaults.

False-positive considerations: a freshly-generated vault address during legitimate TSS key rotation will have no prior vault history but MUST still be derivable from the TSS pubkey. The cryptographic derivation check is the definitive discriminator: if `newVault == deriveAddress(tssPubkey)`, the migration is legitimate regardless of the address's novelty. Conversely, a new address that cannot be derived from the TSS pubkey is a T10.008 signal regardless of how much transaction history it has.
