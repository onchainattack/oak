# OAK-T11.006.001 — User-Initiated Plaintext-Equivalent Seed Storage

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure); sub-Technique of OAK-T11.006 (Cold-storage Seed-phrase Exfiltration at Rest)
**Maturity:** emerging
**Chains:** chain-agnostic (BIP39 seed phrase / private key material stored in third-party service; downstream extraction across all chains the affected wallet holds)
**First documented:** 2022-12 (LastPass encrypted-vault exfiltration → 2023–2025 multi-year crypto-drain cohort)
**Aliases:** "password-manager seed storage compromise", "LastPass-class seed-phrase exfiltration", "plaintext-equivalent seed storage"

## Description

The user **explicitly** stores seed-phrase material in a third-party service — a password manager Secure Notes field, an encrypted note, a cloud-synced text file, an email draft, or a browser-saved password — whose encryption is bound to a master password or account credential. The seed phrase is *plaintext-equivalent* in the cryptographic sense because the key that gates the ciphertext is a user-memorised master password whose entropy is typically far below the 128-bit security level of the seed phrase itself.

Compromise path: the third-party service's encrypted vaults are exfiltrated (server-side breach, insider exfiltration, backup theft); attackers brute-force weak master passwords offline against the exfiltrated vault blob; the resulting plaintext yields BIP39 seed phrases that unlock on-chain wallets across all chains the affected wallet holds. The exfiltration-to-drain time window is **unbounded** — once the encrypted vault blob is in attacker hands, offline brute-force proceeds at the attacker's pace, and no server-side credential rotation closes the window.

The structural feature distinguishing T11.006.001 from sibling T11.006.002:

- **T11.006.001** is opt-in by user action: the user chose to put seed-phrase material into the password manager or storage service.
- **T11.006.002** (Implicit Cloud-Custody via Default-On Cloud-Backup) is opt-in by inaction: the user did nothing to opt out of an OS-level default that the wallet vendor also did not opt out of at the app level.

## Observed indicators

- Vault-breach disclosure or exfiltration event at a third-party storage service with non-trivial crypto-holding user fraction (LastPass December 2022; 1Password / Bitwarden Secure-Notes-equivalent usage patterns).
- Per-victim commonality limited to storage-service usage, with no other attack-vector overlap (no SIM swap, no email compromise, no mobile malware, no on-chain phishing).
- Time-shifted drain pattern: the exfiltration event and the realised on-chain drains are separated by weeks-to-years because offline brute-force of weak master passwords takes time.
- Exfiltration artifact: the attacker holds the encrypted vault blob for an unbounded window; users cannot "rotate" the storage-service password to close the window after the fact.

## Detection signals

- **User-side audit:** enumerate whether seed-phrase material was ever stored in a password manager Secure Notes field, a cloud-synced text file, an email draft, or a browser-saved password. Treat any affirmative as a standing T11.006.001 surface.
- **Per-victim cohort methodology (ZachXBT / Taylor Monahan / TRM Labs):** interview affected victims about non-storage attack vectors and find no commonality other than storage-service usage. Process-of-elimination cohort tracking is the canonical forensic surface.
- **Federal-record signals:** forfeiture filings naming a third-party-storage commonality (March 2025 NDCA Larsen filing) provide slow-but-strong attribution.
- **Seed-phrase rotation trigger:** any user whose seed-phrase material was stored in a third-party service whose vaults were exfiltrated must rotate the seed phrase regardless of master password strength. The brute-force window is unbounded.

## Mitigations

- **Cold-storage seed-phrase invariant (highest-leverage):** never store seed-phrase material in any third-party service — password managers, cloud-synced text files, email drafts, browser-saved passwords, screenshots in cloud-synced galleries. Physical media only (paper, metal).
- **Seed-phrase rotation on breach trigger:** if seed-phrase material was ever stored in a service whose vaults were exfiltrated, rotate the seed phrase immediately regardless of master password strength.
- **User-side audit automation:** wallet vendors should provide "where have I stored my seed phrase?" audit guidance in onboarding flows, explicitly listing password-manager and cloud-storage as surfaces to avoid.
- **Per-victim cohort methodology adoption:** for industry forensics — process-of-elimination cohort analysis can establish attribution even when vendor-side disclosure is constrained.

## Discussion

T11.006.001's structural OAK lesson: **a third-party service that stores plaintext-equivalent seed-phrase material at rest is a custody surface the user did not realise they had created.** The user's mental model of self-custody collapses when the seed phrase's effective security is reduced from 128 bits (the seed phrase) to the entropy of the user's master password (typically 20–40 bits) plus the security posture of the third-party service's server-side vault storage.

The **attack-vector-time-shift** between exfiltration (December 2022) and realised drains (2023–2025) is the class's defining temporal feature. Once the encrypted vault blob is in attacker hands, the brute-force window is unbounded. Users who stored seed-phrase material in a service whose vaults were exfiltrated cannot "change their password and move on" — the only operational response is seed-phrase rotation.

The **per-victim cohort methodology** (ZachXBT / Taylor Monahan process-of-elimination interviews) is the cleanest forensic surface when vendor-side disclosure is constrained by litigation or reputation concerns.
