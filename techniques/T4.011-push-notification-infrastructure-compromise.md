# OAK-T4.011 — Push-Notification Infrastructure Compromise

**Parent Tactics:** OAK-T4 (Access Acquisition)
**Maturity:** emerging
**Chains:** cross-chain (iOS/Android push-notification infrastructure is chain-agnostic; the downstream wallet-compromise surface spans all chains the affected wallet application supports)
**First documented:** 2024-06 (CoinStats iOS push-notification infrastructure compromise)
**Aliases:** "push-notification hijack", "mobile-notification compromise", "wallet-app notification-channel attack"

## Description

Compromise of the push-notification infrastructure (Apple Push Notification Service / Firebase Cloud Messaging) that a wallet application or portfolio tracker uses to communicate with its user base. The attacker gains access to the push-notification sending credentials and broadcasts a malicious notification to all users of the application. The notification contains a link to a phishing page, a malicious app update, or — in the canonical CoinStats case — a malicious MetaMask Snap payload distributed through the application's Snap update channel.

The structural feature is the **push-notification channel as a trusted distribution surface**. Users trust push notifications from installed applications because the notification appears in the OS-level notification system, which users perceive as a trusted channel. An attacker who compromises the push-notification infrastructure can reach all users of the application simultaneously through a channel each user has already authorised.

T4.011 is distinct from:
- **T4.008 (Fake-DEX Clone-Frontend Phishing)** — T4.008's phishing surface is a website the user navigates to; T4.011's surface is a push notification the user receives from an already-installed and trusted application.
- **T4.010 (Fake Security-Tool Browser-Extension Phishing)** — T4.010 requires the user to install a malicious extension; T4.011 exploits the existing trust relationship between the user and an already-installed application — no new installation is required.
- **T11.002 (Wallet-Software Distribution Compromise)** — T11.002 is the downstream effect (the wallet software distribution channel is compromised); T4.011 is the upstream entry vector (the push-notification infrastructure that delivers the compromise to the user's device). The CoinStats case sits at the intersection: T4.011 is the entry vector, T11.002 is the downstream wallet-software distribution compromise.

The on-chain extraction surface chains into T4.001 (Permit2 Authority Misuse), T4.004 (Allowance/Approve Drainer), and T4.005 (setApprovalForAll NFT Drainer) once the malicious payload is active in the user's wallet context.

## Observed indicators

- Mass push notification sent to all users of a wallet application or portfolio tracker, containing a link or update prompt inconsistent with the application's normal notification pattern.
- Sudden spike in support requests or social-media reports from users of a specific wallet application reporting suspicious notifications.
- Push-notification infrastructure credentials (APNs key, FCM server key) accessed from an unrecognised IP address or outside the application vendor's known operational IP range.
- Wallet application's Snap/plugin/extension distribution channel serving a payload that was not published through the vendor's normal CI/CD pipeline — the push notification is the delivery mechanism, the malicious payload is the payload.
- Multiple users of the same wallet application reporting wallet draining within a short time window after receiving a push notification — the mass-simultaneous pattern is the T4.011 diagnostic signature.

## Detection signals

- **Push-notification infrastructure access monitoring.** Monitor APNs/FCM credential usage for access from unrecognised IP addresses, outside business hours, or with anomalous sending patterns (mass broadcast to all users vs. routine per-user notifications).
- **Application-side notification-content integrity verification.** Wallet applications should sign push-notification payloads server-side and verify the signature client-side before displaying the notification; unsigned or invalidly-signed notifications should be suppressed.
- **Cross-user drain-event temporal clustering.** Monitor for wallet-draining events clustered in time across multiple users of the same wallet application — the mass-simultaneous pattern within hours of a push-notification broadcast is the T4.011 detection signal.
- **Snap/plugin/extension distribution-channel integrity monitoring.** Monitor the application's Snap/plugin/extension distribution channel for payloads published outside the normal CI/CD pipeline; a malicious payload distributed via push notification and installed through the application's normal update mechanism is the CoinStats-class T4.011 event.

## Real-world examples

## Mitigations

- **Push-notification payload signing (highest-leverage):** Wallet applications should cryptographically sign push-notification payloads server-side and verify the signature client-side before displaying the notification or acting on its content. Unsigned or invalidly-signed notifications should be suppressed at the client.
- **Push-notification infrastructure access hardening:** APNs/FCM credentials should be stored in a hardware security module (HSM) or cloud KMS with access logging; credential rotation on a regular schedule and immediately upon suspected compromise.
- **Application-side per-user notification-content allowlisting:** Wallet applications should maintain a client-side allowlist of expected notification types; notifications containing URLs, update prompts, or action-required language outside the allowlisted set should be suppressed and surfaced to the user as a security alert.
- **Cross-user drain-event temporal-clustering monitoring:** Security vendors and wallet providers should monitor for wallet-draining events clustered in time across multiple users of the same application — the mass-simultaneous pattern is the T4.011 detection signal.
- **User-side hygiene:** Treat any push notification from a wallet application that requests immediate action (update now, verify wallet, claim airdrop) as suspicious; verify through the application's official website or support channel before acting.

**Canonical mitigations:** OAK-M18, OAK-M22, OAK-M40.

## Discussion

T4.011 captures the push-notification infrastructure compromise as a structurally distinct Access Acquisition entry vector. The push-notification channel is a **trusted, OS-level distribution surface** — notifications appear in the OS notification system with the application's name and icon, which users perceive as a trusted channel. An attacker who compromises the push-notification infrastructure reaches all users of the application simultaneously through a channel each user has already authorised. No new installation, no phishing page, no social-engineering interaction — the notification arrives through the same channel that delivers legitimate application notifications.

The structural distinction from T4.008 (Fake-DEX Clone-Frontend) and T4.009 (Pre-Token Anticipation Phishing) is the **direction of trust flow**. In T4.008/T4.009, the user navigates to an attacker-controlled surface (a fake website, a fake token page). In T4.011, the attacker reaches *into* the user's device through a channel the user already trusts — the push-notification channel of an installed application. The attack inverts the trust flow: instead of "user goes to attacker," it's "attacker comes to user through a trusted channel."

The CoinStats June 2024 case is the canonical T4.011 anchor because it demonstrates the full chain: push-notification infrastructure compromise → malicious Snap distribution → mass wallet draining. The case sits at the intersection of T4.011 (entry vector) and T11.002 (downstream wallet-software distribution compromise), and the distinction matters for mitigation composition: T4.011 mitigations (push-notification payload signing, infrastructure access hardening) prevent the entry vector; T11.002 mitigations (Snap/plugin distribution-channel integrity monitoring) detect the downstream effect.

The class is marked `emerging` rather than `stable` because only one field-confirmed incident (CoinStats June 2024) anchors the technique at v0.1. Additional push-notification infrastructure compromise incidents are expected as the wallet-application ecosystem matures and the push-notification attack surface is recognised as a distinct entry vector.
