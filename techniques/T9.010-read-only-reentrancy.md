# OAK-T9.010 — Read-Only Reentrancy

**Parent Tactics:** OAK-T9 (Smart-Contract Exploit)
**Maturity:** emerging
**Chains:** EVM (primary); any chain whose execution model allows `staticcall` to a contract that is mid-execution in the same transaction
**First documented:** 2022 (class characterised in ChainSecurity's Curve-LP-oracle advisories; Market.xyz on QuickSwap, October 2022, is the earliest clean in-the-wild operational anchor)
**Aliases:** "view reentrancy", "staticcall reentrancy", "stale-read reentrancy", "cross-protocol view manipulation"

## Description

A smart-contract exploit in which the attacker re-enters a `view`/`pure` function on a target contract while the target is mid-execution — reading state that is temporarily inconsistent because the outer execution frame has not yet committed its writes. The re-entry itself does not mutate state on the target (it is a `staticcall`, which the EVM enforces as read-only), and no state is double-spent — the loss is realised on a **third protocol** that consumes the stale view-function output for a pricing, accounting, or liquidation decision. The defining characteristic is that the target contract is "innocent" in the sense that no money leaves it because of the re-entry; the victim is the consumer protocol that trusted the target's view-function output at a moment when that output did not reflect the post-transaction settled state.

T9.010 is structurally distinct from T9.005 (Reentrancy — state-mutating): in T9.005, the attacker re-enters a function that mutates state (typically a withdraw/transfer path), extracting value directly from the target through a double-spend. In T9.010, the attacker re-enters a `view` function, reads a stale value, and the loss occurs on a third protocol that trusted the view function's result. The T9.005 mitigation (checks-effects-interactions + `ReentrancyGuard`) does not close T9.010 because no state mutation occurs during the re-entry — `ReentrancyGuard` blocks re-entry to state-mutating functions; it does not block `staticcall` to view functions. T9.010 is operationally a **cross-protocol trust-substrate attack**: the consumer protocol's assumption that a view-function read returns settled state is violated at a moment when the target is mid-execution.

The canonical reference is the October 2022 Market.xyz incident on QuickSwap (Polygon), where a Market.xyz lending market priced Curve LP-token collateral via the pool's `get_virtual_price()` view. Curve's `remove_liquidity` transfers a native asset to the caller before it settles the pool's accounting, so during that callback the LP supply has decreased while the invariant has not been recomputed and `get_virtual_price()` returns an inflated value. The attacker re-entered Market.xyz's borrow path inside that window, the lending market read the inflated price, and a loan was extended against over-valued collateral — realising the loss on the lending market, not on Curve. This is the cleanest stand-alone worked example of T9.010: the Curve pool lost nothing from the `staticcall` read; Market.xyz lost the loan principal. It needs no compiler bug — the mid-execution window is a by-design property of `remove_liquidity`, which is what distinguishes it from the July 2023 Curve/Vyper event (a compiler-level `@nonreentrant`-lock failure whose primary losses were *direct* reentrancy drains of the pools themselves, classified under T9.005; Market.xyz was not part of it).

## Observed indicators

- A transaction trace that shows `staticcall` from protocol C (consumer) to protocol T (target) occurring while protocol T's outermost call frame in the same transaction has not yet returned — the view-function read occurs at a point where T's storage is temporarily inconsistent.
- The consumer protocol (C) extends credit, executes a swap, or settles a liquidation against the stale view-function output within the same transaction — the loss is crystallised at C, not at T.
- Post-event: a reproducible test case demonstrating that the view function returns value V1 at a block when a re-entrant transaction is in-flight, and returns value V2 (< V1, with V1 – V2 representing the inflated component) when called at the same block with no re-entrant transaction — V1 is the stale read, V2 is the settled read.
- The target contract's view function is documented or understood to be consumed by third protocols as a pricing or accounting reference — this is a structural T9.010 indicator at the integration layer, independent of any specific attack transaction.

## Detection signals

- Pre-deployment: for every protocol integration where protocol C reads a view function on protocol T for pricing/accounting, verify that T cannot be mid-execution (re-entered) at the moment C reads it. This is an integration-level design review, not a per-contract review.
- Per-transaction monitoring at deployed protocols: alert on transactions where a `staticcall` to protocol T occurs while T's outermost call frame in the same transaction has not yet returned, AND the `staticcall` result is consumed by protocol C for a value-transfer decision (loan, swap, liquidation) within the same transaction.
- Read-only reentrancy guard at the consumer protocol: consumer protocols should adopt the discipline of only reading view functions from target contracts whose outermost call frames have returned in the current transaction. This is implementable as a per-transaction check: before consuming a view-function result, verify that the target contract is not mid-execution.
- Post-deployment integration audit: for each protocol, enumerate all external view-function reads consumed for pricing/accounting decisions; for each, verify that the target contract's reentrancy surface (including read-only reentrancy) has been assessed. Treat any integration where the target contract's view function can be read mid-execution as an elevated T9.010 surface.

## Real-world examples

- **Market.xyz — QuickSwap (Polygon) — 2022-10-24.** A Market.xyz lending market priced Curve LP-token collateral via the pool's `get_virtual_price()` view. Curve's `remove_liquidity` transfers a native asset to the caller before settling the pool's accounting; during that callback the LP supply has decreased while the invariant has not been recomputed, so `get_virtual_price()` returns an inflated value. Using flash-loaned capital, the attacker added liquidity, called `remove_liquidity`, and re-entered Market.xyz's borrow path inside the callback — the market read the inflated price, extended a loan against over-valued collateral, and the attacker withdrew ~\$220K of MAI (QiDAO-seeded liquidity; no user deposits lost). The Curve pool was not the loss surface — the `staticcall` read moved none of its assets. This is the canonical OAK T9.010 anchor, and the earliest clean in-the-wild instance. Distinct from the July 2023 Curve/Vyper event (compiler-level lock failure, direct drains, T9.005 — Market.xyz not involved). See `[quillauditsmarketxyz2022]`, `[chainsecuritycurvelp2022]`.
- **Sentiment Protocol — Balancer LP integration — Ethereum — 2023-04.** Sentiment Protocol accepted Balancer LP tokens as collateral and priced them via a view-function read of the Balancer pool's state. An attacker exploited a cross-protocol reentrancy path (the Balancer pool was mid-execution during a callback from Sentiment's own liquidation path) to read a stale LP-token price, borrow against the inflated valuation, and extract the loan principal. The Balancer pool itself lost nothing; the loss was on Sentiment. The case is structurally T9.010: a lending protocol consumed a stale view-function read from a pool that was mid-execution.
- **Audit-firm and developer literature — 2023–2025.** Read-only reentrancy is increasingly cited as a distinct attack class in audit-firm advisories (Trail of Bits, Spearbit, OpenZeppelin) and DeFi-developer security guides. The Solidity docs reference the pattern; the EVM's `staticcall` semantics (no state mutation) are what make the class distinct from T9.005 but also what make it harder to detect at the per-contract layer — the target contract's own invariants are not violated.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1 (T9.010 detection requires per-transaction call-trace monitoring that identifies `staticcall` to a target contract that is mid-execution in the same transaction, combined with awareness of which consumer protocols consume which view-function reads — a cross-protocol integration-layer analysis that is out of MG scope).
- Vendor coverage: Forta (can detect read-only reentrancy patterns at the call-trace layer with custom detection bots), BlockSec PhalconHQ (call-trace-level monitoring of cross-protocol reentry), OpenZeppelin Defender (runtime monitoring with custom rules). Pre-deployment: audit firms (Trail of Bits, Spearbit, OpenZeppelin) treat read-only reentrancy as a standard protocol-integration review item.

## Mitigations

- **Consumer protocols (OAK-M01 — cross-protocol integration audit):** enumerate every external view-function read consumed for pricing/accounting decisions; for each, verify that the target contract cannot be mid-execution (re-entered) at the moment of the read. Treat any integration where the target's view function can return mid-execution state as an elevated T9.010 surface.
- **Consumer protocols (read-only reentrancy guard):** implement a per-transaction check that verifies the target contract's outermost call frame has returned before consuming its view-function output for a value-transfer decision. This is a lightweight runtime guard that closes the stale-read window.
- **Target protocols (view-function purity):** where feasible, design view functions to return a snapshot of state that is atomic per-transaction, rather than reading live storage that may be mid-update. This is aspirational at v0.1 — most deployed protocols do not implement this.
- **Risk teams:** maintain per-protocol inventories of cross-protocol view-function integrations; treat any integration where the consumer protocol reads a view function from a protocol known to have a reentrancy surface (even a guarded one) as elevated T9.010 risk.

## Discussion

T9.010 is the reentrancy class that does not look like a reentrancy at the target contract. The target contract's state is not violated — no double-spend occurs, no assets leave the target. The loss occurs on a third protocol that trusted the target's view-function output. This makes T9.010 an integration-layer attack: the vulnerability is not in the target contract's code (which may be perfectly correct post-execution) but in the consumer protocol's assumption that view-function reads are atomic with respect to the target's execution state.

The class is marked emerging rather than stable because (a) public awareness of T9.010 as a discrete class is relatively recent (2023–2024), (b) most DeFi protocols do not yet implement read-only reentrancy guards at the consumer layer, and (c) the surface grows with every new cross-protocol integration that consumes a view function for pricing/accounting. As DeFi composability deepens (more protocols reading more view functions from more targets), T9.010's surface expands proportionally; the detection and mitigation surface lags the integration surface at v0.1.

A structural observation: T9.010 is one of the few OAK Techniques where the victim protocol and the vulnerable protocol are different. In T9.005, the target is the victim; in T9.010, the target (whose view function was read) may be entirely unaware it was party to an exploit. This has implications for incident response — forensic analysis must identify the consumer protocol's stale-read moment, not only the target's mid-execution window.

False-positive considerations: a `staticcall` from protocol C to protocol T while T is mid-execution is not automatically T9.010. Legitimate cross-protocol composability routinely involves view-function reads during multi-contract transactions (e.g., a DEX aggregator reading pool reserves to compute the optimal route while the user's transaction is in-flight). The distinguishing signal is that the stale read produces a value that (a) differs materially from the post-transaction settled value, and (b) causes protocol C to take an economic action (loan, swap, liquidation) that would not have been taken against the settled value. The defender heuristic should focus on the economic-action-consumer leg, not on the mere presence of a mid-execution `staticcall`. Consumer protocols that implement the read-only reentrancy guard (verifying the target's outermost frame has returned before consuming the view-function output) will produce zero T9.010 false positives because the guard eliminates the stale-read window regardless of the target's reentrancy surface.

v0.x deferred items: a structured inventory of major DeFi protocol integrations that consume view-function reads for pricing/accounting decisions (which consumer reads which target's view function, for what economic decision, at what call depth); a reference implementation of the consumer-side read-only reentrancy guard as a Solidity library that protocols can import with minimal integration effort; distinction between read-only reentrancy and the adjacent "price-oracle-staleness" class (T9.001) where the oracle read is also a view function but the staleness arises from oracle-update-cadence rather than from mid-execution stale state.

Attribution-strength considerations: T9.010 attribution is typically at the `pseudonymous-attacker` strength level — the attacker address that submits the outer transaction is attributable at the on-chain identity layer, and the read-only reentrancy trace path (`staticcall` to target T while T's outermost frame has not returned, followed by consumer C's economic action against the stale read) is reproducible from the transaction trace. The consumer protocol that lost funds is attributable at the protocol-identity layer. The target protocol whose view function was read may have no role in the attack beyond being the source of stale state; its audit posture and reentrancy-guard status are relevant to the forensic attribution but not to the attacker identity.

T9.010 composes with T9.001 (Oracle Price Manipulation) when the view function being read is an oracle price feed — the stale-read mechanism is T9.010, but the price that is read may itself have been manipulated through T9.001-class input-venue moves. The distinction is at the temporal layer: T9.001 manipulates the input price; T9.010 exploits the stale-read window during which the manipulated price has not yet settled. The two compose in the Curve July 2023 + Market.xyz incident chain: the Vyper-level reentrancy (T9.005) manipulated the Curve pool's state; Market.xyz consumed the stale `get_virtual_price()` (T9.010); the distorted LP-token price was the loss surface.
