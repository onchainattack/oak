# OAK-T13.001.003 — Paymaster Reentrancy

**Parent Tactics:** OAK-T13 (Account Abstraction Attacks)
**Maturity:** observed
**Chains:** EVM (ERC-4337 EntryPoint v0.6 / v0.7 / v0.8 / v0.9 deployments); EVM L2s with native account abstraction
**First documented:** 2023 (ERC-4337 reference-implementation guidance warns explicitly against external calls before sponsorship-accounting finalisation); class continues to surface in audit-firm advisories on deployed paymasters
**Aliases:** "validatePaymasterUserOp reentrancy", "postOp reentrancy", "paymaster validation-surface reentrancy", "paymaster-specific T9.005"

## Description

A paymaster-specific instance of the T9.005 reentrancy class restricted to the `validatePaymasterUserOp` / `postOp` call surface. The paymaster makes an external call (token transfer, oracle read, integration callback) before its sponsorship-accounting state is finalised; the attacker re-enters via a token hook (ERC-777 `tokensReceived`, ERC-721/1155 `onERC*Received`) or a peer-protocol callback and either:

- (i) re-validates a UserOp against pre-finalised paymaster state,
- (ii) reads stale paymaster balance / allowance accounting from a peer protocol (read-only reentrancy variant),
- (iii) drives the paymaster into an inconsistent post-state.

T13.001.003 is a strict subset of T9.005 in mechanism, but the call surface (`validatePaymasterUserOp` / `postOp`) and the storage surface (paymaster sponsorship accounting, deposited-balance ledger at the EntryPoint) are specific to the ERC-4337 contract layout. A T9.005 detector tuned for generic reentrancy may miss a paymaster-specific call-graph shape that re-enters the EntryPoint's deposit-balance accounting from inside `postOp`. ERC-4337 reference-implementation guidance specifically warns that `validatePaymasterUserOp` must enforce strict checks and avoid surrendering execution flow to attacker-controlled code before accounting finalisation `[erc4337spec]`.

T13.001.003 is part of the OAK-T13.001 (Paymaster Compromise) umbrella; see the parent for the architectural distinction from T9.005 generic reentrancy.

## Observed indicators

- A paymaster whose `validatePaymasterUserOp` performs an external call (token transfer, oracle read, peer-protocol invocation) before finalising sponsorship-accounting state.
- A paymaster whose `postOp` makes an external call (e.g., to an integration callback, an oracle, or an ERC-20 token with a transfer hook) without a reentrancy guard spanning the validation / `postOp` surface.
- A paymaster that accepts ERC-20 gas tokens with reentrancy-relevant hooks (ERC-777 `tokensReceived`, ERC-1363 receive-callback, certain rebasing tokens) without explicit hook-aware design.
- Cross-protocol integrations where a peer protocol reads the paymaster's deposited-balance / sponsorship-accounting view function during a transaction in which the paymaster is mid-execution (read-only-reentrancy surface).

## Detection signals

- **`validatePaymasterUserOp` / `postOp` review.** Verify checks-effects-interactions discipline across the validation / `postOp` boundary; verify that any external call is preceded by sponsorship-accounting finalisation; verify reentrancy guards on the validation surface.
- **Reentrancy-guard granularity verification.** Verify the reentrancy guard spans the *set* of functions sharing paymaster-accounting state (`validatePaymasterUserOp`, `postOp`, deposit / withdraw helpers), not per-function — a per-function guard does not block cross-function reentrancy.
- **Pre-deployment static analysis.** Slither's `reentrancy-eth`, `reentrancy-no-eth`, `reentrancy-benign` detectors, MythX, Securify, tuned for the ERC-4337 paymaster surface.
- **Per-transaction call-trace monitoring.** Alert on transactions whose internal trace shows the paymaster contract being re-entered (paymaster `to` address appearing multiple times within a single outer call) before the outer EntryPoint call returns.
- **Read-only reentrancy detection.** For paymasters whose deposited-balance / sponsorship-accounting view functions are consumed by third protocols, monitor for `staticcall` reads against the paymaster while the paymaster's outermost frame in the same transaction has not yet returned.

## Mitigations

- **Paymaster architects:** treat `validatePaymasterUserOp` as a high-assurance boundary — apply checks-effects-interactions discipline strictly; never make external calls before sponsorship-accounting finalisation; install reentrancy guards on the validation / `postOp` surface explicitly (do not rely on the EntryPoint's reentrancy posture as a substitute); apply the guard across the *set* of functions sharing paymaster-accounting state, not per-function; treat ERC-777 / ERC-721 / ERC-1155 / ERC-1363 hook surfaces as implicit external calls and audit them as such; treat view functions read by third protocols as part of the reentrancy surface (read-only reentrancy mitigation).
- **Risk teams:** when reviewing a new paymaster integration, require (1) an audit report covering reentrancy at the validation / `postOp` surface, (2) the reentrancy-guard granularity statement (per-function vs state-sharing-set), (3) the gas-token allowlist with explicit handling of hook-bearing tokens, (4) the inventory of view functions consumed by peer protocols.
- **Cross-comparison:** unlike generic T9.005, a `ReentrancyGuard` on user-facing functions is not sufficient — the validation / `postOp` surface must be guarded as its own state-sharing set; the EntryPoint's reentrancy posture does not substitute for paymaster-internal guarding.

## Discussion

T13.001.003 is included as a separate sub-Technique under T13.001 — rather than folded into T9.005 — because the paymaster's call surface and storage surface are ERC-4337-specific and the relevant detection rules (call-graph monitoring of `validatePaymasterUserOp` / `postOp`; read-only-reentrancy monitoring of paymaster view functions consumed by peer protocols) require contract-layout-aware tooling. A generic T9.005 reentrancy detector treats the paymaster as just another contract and may miss the EntryPoint-deposit-balance-accounting reentry path entirely.

Read-only reentrancy at the paymaster surface is a particularly under-monitored sub-pattern: a paymaster's deposited-balance / sponsorship-accounting view functions are increasingly consumed by peer protocols (gas-abstraction routers, sponsorship-marketplace aggregators) that may take economic action on stale reads. As paymaster integrations multiply this surface will grow.
