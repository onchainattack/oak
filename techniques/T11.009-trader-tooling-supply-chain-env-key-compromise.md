# OAK-T11.009 — Trader-Tooling Supply-Chain Compromise targeting `.env` Private Keys

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure)
**Maturity:** emerging
**Chains:** chain-agnostic (the substrate-of-extraction is the developer-environment plaintext key file; downstream extraction occurs across whichever chains the bot operates on — Polygon, Solana, Ethereum, Base, Arbitrum, BNB Chain are all in scope across the cohort)
**First documented:** 2025-12 / 2026-01 (Polymarket trader-tooling supply-chain compromise via npm `polymarket-clob` and `dev-protocol` GitHub-org hijack); the broader cohort context spans 2024–2026 with overlapping infrastructure to DPRK-attributed BeaverTail / InvisibleFerret npm campaigns
**Aliases:** "trader-bot npm supply-chain", "developer-environment .env exfiltration", "GitHub-org-hijack trojan-bot distribution", "wallet.json infostealer via package registry"

## Description

Compromise of trader-tooling distribution channels — npm package registry, PyPI, GitHub repositories — used to ship code that exfiltrates plaintext private-key material from developer environments. The target cohort is *bot builders, quant developers, and copy-trading-bot users* who maintain plaintext private keys in `.env` files, `wallet.json` / `keys/*.json` files, or other developer-environment storage to enable automated signing. The malicious package or repository contains code that reads these key files at install / build / first-run time and exfiltrates them to attacker-controlled infrastructure.

T11.009 is **structurally distinct** from adjacent T11 sub-Techniques:

- **T11.002** (Wallet-Software Distribution Compromise) — substitutes the wallet binary delivered to *end-user* wallets (Atomic Wallet pattern). T11.009 targets *developer environments* — the bot's runtime is the victim, not the end-user's wallet UI.
- **T11.006** (Cold-storage Seed-phrase Exfiltration at Rest) — third-party storage-service-at-rest failure surface (LastPass / iCloud-MetaMask). T11.009's failure surface is the *developer's own machine* where plaintext keys are stored to enable bot operation; the compromise vector is the package registry, not the storage service.
- **T11.007** (Hardware-wallet Supply-chain / Physical-access Compromise) — brand-impersonation hardware substitution and active phishing for seed-phrase exfiltration. T11.009 targets developers who do not use hardware wallets (because their bots require automated signing).
- **T8** (Operational Reuse) — laundering / reuse-pattern-side. T11.009 is custody-side: the substrate of extraction is the plaintext key file in the developer environment.

The cohort overlaps DPRK-attributed npm-supply-chain operations at the infrastructure-fingerprint layer (BeaverTail / InvisibleFerret / Vidar Stealer C2 endpoint reuse) but the attribution at the named-individual level remains pseudonymous for most sub-campaigns at v0.1.

## Observed indicators

- **Package-registry fingerprints:** trojanised npm / PyPI packages whose names target a specific platform's tooling ecosystem (`polymarket-clob`, `polymarket-copytrading-bot-*`); sibling packages with seemingly-unrelated names (`sha256-validation`, `sha256-validator-pro`, `solana-utils-sdk`, `synced-plus-agent`) maintained by the same publisher; download-count manipulation to produce social-proof signal.
- **Repository-side fingerprints:** GitHub repositories under hijacked organisations (the `dev-protocol` GitHub-org hijack in the canonical case is a real Japanese DeFi project whose org credentials were compromised); polished READMEs with artificially-inflated star counts; repositories that present as genuine open-source bot projects on first inspection.
- **Code-side fingerprints:** code that reads `.env`, `wallet.txt`, `wallets.json`, `keys/*.json`, `~/.config/solana/id.json`, or `~/.aws/credentials` at install / build / first-run time; exfiltration via HTTPS POST to an attacker-controlled IP / domain; secondary backdoor capabilities (port-22 SSH backdoors, IP fingerprinting, reverse shells).
- **Infrastructure overlap:** C2 endpoint overlap with known infostealer / DPRK npm-supply-chain campaigns (Vidar Stealer in the canonical case; BeaverTail / InvisibleFerret cohort patterns).

## Detection signals

- **Package-registry-side:** automated scanning for packages that read plaintext key files at install time; reputation-based filtering (publisher age, download history, sibling-package analysis); rapid-takedown coordination across npm / PyPI / GitHub when malicious packages are identified.
- **Developer-environment-side:** dependency-source-pinning (lock files; integrity hashes; explicit allowlists for production dependencies); `.env` / `wallet.json` access auditing at the OS layer; isolation of bot-runtime environments from general developer environments (containers, VMs, separate user accounts).
- **Operator-side (bot-builder):** exclusive use of hardware-wallet-backed signing (Ledger / Trezor) or remote-signing services (Fireblocks, Web3Auth MPC) where automated signing is required; refusal to maintain plaintext private keys in `.env` files for any non-trivial holdings.
- **Public-disclosure surfaces:** Cyb3rhawk / ADHD security-research disclosures; StepSecurity / Socket.dev / Snyk / Phylum supply-chain monitoring services; ZachXBT and SEAL Threat Intel public disclosures of trader-tooling compromise events.

## Mitigations

- **Bot-builder / quant-developer hygiene (highest-leverage):** prefer hardware-wallet-backed signing (Ledger / Trezor) or remote-signing services (Fireblocks, Web3Auth MPC, AWS KMS / GCP KMS) where automated signing is required; refuse to maintain plaintext private keys in `.env` files for any non-trivial holdings. Where plaintext keys are operationally unavoidable, isolate the bot-runtime environment from general developer environments via containers / VMs / separate user accounts.
- **Dependency-source-pinning:** lock files with integrity hashes; explicit allowlists for production dependencies; refusal to install packages from publishers without verified history. Treat any package whose author is newly-created or whose download history is sparse as elevated-risk regardless of name plausibility.
- **GitHub-organisation hardening:** enforce hardware-MFA for all maintainers of organisations that publish bot tooling; rotate credentials regularly; treat any unsolicited collaboration request to a maintainer as a potential credential-phishing precursor (the `dev-protocol` GitHub-org hijack is the canonical T11.009 cohort case for credential-compromise-as-distribution-substrate).
- **Package-registry-side enforcement:** automated scanning for packages that read plaintext key files at install time; reputation-based filtering; rapid-takedown coordination. The npm / PyPI / GitHub takedown infrastructure has improved through 2024–2026 but the time-to-detection vs. time-to-install window remains the binding constraint.
- **Cohort-level attribution coordination:** infrastructure-fingerprint sharing across security-research firms (Vidar Stealer C2 endpoint reuse; BeaverTail / InvisibleFerret cohort patterns) accelerates detection and takedown of follow-on campaigns.
- **Platform-side guidance:** platforms with significant trader-tooling ecosystems (Polymarket, Hyperliquid, Pump.fun, Solana memecoin-tooling ecosystem) should publish authoritative-tooling registries — vetted lists of bot frameworks, copy-trading services, and analytics tooling — so that users can distinguish between legitimate tooling and trojan repositories.

OAK Mitigation cross-references at v0.1 are a **gap** — no existing OAK-M* entry directly maps to the developer-environment supply-chain hardening surface that T11.009 most leverages. v0.x candidate Mitigation entries: "OAK-M-x trader-tooling supply-chain dependency-pinning"; "OAK-M-x bot-runtime hardware-signer invariant"; "OAK-M-x platform-side authoritative-tooling-registry publication".

## Discussion

T11.009 closes a structural gap in OAK v0.1's T11 coverage. The dominant v0.1 T11 worked examples (Bybit / Safe{Wallet} for T11.001, Atomic Wallet for T11.002, WazirX for T11.003) focus on end-user wallet binaries, organisational signing-vendor flows, and on-chain multisig manipulation. T11.009 introduces a custody surface that lives in *developer environments* — the bot's runtime substrate — which is structurally distinct from end-user wallet binaries (T11.002 target cohort) and from organisational signing-vendor surfaces (T11.001 target cohort).

The structural OAK lesson is that **the bot-builder / quant-developer cohort is a distinct target population with its own custody surface**. End-user wallet hygiene (hardware wallet, no plaintext seed storage) maps poorly to bot-runtime requirements (automated signing, programmatic transaction submission, sub-second latency requirements). The cohort's residual reliance on `.env` files for operational reasons creates a custody surface that the package-registry / repository-distribution channels can attack.

The cohort overlap with DPRK-attributed npm-supply-chain operations is structural rather than incidental. The BeaverTail / InvisibleFerret npm cohort (see [`actors/OAK-G01-lazarus.md`](../actors/OAK-G01-lazarus.md)) shares infrastructure-fingerprints (Vidar Stealer C2 endpoints; package-distribution patterns; trojan-payload similarity) with the canonical T11.009 case. The named-individual attribution remains pseudonymous at v0.1 — "Trust412" handle for the GitHub-org hijack; npm publisher pseudonyms for the package campaign — but the operator-cluster-level attribution converges on a structured operator-cluster rather than opportunistic single-operator efforts.

The economic-ceiling-vs-substrate-compromise distinction is structurally important. The realised on-chain loss in the canonical case is small (~$44 confirmed) because most affected developers had not yet funded the wallets associated with the trojan'd `.env` files at the time of takedown. The operative risk is *the substrate compromise* — the trader-tooling ecosystem's residual exposure to follow-on campaigns and to the persistence of trojan repositories that survive initial takedown waves. Defender practice should weight the substrate-compromise dimension at least as heavily as the realised-loss dimension.

The platform-side authoritative-tooling-registry mitigation is a high-leverage but operationally awkward intervention. Platforms with significant trader-tooling ecosystems can publish vetted lists of bot frameworks and analytics tooling, but the publication imposes a ongoing curation burden and exposes the platform to liability framing. The Polymarket case at v0.1 has not seen platform-side registry publication; this is a v0.x candidate intervention.

Promotion to `stable` will follow once additional cohort cases land (Hyperliquid trader-tooling cohorts; Pump.fun copy-trading-bot cohorts; Solana memecoin-tooling-ecosystem cohorts) and once at least one attribution case reaches the `confirmed` strength label (DOJ-named-individual attribution to a specific operator-cluster behind an npm-supply-chain campaign targeting trader tooling).
