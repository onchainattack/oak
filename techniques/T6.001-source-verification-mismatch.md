# OAK-T6.001 — Source-Verification Mismatch

**Parent Tactics:** OAK-T6 (Defense Evasion); modifier on T1 (Token Genesis) and T9 (Smart-Contract Exploit) classes when paired
**Maturity:** stable
**Chains:** EVM (primary)
**First documented:** widespread from approximately 2020 onward
**Aliases:** "verified-but-not-really", "fake source verification", "bytecode-source mismatch"

## Description

A class of evasion patterns in which a contract's deployed bytecode differs from the source code claimed at the block explorer or via off-chain marketing — typically because the verified source corresponds to an earlier, benign deployment that has since been replaced by a different bytecode at the same address (via proxy upgrade, `selfdestruct` + redeploy via CREATE2, or claim-only-no-actual-verification). From a defender's perspective, T6.001 defeats static analysis tools that read the verified source — the analyst sees one contract while the chain executes a different one — and is the canonical OAK-T6 modifier on T1 and T9 classes when the underlying primary Technique would otherwise be detectable via source-level review.

T6.001 is the first standalone OAK-T6 (Defense Evasion) Technique because the failure mode is *generic* (it modifies any source-readable Technique) and the detection signal is *unambiguous* (compare deployed bytecode against the bytecode produced by recompiling the verified source — they should match within compiler-deterministic noise).

## Observed indicators

- A contract whose verified-source compilation does not produce the deployed bytecode under reasonable compiler-version / optimisation-flag combinations.
- A contract whose verified-source claim has been updated multiple times against the same address (suggesting either legitimate proxy-pattern usage or T6.001 source-shuffling — the distinction requires deeper review).
- A contract whose source is "verified" only via partial-match (matching constructor arguments and metadata but with bytecode differences) — partial-match verification is a real block-explorer feature and is not in itself indicative of T6.001, but partial-match plus a marketing claim of full verification is.

## Detection signals

- Deterministic recompile: pull the verified source, recompile under the claimed compiler version + flags, compare the produced bytecode against the on-chain deployed bytecode. Non-match (beyond compiler-deterministic noise) is the canonical T6.001 signal.
- Proxy-pattern review: where the contract is a proxy, ensure the implementation address used at runtime corresponds to the verified-source claim; T6.001 frequently combines with proxy-upgrade patterns to point the proxy at unverified implementation contracts.
- CREATE2-replay detection: if a contract was deployed via CREATE2 and a `selfdestruct` event is observed, treat any subsequent redeploy at the same address as a candidate T6.001 vector.

## Real-world examples

- **Cohort scale — multi-chain — 2020 onward** — T6.001 is most visible at the cohort level rather than at the per-incident level: many memecoin and rug-pull tokens combine T1.001 / T1.002 with T6.001 to evade pre-trade static analysis. Industry retrospectives (`[chainalysis2025rug]`, `[slowmist2024report]`) cover the cohort-scale impact without naming individual T6.001 cases as canonical, because the underlying primary Technique (T1, T5, etc.) is typically the more publicly-reportable framing.
- **Per-incident named cases** — OAK does not at v0.1 name a single canonical T6.001 incident. Contributors with public forensic write-ups of specific named incidents whose primary failure mode is T6.001 (rather than a primary T1 / T9 incident with T6.001 as modifier) are encouraged to submit them via PR.

### Additional incidents

- [`examples/2025-12-uspd-cpimp-clandestine-proxy.md`](../examples/2025-12-uspd-cpimp-clandestine-proxy.md) — USPD CPIMP "Clandestine Proxy In the Middle of Proxy" stablecoin exploit (Ethereum, December 2025). Anchors a discrete advanced-evasion sub-class of T6.001: the attacker engineered the on-chain artefacts (event payloads + EIP-1967 storage slot) that drive Etherscan's display logic to *match* the audited code by construction, while the runtime authority structure routed through an attacker-controlled shadow proxy. Verification surface is the attack target rather than bypassed; defender-side Mitigation surface migrates to runtime-authority verification at the EIP-1967 slot level. Three-month dwell (2025-09-16 deploy-init frontrun → 2025-12 mint-and-drain), ~\$1M extraction.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1 (T6.001 detection is in scope for MG's static-analysis pipeline; calibration in progress).
- Vendor coverage: block-explorer "verified bytecode" checks (Etherscan, Sourcify); pre-deployment audit firms (Halborn, Trail of Bits, OpenZeppelin); deployed-contract monitoring (Forta, OpenZeppelin Defender) for proxy-upgrade events tied to source-claim mismatch.

## Mitigations

- **Risk teams / static-analysis tooling:** treat `Etherscan-verified` claims as a starting point, not a conclusion; deterministic-recompile against the claimed source as a confirming step before downstream Technique-level static analysis.
- **Block explorers (highest-leverage):** continue to mature partial-match-vs-full-match labelling; surface the distinction prominently in the UI.
- **Standardisation:** Sourcify-style cryptographic source-verification (where the on-chain bytecode is verified against an off-chain source artefact via metadata hash matching) is a stronger signal than block-explorer-side verification and should be preferred where available.

## Discussion

T6.001 is OAK's first standalone T6 Technique. The choice to start with source-verification mismatch reflects three considerations: the failure mode is generic across Tactic classes; the detection signal is unambiguous (deterministic recompile); and the underlying tooling for detection is mature enough that contributors can implement detection without extensive new infrastructure. Future T6 Techniques in v0.x updates may include "audit-claim mismatch" (claimed audit by a firm that did not audit), "renouncement-but-not-really" (proxy-admin retains control after marketing-claimed renouncement), and "audit-pending marketing-claim" (token marketed as audit-pending where no audit is in progress) — each reflects a generic evasion pattern that modifies primary Techniques across multiple Tactics.
