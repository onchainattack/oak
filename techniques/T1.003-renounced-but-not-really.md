# OAK-T1.003 — Renounced-But-Not-Really (Proxy-Upgrade Backdoor)

**Parent Tactics:** OAK-T1 (Token Genesis), OAK-T6 (Defense Evasion)
**Maturity:** stable
**Chains:** EVM (primary)
**First documented:** systematic 2022 onward; Shido 2024 as a canonical large-scale named case
**Aliases:** "fake renounce", "proxy backdoor", "ghost owner", "transferOwnership-not-really"

## Description

A class of OAK-T1 patterns in which a token contract publicly claims (in marketing or via on-chain ownership-renouncement events) that the deployer no longer holds privileged authority over the contract — while in fact retaining authority through a proxy-pattern indirection, an EIP-1967 admin slot, an ownership-bearing dependency contract, or a separate authority-bearing role that was not part of the renouncement claim. From a defender's perspective, T1.003 is the primary pattern that defeats the "is the contract renounced?" heuristic that risk-team and wallet-vendor static analysis traditionally relied on; per `[nomicproxybackdoor]` and `[quillauditsbackdoor]`, multiple distinct sub-patterns under this umbrella are used in the wild.

T1.003 typically combines with T6.001 (Source-Verification Mismatch) and is itself a strong T6 (Defense Evasion) modifier on whatever underlying T1 or T5 Technique it accompanies — by hiding the residual authority surface, it defeats pre-trade risk assessment that would otherwise flag the underlying primary Technique.

## Observed indicators

- Public claim or on-chain `OwnershipTransferred(prior, 0x000...0)` event suggesting renouncement, paired with continued evidence of privileged actions (parameter changes, upgrades, role grants) executed against the contract from the prior or a related EOA.
- Proxy-pattern contract (transparent / UUPS / Beacon) whose `implementation()` slot can be updated by an admin role that is *not* the renounced owner.
- EIP-1967 admin slot (`0xb53127684a568b3173ae13b9f8a6016e243e63b6e8ee1178d6a717850b5d6103`) reads non-zero after surface-level `renounceOwnership` — read directly via `eth_getStorageAt(contract, slot)`.
- EIP-1967 implementation slot (`0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`) reads to a contract that is itself owned / upgradable by a non-renounced authority.
- AccessControl `DEFAULT_ADMIN_ROLE` (`0x00...0`) holder set is non-empty after surface-level renouncement; verify via `getRoleMember(DEFAULT_ADMIN_ROLE, 0)` against the live contract.
- Beacon-pattern indirection: contract reads its implementation from a beacon contract whose `upgradeTo` is callable by a third party.
- Dependency contracts (timelock, governance, fee-collector, staking) that retain authority over the "renounced" token contract — direct ownership passthrough.
- Source-verified contract whose deployed bytecode introduces unexpected post-renouncement upgrade paths or selectors that mutate authority state — combine with T6.001.
- Cross-token cluster: deployer EOA appears as the proxy admin on multiple tokens whose surface-level `Ownable` is renounced.

## Detection signals

- **Static, pre-trade — authority-graph enumeration (M05):** enumerate every authority-bearing role in the contract before treating any single-role renouncement as conclusive. Required reads: `owner()`, `getRoleAdmin(DEFAULT_ADMIN_ROLE)`, `getRoleMember(DEFAULT_ADMIN_ROLE, n)`, EIP-1967 admin slot, EIP-1967 beacon slot, OZ AccessControl events history. Flag any non-renounced authority surface.
- **Proxy-pattern review:** read the EIP-1967 admin slot directly via `eth_getStorageAt(contract, 0xb531...6103)`; flag mismatch with the surface-level `owner()` view-call result. The same check on the implementation slot identifies UUPS-pattern indirection.
- **Bytecode diff (M03):** for source-verified contracts, diff canonical compiled bytecode against deployed runtime; flag any selector that mutates authority state but is absent from the verified source. This catches T6.001-combined T1.003.
- **Dynamic, simulation-based:** simulate post-renouncement parameter / upgrade transactions from the historical owner address and from any address holding `DEFAULT_ADMIN_ROLE` or proxy admin authority; if any succeed, the renouncement is not effective.
- **Indexer query (Dune / Goldsky):** join the set of `OwnershipTransferred` events with subsequent privileged-action events (`Upgraded`, `RoleGranted`, `Initialized`) on the same contract; alert on any privileged action emitted after a `OwnershipTransferred(_, 0x0)` event.
- **Cross-token deployer cluster (M04):** if a deployer's prior contracts used the same renouncement-but-not-really pattern, treat new deployments from the same deployer as elevated-risk regardless of source-verification claims.

## Real-world examples

- **Shido — Ethereum — 2024-01** — The StakingV4Proxy contract experienced an unauthorised ownership transfer followed by a proxy upgrade introducing a hidden `withdrawToken` function; the upgrade drained approximately 4.3 billion SHIDO tokens. Canonical T1.003 example for the proxy-upgrade-backdoor sub-pattern: the surface-level renouncement claim did not bind the proxy admin slot, which was subsequently exploited. Source: `[quillauditsbackdoor]`.
- **Chibi Finance — Arbitrum — 2023-06-27** — Approximately $1M drained via a `panic` function gated by `onlyGov`, where the deployer granted the `_gov` role to a malicious contract and triggered emergency-withdrawal of all funds across eight protocol contracts. The pattern is structurally T1.003: a residual `_gov` authority that was not visible in the surface-level renouncement framing. The protocol UI and social presence were taken offline immediately after extraction; CHIBI fell ~98%. Canonical 2023 worked example for the **`onlyGov`-modifier residual-authority sub-class**: the residual authority was *named in the verified contract source*, gated by `onlyGov` rather than `onlyOwner`, and reachable via the deployer's ability to point `_gov` at a malicious contract. Sources: `[halbornchibi2023]`, `[certikchibi2023]`, `[cointelegraphchibi2023]`, `[coindeskchibi2023]`. See `examples/2023-06-chibi-finance-onlygov-arbitrum.md`.
- **Magnate Finance — Base — 2023-08** — Approximately $6.4M exit-scam on a Base-native lending protocol; the deployer modified the price oracle (a residual non-renounced authority) to drain user deposits. The deployer EOA was directly linked on-chain to the Solfire (Solana, January 2022, ~$4.8M) and Kokomo Finance (BNB Chain, March 2023, ~$5.5M) deployer cluster — cumulative cluster extraction ~$16.7M across 19 months / three chains. ZachXBT publicly warned the community **hours before** the Magnate rug, anchored to the cluster-attribution evidence. The Magnate case is the canonical T1.003 + M04 (funder-graph) + T8.002 (cross-chain operator continuity) chained-attribution worked example for v0.1. Sources: `[theblockmagnate2023]`, `[cointelegraphmagnate2023]`, `[coingapemagnatecluster2023]`, `[trmsolfire2022]`. See `examples/2023-08-magnate-finance-base-deployer-cluster.md`.
- **Cohort context** — `[nomicproxybackdoor]` documents the broader pattern of malicious proxy backdoors and their detection methodology. Per-token scam-scanner projects (Token Sniffer, GoPlus, RugCheck) maintain risk scoring that includes T1.003-style residual-authority detection as one component; cohort-scale prevalence is tracked in `[chainalysis2025rug]` and `[slowmist2024report]`.

## Mitigations

- **OAK-M05 (Authority-Graph Enumeration)** — required: enumerate every authority-bearing role before treating any single-role renouncement as conclusive. This is the canonical mitigation for T1.003.
- **OAK-M01 (Source-Bytecode Verification)** — paired: ensures the verified source includes every authority-mutating selector present in deployed bytecode.
- **OAK-M03 (Continuous Bytecode-Diff Monitoring)** — for proxy-pattern tokens, monitor implementation upgrades after listing.
- **OAK-M02 (Static-Analysis Pre-Deployment)** — flag proxy patterns and AccessControl roles at deployment time.
- **OAK-M16 (Pre-Deployment Audit and Formal Verification)** — for tokens claiming "renounced", require explicit audit attestation that *all* authority-bearing roles are renounced or governance-gated.
- **OAK-M17 (Time-Locked Governance and Multi-Block Quorum)** — for legitimate non-renounced upgrade authorities, require timelocked governance gating with non-trivial delay.
- **OAK-M25 (Listing-Time Source-Verification + Audit-Status Gate)** — venue / launchpad gate.

Operational: risk teams enumerate every authority-bearing role; treat proxy-pattern contracts as elevated risk regardless of surface-level renouncement claims. Token vendors / launchpads require either (a) immutable non-proxy contracts or (b) full disclosure of every authority-bearing role and its renouncement / governance-gating status as a listing precondition. Wallet-vendor pre-trade UX surfaces residual-authority surface; do not surface "renounced" labels based on surface-level `Ownable` checks alone.

## Discussion

T1.003 is one of the highest-leverage Techniques in OAK for the static-analysis-tooling-improvement argument. The traditional "is the contract renounced?" heuristic — surface-level `owner() == 0x0` check — is the canonical example of a defensive control that operators have systematically engineered around. Modern detection requires multi-role authority enumeration, proxy-admin-slot direct reads, AccessControl-role enumeration, and dependency-contract review.

Sub-pattern split deferred to v0.x: the umbrella class includes proxy-admin retention, EIP-1967 slot retention, AccessControl role retention, dependency-contract authority retention, and beacon-indirection retention as structurally distinct sub-patterns that may warrant their own T1.003.NN sub-Techniques. The detection signal is similar across sub-patterns (M05 + M03), but the static-rule shape differs enough that operators of detection tooling will eventually want sub-Technique granularity.

Calibration caveat: legitimate proxy-pattern contracts with timelocked governance authority are *not* T1.003 — they are normal upgradable protocol architecture. v0.1 detection rules will produce false positives in this regime; the pragmatic resolution is M05 + a governance-allowlist of known timelock contracts, mirroring the T1.001 calibration approach.

Attribution-strength caveat: at the cohort scale, "deployer cluster reuses T1.003 across multiple deployments + at least one prior incident in the cluster" is a high-confidence malicious classification. At the single-deployment scale, T1.003 alone is "elevated risk" pending behavioural confirmation.
