# OAK-T6.005 — Proxy-Upgrade Malicious Switching

**Parent Tactics:** OAK-T6 (Defense Evasion); modifier on OAK-T1 (Token Genesis), OAK-T9 (Smart-Contract Exploit), OAK-T10 (Bridge / Cross-Chain) where the upgrade itself is the attack vector
**Maturity:** emerging
**Chains:** EVM (primary); cross-chain bridge variants (Polkadot ⇄ EVM canonical at v0.1)
**First documented:** structural class anchored from approximately 2022; canonical contract-layer-message-forgery 2026-04 (Hyperbridge)
**Aliases:** "verified-then-malicious upgrade", "proxy implementation switching", "upgrade-as-attack-vector", "post-verification implementation substitution"

## Description

Contracts verified at deployment with a benign implementation, then proxy-upgraded to a malicious implementation post-verification. The structural feature is that the *upgrade itself* is the attack vector — the protocol's standing surface (audit, source verification, defender-tool baseline) holds, but the implementation logic is substituted for attacker-controlled bytecode after the trust-substrate claims have been established.

T6.005 has **two structurally-distinct sub-classes**, each with its own detection-signal locus and mitigation surface:

- **(a) Operator-key-compromise sub-class** — the upgrade authority is reached via stolen / compromised operator keys. The on-chain extraction primitive is the upgrade; the proximate cause is off-chain key compromise. Detection-signal locus: off-chain key-handling-monitoring layer. Canonical: Steadefi August 2023 (~\$1.14M, internal-key compromise); Audius July 2022 storage-collision (~\$6M, adjacent operator-key-compromise sub-class); Magnate Finance August 2023 deployer-cluster (adjacent operator-side anchor).
- **(b) On-chain-message-forgery sub-class** — the upgrade authority is reached via forged cryptographic proof or message-verification flaw without any pre-stolen keys. The entire attack chain lives on-chain. Detection-signal locus: on-chain message-verification correctness layer. Canonical: Hyperbridge April 2026 (~\$2.5M realised / ~\$1.2B notional, MMR-proof-reuse forgery); Wormhole February 2022 white-hat-disclosed uninitialised-implementation pattern; AllianceBlock August 2024 re-initialisation flaw.

T6.005 is **structurally distinct** from adjacent defense-evasion-class Techniques:

- **T6.001 (Source-Verification Mismatch)** — covers verified-but-different-bytecode at deployment. T6.005's distinguishing feature is that the verification was correct *at the moment of deployment* and the divergence is introduced via a subsequent legitimate-mechanism upgrade event.
- **T6.002 (Fake Audit Claim)** — false audit attestation. T6.005's audit may have been entirely legitimate; the upgrade post-audit is what introduces the malicious implementation.
- **T9.004 (Access Control Misconfiguration)** — covers access-control flaws as the standing root cause. T6.005 is the *upgrade-as-vector* specialisation where the upgrade-authority access path is the attack chain's load-bearing primitive.
- **T11.001 (Third-Party Signing-Vendor Compromise)** — adjacent for the operator-key-compromise sub-class where the upgrade-authority key is reached via a signing-vendor compromise; the cases chain when the vendor surface is the upstream entry-vector.

T6.005 is the canonical generic modifier on T1 / T9 / T10 classes when the upgrade itself is the attack vector. The downstream extraction primitive can be a token mint (T6.005 + T6.006 chain in the bridge-internal-mint sub-pattern), a vault drain (T6.005 + T9.004 chain), or a counterfeit-token deployment (T6.005 + T6.006 chain in the deployer-cluster-rug sub-pattern).

## Observed indicators

- **Pre-event:** legitimate audited deployment with a non-trivial upgrade-authority surface (multisig, governance contract, cross-chain message handler, single deployer EOA). Standing T6.005 surface from the moment of deployment.
- **At-event (operator-key sub-class):** off-chain key-handling anomaly (workstation breach, internal-IR escalation, anomalous authentication on the upgrade-authority signer set); on-chain `upgradeTo` / `upgradeAndCall` / equivalent event emitted from a non-governance pathway.
- **At-event (on-chain-forgery sub-class):** on-chain admin-authority change event from a non-deployer / non-governance EOA following a cross-chain message receipt or proof-verification call; mint events / state-mutation events without corresponding upstream-source events (e.g., bridge mint without deposit-correlation).
- **Pattern continuity:** T6.005 typically chains downstream into T6.006 (counterfeit-token impersonation via the post-upgrade authority) or into T5 outflow (vault drain following implementation switch).

## Detection signals

- **Off-chain (operator-key sub-class):** key-handling-monitoring at the upgrade-authority signer set; vendor-disclosure feeds for internal-IR escalations; behavioural anomaly detection on signer-workstation telemetry.
- **On-chain (both sub-classes):** alert on `upgradeTo` / `upgradeAndCall` / proxy-implementation-change events emitted from a contract whose standing pattern is upgrades only via governance proposal; alert on admin-authority-change events without corresponding governance event; alert on cross-chain-message-receipt admin-authority changes.
- **Bridge-monitoring layer:** mint events without deposit-correlation; admin-rotation events triggered by cross-chain messages whose proof binding is below current-best-practice strength.
- **Cross-chain-proof-verification correctness:** auditor-side verification that cryptographic proofs commit *uniquely* to the message they authorise — not just that the proof is cryptographically valid. The Hyperbridge MMR-proof-reuse flaw is the canonical 2026 case for this defender check.
- **Contract-monitoring tooling:** Forta, OpenZeppelin Defender, BlockSec PhalconHQ runtime detectors for proxy-implementation-change events; Etherscan / chain-explorer "implementation changed" alerts.

## Real-world examples

- **Hyperbridge — Polkadot ⇄ Ethereum / Base / BNB / Arbitrum — 2026-04-13 (canonical on-chain-message-forgery sub-class)** — Loss approximately \$2.5M realised; \~\$1.2B notional in counterfeit bridged-DOT plus comparable counterfeit mints across ARGN / MANTA / CERE. The attacker exploited an MMR-proof-verification flaw that allowed an outdated valid proof from an earlier legitimate transaction to be accepted as authorisation for a different message. By pairing the proof with a forged admin-rotation message, the attacker re-assigned admin authority on the bridged-DOT token contract and used the acquired authority to mint counterfeit tokens without depositing backing collateral. The case is the canonical contract-layer-message-forgery T6.005 worked example — no operator keys were stolen, no pre-deployment social engineering occurred, the bridge contract was the legitimate audited deployment up to the moment the forged proof was accepted. See [`examples/2026-04-hyperbridge-merkle-proof-counterfeit-mint.md`](../examples/2026-04-hyperbridge-merkle-proof-counterfeit-mint.md).
- **USPD CPIMP — Ethereum — 2025-12** — Deploy-init frontrun + shadow proxy malicious-switching sub-class. Attacker seized proxy-upgrade authority via Multicall3 frontrun of the deployer's initialization sequence (2025-09-16), held admin authority for ~90 days while presenting a legitimate Etherscan verification surface, then upgraded the proxy from a forwarding shadow to an extraction-enabling implementation that minted 98M USPD and drained 232 stETH (~\$1M). Pairs T6.005 (the malicious upgrade) with the engineered-verification-surface T6.001 sub-class. See [`examples/2025-12-uspd-cpimp-clandestine-proxy.md`](../examples/2025-12-uspd-cpimp-clandestine-proxy.md).
- [`examples/2024-06-uplift-dao-proxy-upgrade.md`](../examples/2024-06-uplift-dao-proxy-upgrade.md) — Uplift DAO malicious proxy-upgrade treasury drain — Ethereum — June 2024. ~$1.8M drained via single-EOA `upgradeTo()` call on treasury proxy. Canonical T6.005 anchor for the single-EOA-upgrade-authority sub-pattern within the operator-key-compromise sub-class: the structural failure mode is the concentration of upgrade authority in a single EOA, which bypasses all access-control logic in the audited implementation regardless of audit quality.
- [`examples/2026-04-wasabi-protocol-uups-proxy-admin-key-exploit.md`](../examples/2026-04-wasabi-protocol-uups-proxy-admin-key-exploit.md) — Wasabi Protocol UUPS Proxy-Upgrade Admin-Key Exploit — Apr 2026 — $5.9M — T6.005 + T9.004; the canonical 2026 single-EOA upgrade-authority anchor. Attacker compromised the UUPS proxy-admin private key and deployed a malicious implementation, draining ~$5.9M. No timelock on upgrades, no multisig on admin key — a single-key compromise converted into a full protocol takeover. Most funds recovered through exchange freezes and white-hat negotiation.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T6.005 is a runtime-monitoring + audit-control surface; the bridge-monitoring sub-pattern (admin-authority-change without governance event; mint without deposit-correlation) is closer to MG's runtime risk-engine scope than the audit / static-analysis sub-pattern. v0.x integration target.
- Vendor coverage outside MG: pre-deployment auditors (Halborn, Trail of Bits, OpenZeppelin, ConsenSys Diligence) for upgrade-pathway review; runtime monitoring (Forta, OpenZeppelin Defender, BlockSec PhalconHQ) for proxy-implementation-change events; bridge-side telemetry (varies by bridge; Hyperbridge did not have such monitoring at the time of the April 2026 incident).

## Mitigations

- **Pre-deployment auditors:** verify upgrade-authority pathway as a first-class review item; flag single-EOA upgrade authorities; flag absence of timelock between upgrade proposal and execution; verify cross-chain-message proof-binding strength (per-message commitment, not just proof validity).
- **Protocols:** apply timelock between upgrade proposal and execution to expose proposed implementation bytecode to a defender-review window; restrict upgrade authority to multisig / governance contract; for bridges, bind cryptographic proofs uniquely to the message they authorise via per-message commitments.
- **Risk teams:** maintain per-protocol inventories of (a) upgrade-authority signer set, (b) timelock duration if any, (c) cross-chain-message-driven admin-authority change pathways. Treat protocols whose admin-authority on destination-chain token contracts is gated by cross-chain-message verification as elevated T6.005 risk if the message-verification primitive admits replay, forgery, or signature-aliasing.
- **Bridge-side telemetry:** real-time monitoring of admin-authority-change events without governance correlation; mint events without deposit-correlation; cross-chain-message-receipt admin events.
- **Operator-side (operator-key sub-class):** key-handling-monitoring at the upgrade-authority signer set as a first-class operational-security control; cross-reference T11.001 mitigations for vendor-side build-pipeline integrity.

OAK Mitigation cross-references at v0.1: gap. v0.x candidates: "OAK-M-x upgrade-authority timelock-and-multisig discipline"; "OAK-M-x cross-chain-proof per-message commitment binding"; "OAK-M-x bridge admin-authority-change real-time monitoring".

## Discussion

T6.005 sits at an unusual intersection of OAK's Tactic structure: it is filed under T6 (Defense Evasion) because the structural feature is the *evasion of the audit / source-verification / standing-implementation trust-substrate*, but it is operationally a generic modifier on T1 / T9 / T10 attacks where the upgrade is the attack vector. Future contributors should preserve this dual framing — T6.005 is the parent for the upgrade-itself-as-attack-vector primitive, and the downstream extraction primitive determines whether the case chains into T1 (token genesis), T9 (smart-contract exploit), or T10 (bridge / cross-chain).

The two-sub-class split (operator-key vs. on-chain-forgery) is the analytically distinctive feature of the Technique. The detection-signal locus differs entirely between the sub-classes:

- **Operator-key sub-class** lives at the off-chain key-handling-monitoring layer. Mitigation surface is at the operator's signing-infrastructure security posture. Steadefi (August 2023) is the canonical compact reference; the case is structurally adjacent to T11.001-class signing-vendor compromise but with the customer-organisation-as-victim shape inverted to operator-as-target.
- **On-chain-forgery sub-class** lives at the on-chain message-verification correctness layer. Mitigation surface is at the cryptographic-proof-binding strength. Hyperbridge (April 2026) is the canonical clean reference because no off-chain operator key was compromised — the entire attack chain lives on-chain.

The case generalises beyond bridges. Any contract whose upgrade authority is gated by message verification (cross-chain bridge admin authority; oracle-driven upgrade pathway; governance-driven upgrade pathway with off-chain vote aggregation) is exposed to the on-chain-forgery sub-class if the verification primitive admits replay, forgery, or signature-aliasing. Wormhole's 2022 uninitialised-implementation bug is the EVM-internal analogue (different mechanism, same downstream); Audius 2022 is the storage-collision EVM-internal analogue.

The notional-vs-realised loss gap (Hyperbridge: \~\$1.2B notional vs. \~\$2.5M realised) reflects destination-pool-liquidity constraints when the downstream chain is T6.006 counterfeit-mint plus DEX cash-out. T6.005 contributors should record both notional and realised loss when the chain includes T6.006 because the ratio is a useful structural signal across the cohort. Forensic coverage that reports only the realised cash-out figure underestimates the *attempted* extraction scale by 1–3 orders of magnitude.

Promotion to `stable` will follow once the worked-example anchor set expands to multiple per-incident files for each sub-class. Currently the operator-key sub-class is anchored at Audius and Magnate (with Steadefi as the canonical pipeline target for v0.x) and the on-chain-forgery sub-class is anchored at Hyperbridge alone (with Wormhole 2022 and AllianceBlock 2024 as pipeline targets). The single-anchor-per-sub-class state is sufficient for `emerging` but warrants additional anchoring before `stable`.
