# OAK-T4.010 — Fake Security-Tool / Browser-Extension Phishing

**Parent Tactics:** OAK-T4 (Access Acquisition)
**Maturity:** emerging
**Chains:** EVM, Solana (wallet-browser-extension surface is cross-chain; fake MetaMask, Rabby, Phantom, Trust Wallet extensions target the respective chain ecosystems); cross-chain
**First documented:** 2022–2023 (fake MetaMask extension campaigns; fake "Ledger Live" Chrome extensions)
**Aliases:** "fake browser extension", "counterfeit security tool", "fake wallet extension", "malicious browser extension phishing"

## Description

Phishing via fake browser extensions that impersonate legitimate security tools — MetaMask, Ledger Live, Rabby, Phantom, Trust Wallet, or "wallet security scanners." Users install the fake extension believing it enhances their wallet security; the extension reads wallet state, exfiltrates seed phrases/private keys, or injects malicious transactions at sign time. The extension persists across browser sessions, has access to all browser tabs via its declared permissions, and can modify transactions in-flight — a structurally deeper compromise surface than a one-time phishing web page.

T4.010 is distinct from T4.008 (Fake-DEX Clone-Frontend Phishing — website-based) and T4.009 (Pre-Token Anticipation Phishing — token-launch timing) because the phishing surface is a **persistent browser extension** rather than a web page. The extension survives tab closure, browser restart, and in some cases OS reboot; it can observe all browsing activity, inject content scripts into every page the user visits, and intercept wallet-signing flows at the browser-extension API layer rather than at the DOM/page level.

Structurally adjacent to T4.002 (Compromised Front-End) but the compromise surface is the user's own browser (the extension is installed voluntarily by the user who believes it to be a legitimate security tool), not the protocol's frontend infrastructure. The legitimate extension vendor's publisher account and extension store listing are not compromised; the attacker publishes a separate counterfeit listing under a typosquatted publisher name or developer account.

The on-chain extraction surface chains into T4.001 (Permit2 Authority Misuse), T4.004 (Allowance/Approve Drainer), T4.005 (setApprovalForAll NFT Drainer), and T11.007.003 (Brand-Trust-Leveraged Active Phishing for Seed-Phrase Exfiltration) — the fake extension can both exfiltrate the seed phrase (T11 chain) and inject malicious approval transactions (T4 chain). The dual-extraction surface is structurally broader than a single-mechanism phishing web page.

## Observed indicators

- Browser extension with name and icon matching a legitimate wallet/security-tool brand but published from an unofficial Chrome Web Store / Firefox Add-ons publisher account.
- Extension requests broad permissions: read/write to all sites (`*://*/*`), clipboard access, tab access, or native messaging host permission.
- Extension manifest includes wallet-keyword typosquat (e.g., "MetamaskPro," "LedgerLiveSecurity," "RabbyWalletPlus").
- Extension's store listing has significantly fewer reviews/ratings than the legitimate extension, or reviews are templated/bot-generated.
- Extension injects content scripts that hook `window.ethereum`, `window.solana`, or equivalent wallet-provider objects to intercept transaction parameters at sign time.
- Extension's update URL in the manifest points to an attacker-controlled server rather than the legitimate vendor's update infrastructure.

## Detection signals

- Browser-extension permission audit: flag extensions requesting `*://*/*` host permissions combined with clipboard or native-messaging permissions, particularly when the publisher is not a verified wallet-vendor account.
- Extension-publisher verification against the legitimate vendor's official publisher account and store-listing URL; flag publisher-name typosquats.
- Domain-reputation services (ScamSniffer, Web3 Antivirus, SEAL ISAC) flagging typosquat extension-names and associated phishing domains.
- Wallet-side warnings when an unofficial browser extension interacts with wallet-signing flows — the wallet can detect injected content scripts that hook the provider object and warn the user.
- Browser-extension hash/ID blocklisting: community-curated lists of known-malicious extension IDs distributed to wallet vendors and browser security teams.

- **Fake MetaMask Chrome extensions — 2022–2023.** Multiple counterfeit MetaMask extensions published to the Chrome Web Store under typosquatted publisher names; extensions exfiltrated seed phrases by injecting a fake "import wallet" / "verify seed phrase" prompt into the legitimate MetaMask UI flow. Several campaigns operated for weeks before Google's extension-review team removed the listings.
- **Fake Ledger Live browser extensions — 2022–2024.** Counterfeit "Ledger Live" Chrome extensions prompting users to enter their 24-word recovery phrase to "sync" their hardware wallet with the browser; the recovery phrase was exfiltrated to attacker-controlled servers. Ledger has no legitimate browser-extension-based Live application — the entire extension class is counterfeit.
- **Fake Rabby / Phantom extensions — 2023–2025.** Counterfeit Rabby Wallet and Phantom Wallet extensions targeting EVM and Solana users respectively; extensions injected malicious transaction-parameter overrides at sign time, replacing the user's intended recipient address with an attacker-controlled address while the confirmation UI displayed the legitimate destination.
- [`examples/2024-01-fake-metamask-extension-chrome-store.md`](../examples/2024-01-fake-metamask-extension-chrome-store.md) — Fake MetaMask Chrome extension campaign under typosquatted developer name "MetaMask Labs," January–June 2024. Over 30,000 installations and ~$3.2M drained via dual seed-phrase-exfiltration and transaction-injection surfaces. Canonical illustration of the typosquatted-developer-name sub-pattern of T4.010.
- [`examples/2023-2025-fake-revoke-cash-wallet-security-extension-phishing.md`](../examples/2023-2025-fake-revoke-cash-wallet-security-extension-phishing.md) — Fake Revoke.cash / wallet-security browser-extension phishing, 2023–2025; ~$3.5M+ cumulative. Counterfeit security-tool extensions (Revoke.cash, Wallet Guard, Pocket Universe) exploiting user security-seeking behaviour as the compromise vector. Includes the simulation-display-to-confirmation-injection gap sub-pattern (fake transaction-simulation extensions that display benign simulation results then inject malicious transaction parameters at confirmation).

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T4.010's primary detection surface is at the browser-extension store review layer and the wallet-side extension-origin verification layer, not the on-chain runtime risk-engine surface MG covers. Extension-ID blocklisting and publisher-verification are browser-platform-side mechanisms.
- Vendor coverage: ScamSniffer (phishing-extension domain and ID tracking), Web3 Antivirus / SEAL ISAC (extension blocklists), MetaMask (extension-origin warnings and phishing detection), Rabby (extension-integrity checks), Phantom (extension-origin verification), Chrome Web Store / Firefox Add-ons review teams.

## Mitigations

- **Official extension-store publisher verification (highest-leverage):** Chrome Web Store and Firefox Add-ons should require enhanced verification for extensions using wallet/security-tool brand keywords in their name or description; verified publisher badges should be visually prominent and user-facing.
- **Browser-extension permission review before installation:** users should review requested permissions before installing any wallet-adjacent extension; extensions requesting `*://*/*` plus clipboard access should trigger elevated scrutiny.
- **Wallet-side extension-origin verification at sign time:** wallet providers (MetaMask, Rabby, Phantom) should detect injected content scripts that hook the wallet-provider object and warn the user; extension-ID allowlisting for known-legitimate extensions that interact with the wallet API.
- **Legitimate-vendor defensive registration:** wallet and security-tool vendors should defensively register typosquatted extension names and publisher accounts on all major browser extension stores; publish canonical extension IDs and store-listing URLs that wallet UIs and security tools can verify.
- **User-side hygiene:** install browser extensions only from the official store listing linked from the vendor's verified website; verify the publisher name matches the legitimate vendor; treat any extension requesting seed-phrase or private-key input as malicious by default — no legitimate wallet extension ever asks for a seed phrase.

## Discussion

T4.010 captures the browser-extension phishing surface as a structurally distinct sub-class of Access Acquisition (T4). The extension's persistence — it survives tab closure, browser restart, and in some cases OS reboot — gives the attacker a **standing access surface** that a one-time phishing web page does not provide. The extension can observe all browsing activity, intercept every transaction the user signs, and exfiltrate data continuously over the extension's installed lifetime.

The structural distinction from T4.008 (Fake-DEX Clone-Frontend) is the **substrate of persistence**. T4.008's phishing surface is a web page — the user lands on the phishing page, signs a transaction, and leaves; the attack is per-session. T4.010's phishing surface is a browser extension — the user installs it once and the extension persists across all subsequent sessions, observing and potentially modifying every wallet interaction until the user manually uninstalls it. The detection surface shifts from "detect the phishing domain before the user visits it" (T4.008) to "detect the malicious extension before the user installs it, or detect the extension's injection at sign time" (T4.010).

The dual-extraction surface — seed-phrase exfiltration (T11 chain) plus transaction-parameter injection (T4 chain) — makes T4.010 broader in realised-loss shape than a single-mechanism phishing page. A fake extension that exfiltrates the seed phrase captures **all** wallets derived from that phrase across all chains and all accounts; a fake extension that injects malicious transaction parameters captures only the assets in the currently-connected wallet. The seed-phrase-exfiltration arm is the higher-severity extraction path and maps to T11.007.003 (Brand-Trust-Leveraged Active Phishing for Seed-Phrase Exfiltration).

The class is marked `emerging` rather than `stable` because (a) the per-campaign attribution depth is partial — most campaigns are attributed to pseudonymous extension-store publisher accounts without identified human operators, (b) the extension-store review enforcement maturation is ongoing, and (c) the wallet-side extension-origin verification surface is not uniformly deployed across wallet vendors. Promotion to `stable` will follow once per-incident worked-example anchors land for the major named campaigns (MetaMask fake-extension cohort, Ledger Live fake-extension cohort, Rabby/Phantom fake-extension cohort) and the wallet-side extension-origin verification surface matures.
