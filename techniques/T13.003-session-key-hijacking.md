# OAK-T13.003 — Session-Key Hijacking

**Parent Tactics:** OAK-T13 (Account Abstraction Attacks)
**Maturity:** emerging
**Chains:** EVM (ERC-4337 / ERC-7579 smart accounts); Solana analogues out of scope at v0.1
**First documented:** vendor advisories 2023 onward; class-level discussion in `[zhou2023sok]`; cohort of public incident write-ups remains thin at v0.1 freeze
**Aliases:** "session-key compromise", "delegated-signer hijack", "smart-session abuse", "scoped-key drainer"

## Description

A class of attacks in which an attacker obtains usable authority over an ephemeral session key that a smart account has delegated to a dApp for time-bounded and scope-bounded actions, then exercises that authority against the delegating account. Session keys are an ERC-4337 / ERC-7579 smart-account UX primitive: rather than approving every transaction interactively, the user authorises a short-lived key (often per-dApp, per-session) under explicit scope — allow-listed selectors, allow-listed targets, spend caps, expiry, gas budget — that the dApp uses on the user's behalf for the duration of the session. The defender's analytical question is not whether the underlying signature is valid; the session key's signature *is* valid by construction. The question is whether the **scope** under which the key was registered, the **persistence** of the key past the user's intended session, and the **storage** of the key material on the dApp side together limit blast radius to what the user actually authorised.

T13.003 collects four sub-patterns whose detection signal and mitigation surface differ but whose artefact (a smart-account-registered session key) is shared:

- **(a) dApp-side key compromise** — a session key generated and held by a dApp's backend leaks via logging, database breach, malicious dependency, or operator workstation malware. Within the key's remaining expiry and within its registered scope, the attacker has signing authority indistinguishable from the legitimate dApp.
- **(b) Over-broad scope at delegation** — a malicious or careless dApp solicits session-key authority whose scope (gas cap, target list, selector list, spend cap) is far broader than the dApp's stated functional need, converting what the user understood as "let this game submit moves" into a near-general signing authority within the expiry window.
- **(c) Revocation / expiry failure** — a session key persists past its intended lifetime because the smart-account implementation does not cleanly invalidate expired keys, or because per-app revocation paths are missing or hard to invoke. The artefact behaves like a permanent grant under conditions the user thought were temporary.
- **(d) Cross-application reuse** — a single session key is reused across multiple dApps that share a vendor / platform. Compromise at any one consumer expands blast radius to every account that delegated to the shared key.

T13.003 is **distinct from T4.001** (Permit2 Signature-Based Authority Misuse): the artefact in T4.001 is an EIP-712 signed permit over an EOA-held token allowance with EIP-2612-style revocation semantics; the artefact in T13.003 is a session-key registration on a smart account with vendor-defined scope schema and revocation flow. The two artefacts have different storage, different revocation UX, different blast-radius shape, and different per-vendor implementation maturity. T13.003 is **distinct from T4.005** (setApprovalForAll) because session keys can be substantially more granular — per-call selector, per-target-contract, per-spend-cap — rather than per-collection or per-token blanket authority; the mitigation argument is therefore not "avoid the primitive" but "scope the primitive correctly." T13.003 is **distinct from T4.006** (WalletConnect Session Hijack) because the compromised artefact is the on-chain-registered session key, not the off-chain WalletConnect transport session.

## Observed indicators

- Session-key registration events at smart accounts whose scope (selectors, targets, spend caps, expiry) is permissive relative to the dApp's stated functional need — particularly any-target / any-selector / large-spend-cap registrations dressed as bounded-scope keys.
- Bursts of UserOperations signed by the same session key whose target set or selector set is consistent across many delegating accounts that share a vendor, a frontend, or a SDK version — the cross-application-reuse signature.
- Session-key activity whose timestamp is past the registered expiry, where the smart-account implementation accepts the signature anyway — the revocation-failure signature.
- Session-key signing payloads whose recipient cluster matches infrastructure previously flagged in T4 / T11 / T13 cohorts — i.e., the session key is the entry artefact but the extraction destination is shared with other adversarial activity.
- dApp-side off-chain telemetry: leaked session-key material surfacing in paste sites, public repository commits, or breach corpora associated with the dApp's backend.

## Detection signals

- **Per-session-key usage tracking (smart-account / wallet layer):** maintain a per-key ledger of (registered scope, observed scope, time-to-expiry, accumulated spend) and alert on per-key drift — observed selectors not in the registered allow-list, observed targets not in the registered allow-list, spend approaching cap, signing activity past expiry. The cleanest single signal at v0.1 is "session-key signature accepted on a UserOperation whose `validateUserOp` should have rejected it under the registered scope."
- **Per-dApp session-key scope review at delegation time (wallet UX layer):** at the moment the user is asked to register a session key, surface the actual scope (selectors, targets, spend cap, expiry, gas budget) in clear language and warn on permissive values. The highest-leverage layer for T13.003 is delegation-time consent — the same lesson as T4.001 / T4.006 — because once an over-broad key is registered, on-chain detection is post-event.
- **Cross-application correlation (vendor / SDK layer):** session-key vendors should track which smart accounts delegated to a given session-key public key across consumer dApps and treat any cross-consumer key as a finding requiring justification.
- **Off-chain CTI:** monitor for leaked session-key material in dApp backend breach disclosures, public repository commits, and credential-leak corpora; cross-reference with on-chain session-key registries to enumerate exposed accounts within remaining expiry windows.

## Real-world examples

The public forensic record for T13.003 at v0.1 freeze is **thin** — there is no canonical headline incident analogous to Bybit / WazirX for T11 or to the Inferno Drainer cohort for T4. The class is real and is documented in vendor advisories and academic taxonomy work, but cohort-level public attribution is still consolidating. OAK's position is to document the class now (so that incidents can be classified cleanly when the public record lands) rather than wait for a single headline event to anchor the page.

- **Vendor advisory cohort — 2023 onward** — Major smart-account vendors (Argent, Safe via the Rhinestone / SmartSession ERC-7579 module, Biconomy, Privy via embedded smart-wallet flows) have published session-key best-practice guidance whose presence is itself the cohort-level T13.003 signal: each advisory enumerates failure modes (over-broad scope, revocation gaps, dApp-side key storage) that the vendor has observed in practice. These advisories are the cleanest public artefact for the T13.003 class at v0.1 and are tracked as `[smartsessions2024]` and `[openfortssa2026]` in the citation set.
- **Academic taxonomy framing — 2023** — `[zhou2023sok]` documents session-key authority as a distinct attack surface within the broader account-abstraction primitive set, anchoring the boundary against T4 (signature / approval phishing) and T11 (legacy custody-and-signing) at the artefact-and-scope level rather than at the symptom level.
- **OWASP smart-contract Top-10 framing — 2025** — `[owaspscstop10]` includes access-control misconfiguration as a top-tier class; session-key over-scoping is the smart-account-era specialisation of the same root failure (authority granted exceeds authority actually needed), which gives T13.003 a clean inheritance line to a familiar defender taxonomy.

Contributors who classify a public T13.003 incident in OAK should expect to chain into T7 (Laundering) for the downstream rails and should be explicit about which sub-pattern (a / b / c / d) the incident exhibits — these have meaningfully different mitigations.

## Reference implementations

- `mg-detectors-rs` — coverage **gap** at v0.1. T13.003 detection lives at the smart-account / wallet-UX layer (delegation-time scope review, per-key usage tracking) and at the session-key-vendor-infrastructure layer (cross-consumer correlation, key-storage hardening). MG's operator-behaviour scope does not reach delegation-time UX or vendor-side infrastructure; the artefact is registered at a smart account, not minted at a token launch.
- **Smart-account-vendor session-key UX maturity (the load-bearing layer at v0.1):** Argent has the longest production track record on session-key UX with explicit scope-binding semantics; Safe's ERC-7579 SmartSession module (Rhinestone) is the leading audited reference for modular session-key implementation; Biconomy publishes a session-key module with published scope semantics; Privy's embedded smart-wallet flow exposes session keys as part of its consumer onboarding surface. The v0.1 OAK position is that vendor-side maturity (clear scope language, default-tight scopes, working revocation flows, per-app revocation lists) is the single highest-leverage T13.003 mitigation surface.
- **Wallet-side coverage:** wallets that surface session-key registrations in their permission-management UI (alongside ERC-20 allowances, NFT approvals) provide the user-side complement to vendor-side scope hygiene; coverage is uneven across wallets at v0.1.

## Mitigations

- **Smart-account vendors (highest-leverage):** ship session-key modules with **default-tight scopes** (specific selectors, specific targets, conservative spend caps, short expiry, conservative gas budgets); make scope **legible** to wallet UI consumers via standardised metadata; ship working **per-app revocation** UX in the account dashboard; treat any session-key whose registered scope is "any selector / any target" as a UX anti-pattern that requires explicit elevated-confirmation flow; verify via audit that expired keys are unconditionally rejected at `validateUserOp`.
- **dApps consuming session keys:** request the **minimum scope** sufficient for the functional flow; refresh keys at session boundaries rather than holding long-lived keys; store key material in HSM-class infrastructure with the same operational discipline applied to production signing keys; publish session-key handling practices in the dApp's security documentation; treat dApp-backend session-key leakage as a high-severity incident with explicit user-notification SLA.
- **Wallets:** at delegation time, surface the registered scope (selectors, targets, spend cap, expiry, gas budget) in **clear language** and warn on permissive values; surface registered session keys in the same management surface as ERC-20 allowances and NFT approvals; provide one-click revocation; warn on registrations whose scope is structurally inconsistent with the dApp's stated category.
- **Users / operators:** prefer wallets and smart accounts whose session-key UX is mature; review registered session keys on the same cadence as ERC-20-allowance review; revoke session keys for dApps no longer in active use rather than relying on expiry alone.
- **Off-chain CTI integration:** subscribe to dApp-side breach-disclosure feeds; treat any disclosed dApp backend compromise as a session-key-exposure event for the affected user cohort within the remaining expiry window.

## Discussion

T13.003 is the smart-account-era specialisation of an old defender refrain — *the authority granted should not exceed the authority actually needed* — applied to a primitive whose UX presentation actively encourages the opposite (the entire selling point of session keys is that they reduce signing prompts, which creates pressure on dApps to request broader scopes and on wallets to surface scope details less aggressively). The defender's analytical question is therefore not whether session keys are safe in the abstract — they can be substantially safer than the T4.005-style blanket approvals they replace — but whether the vendor / dApp / wallet stack collectively defaults to tight scope, working revocation, and legible delegation-time consent.

The v0.1 maturity tag for T13.003 is **emerging** rather than **stable** because the public incident record has not yet consolidated into a canonical cohort. OAK contributors writing future T13.003 examples should expect to find the load-bearing detail in vendor advisories, dApp post-mortems, and audit-firm write-ups rather than in mainstream incident reporting; the technique page is structured to absorb that record as it arrives without requiring restructuring.

The boundary against T4 is the artefact: if the compromised artefact is a smart-account-registered session key, the case is T13.003. If the compromised artefact is an EIP-712 permit, an ERC-20 allowance, or an NFT approval over an EOA-held asset, the case stays in T4. Edge cases — for example, a dApp that mixes Permit2 flows with session-key flows, or a smart account that grants a session key whose scope includes calling Permit2 on behalf of the smart account — should be classified by the **first artefact compromised** in the chain, with the chained artefact noted explicitly in the incident write-up.
