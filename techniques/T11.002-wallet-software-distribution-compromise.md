# OAK-T11.002 — Wallet-Software Distribution Compromise

**Parent Tactics:** OAK-T11 (Custody and Signing Infrastructure)
**Parent Techniques:** OAK-T15.002 (T11.002 is the wallet-binary-distribution sub-class of generic supply-chain / vendor-pipeline compromise)
**Maturity:** stable
**Chains:** multi-chain (any chain the affected wallet supports)
**First documented:** systematic 2023 onward; canonical case Atomic Wallet June 2023
**Aliases:** "wallet supply-chain attack", "trojanised wallet update", "self-custodial wallet compromise"

## Description

A class of attacks in which the wallet software *itself* is compromised — at build time, at distribution time, or via a malicious update channel — such that a large fraction of the wallet's end-user installation base produces extraction events without any individual user-side compromise. The on-chain manifestation is a coordinated burst of extractions across thousands of distinct end-user wallets within a short window. The defender-side question is structurally different from T11.001 (vendor compromise affects organisational customer): T11.002's victims are individual self-custodial users whose security model assumed the wallet software they installed was authentic.

T11.002 is distinct from T4 (Access Acquisition) because the user did *not* sign any phishing-driven authority grant; the wallet itself is malicious or compromised, and the entry vector is the user's reasonable trust in their installed wallet software. It is also distinct from T11.001 because the affected entity is the wallet vendor's distribution channel, not a third-party signing platform used by an organisational customer.

## Observed indicators

- Coordinated burst of extraction events across thousands of distinct end-user wallet addresses, all running the same wallet software, within a short window.
- Source-cluster of the extracted funds traces back to a small set of attacker-controlled destination addresses.
- Vendor-side disclosure of build-pipeline compromise, malicious update, or compromised distribution channel at the affected wallet vendor.
- Litigation / class-action filings by affected end users targeting the wallet vendor — public-record artefact even in cases where the technical cause is not fully publicly disclosed.

## Detection signals

- Real-time on-chain: cluster outflow events by wallet-software fingerprint (transaction-construction conventions can sometimes identify the originating wallet); alert on burst patterns of structurally-similar extractions across thousands of distinct senders.
- Vendor-side: build-pipeline integrity monitoring; signed-binary verification at distribution time; user-side checksum verification for installed wallet binaries.
- Off-chain CTI: monitor wallet-vendor disclosure channels; cross-reference user-reported losses on social media against on-chain extraction patterns.

## Real-world examples

- **Atomic Wallet — multi-chain — 2023-06-03** — Loss \~\$35M initial / \~\$100M+ total across approximately 5,500 compromised end-user wallets. Elliptic attributed the incident to OAK-G01 Lazarus Group on June 6, 2023; FBI confirmed the DPRK / TraderTraitor attribution subsequently. The technical cause was not fully disclosed publicly, but per public reporting and the litigation record, the wallet software itself was the vector — affected users had not interacted with phishing dApps or signed unauthorised payloads. Funds were laundered through Garantex (Russian sanctioned exchange) post-extraction. Class-action lawsuit filed against Atomic Wallet and its proprietor. See worked example at `examples/2023-06-atomic-wallet.md`. Source: `[ellipticatomic2023]`.
- **Ledger Connect Kit — npm supply-chain — 2023-12-14** — Loss \~\$600K. The Ledger Connect Kit JavaScript library, used by hundreds of dApps to integrate Ledger hardware-wallet flows, was compromised when an attacker phished credentials from a former Ledger employee with access to the npm package. The attacker published malicious versions (1.1.5, 1.1.6, 1.1.7) that injected drainer code into any dApp that loaded the affected versions during the ~5-hour window. This is a *supply-chain* T11.002 sub-case where the compromised software is a wallet-integration library rather than a wallet binary, but the failure mode is structurally identical (user trust in installed/embedded wallet-related code). See worked example at `examples/2023-12-ledger-connect-kit.md`. Sources: industry post-mortems published by Ledger and dApp partners.
- **3CX VOIP-app supply-chain — 2023-03-29** — Cross-context T11.002-adjacent: a corporate VOIP application's build pipeline was compromised by OAK-G01 / TraderTraitor in a multi-stage chain (`[mandiant3cx2023]`). The downstream attack surface included financial-sector employees with crypto access, and the case is canonically documented as the precursor to the Bybit-class custody-vendor compromise pattern. Not a wallet-software case per se but informs the same defensive baseline (signed binaries, build-pipeline integrity).
- **BeaverTail / InvisibleFerret npm supply-chain — 2023 onward** — DPRK-attributed Contagious Interview / Wagemole campaign distributing malware via npm packages disguised as legitimate developer tooling. The malware (`[unit42beavertail2023]`) targets developer machines that hold crypto-wallet credentials and signing keys; secondary effect is wallet-software-adjacent compromise via developer-environment access. Per Unit 42 and SecurityScorecard reporting, the campaign continues at v0.1 with rolling new package names. Adjacent rather than canonical T11.002 — the affected software is developer tooling, not the wallet binary itself.

## Mitigations

- **Wallet vendors (highest-leverage):** signed-binary distribution with hardware-security-module-backed signing keys; reproducible builds; runtime integrity verification of installed binary against signed reference; separation of build pipeline from update-distribution pipeline; mandatory breach-disclosure SLA in user agreements.
- **End users:** install wallet software only from canonical sources (vendor-published checksums, signed binaries); prefer hardware-wallet integrations for high-balance holdings (T11.002 cannot extract from a hardware wallet that requires per-transaction user confirmation, even if the host wallet software is compromised — provided the hardware wallet's display shows the actual destination).
- **Regulators / certification bodies:** wallet-software certification programs that require build-pipeline integrity, signed binaries, and breach-disclosure SLAs as a precondition for certification.

## Discussion

T11.002 is the canonical "self-custodial does not equal self-secured" case study. End users of self-custodial wallets typically reason that they hold their own keys and therefore are not exposed to centralised-custody breach risk; T11.002 demonstrates that the wallet software that holds those keys is itself a single point of failure if compromised. The structural lesson — hardware-wallet integration for high-balance holdings — has been widely absorbed by the industry post-Atomic-Wallet, but the T11.002 attack surface remains for users who do not use hardware wallets.

A live debate is whether wallet vendors should bear additional regulatory or contractual obligations for build-pipeline integrity. OAK does not opine; it documents that the attack class is real, the affected end users have legal recourse via class action (slow), and the operational-security layer (signed binaries, reproducible builds, certified distribution) is the highest-leverage mitigation surface.
