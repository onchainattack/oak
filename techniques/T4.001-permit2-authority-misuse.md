# OAK-T4.001 — Permit2 Signature-Based Authority Misuse

**Parent Tactics:** OAK-T4 (Access Acquisition)
**Maturity:** stable
**Chains:** EVM
**First documented:** 2022–2023 (industry incident reports)
**Aliases:** "permit2 phishing", "signature-based asset transfer", "off-chain approval drainer"

## Description

A pattern in which a victim signs an EIP-712 / Permit2 authorization whose stated scope (as presented in their wallet UI) does not match the actual scope of the on-chain authority being granted, after which an attacker-controlled address exercises that authority to transfer the victim's assets. From a defender's perspective, T4.001 is characterized by **signature artifacts**: the on-chain transaction that exercises the authority is publicly observable, and the signed payload, when reconstructed, makes the authority's scope explicit in retrospect — but no on-chain pre-event signal exists between signing and exercising.

The off-chain-signature property is what makes Permit2 the dominant flow in modern Drainer-as-a-Service kits (Inferno, Pink, Angel, Vanilla cohort lineage per `[scamsniffer2024lineage]`): the operator collects a signed payload over HTTP, then submits `permit` / `permitTransferFrom` from infrastructure that the victim never interacts with directly, denying any wallet-, RPC-, or simulator-layer pre-event signal.

## Observed indicators

- Calls to canonical Permit2 (`0x000000000022D473030F116dDEE9F6B43aC78BA3`) `permitTransferFrom(...)` or `permitWitnessTransferFrom(...)` whose `signer` is a high-balance EOA and whose `to` clusters with previously flagged drainer infrastructure.
- ERC-2612 `permit(owner, spender, value, deadline, v, r, s)` calls where `spender` is a freshly deployed router with no other inbound legitimate flow.
- `Approval` event with `value = uint256.max` emitted by Permit2 (or by an ERC-2612 token directly) immediately preceding a `transferFrom` to a non-allowlisted address in the same transaction or following block.
- Permit-grantee ("spender") address concentrating outsized aggregate authority across many distinct victim wallets and many distinct token contracts within hours.
- Signed-permit scope (token list, allowance size, deadline) inconsistent with what a legitimate wallet UI would surface as a typical action — e.g., 30-day deadline and `uint256.max` for a one-time swap.
- Victim-side correlation: previous interaction with a domain or front-end whose signing flow has been reported as misleading; web traffic to known phishing-domain clusters tracked by ScamSniffer / SlowMist feeds.

## Detection signals

- On-chain: monitor Permit2 `permitTransferFrom` / `permitWitnessTransferFrom` events; flag those whose grantee cluster matches known-adversarial allowlists.
- Per-EOA approval-velocity outlier detection: a high-balance EOA generating multiple distinct permit-grantees within a short window is a strong post-event signal of phishing-context interaction.
- Wallet-side (telemetry where available, OAK-M31): compare claimed-action UI strings against the actual signed payload; warn on `uint256.max` value or deadlines outside a sane interactive window.
- Mempool watcher for known drainer-router target addresses (forward-deployed via `[checkpoint2023drainers]` and ScamSniffer threat-feed integrations) — provides a brief pre-block intervention window for monitored treasury wallets.

## Mitigations

- **OAK-M08 (Per-Spender Approval Audit and Revocation):** revoke unused Permit2 allowances on a recurring cadence; treat any permit grant from a phishing-context interaction as compromised regardless of whether the attacker has yet exercised it.
- **OAK-M31 (EIP-712 Permit Display and Signing Risk Heuristics):** highest-leverage layer — surface full permit-payload scope (token list, allowance size, deadline, grantee) in clear language; warn on permits whose grantee clusters with flagged infrastructure.
- **OAK-M30 (Per-dApp Domain and App-Store-Package Allowlist):** block signing-prompt origination from non-allowlisted domains.
- **OAK-M19 (Air-Gap Cold Wallet Signing):** for treasury-class wallets, route Permit2 signing through air-gapped device with full-payload review; defeats the primary phishing entry vector.
- **Protocols:** publish canonical permit-payload templates so wallets can verify expected vs actual.

## Discussion

Detection at the on-chain layer is post-event: between signing and exercising the permit, no on-chain artefact exists. The highest-leverage layer is therefore the wallet UX (OAK-M31). Reference implementations should be explicit about which layer they cover.

False-positive considerations: legitimate Permit2 flows (Uniswap router, 0x router, CowSwap) routinely use `uint256.max` allowances and 30-day deadlines, so the value/deadline heuristics in isolation are noisy. The high-precision signal is grantee-clustering against a known-bad allowlist combined with the originating-domain check (OAK-M30). Edge case: legitimate batch-trading bots produce Permit2-grantee patterns superficially similar to drainer behaviour; cluster-based attribution must incorporate originating-domain provenance to avoid suppressing them.

v0.x deferred items: cross-chain Permit2 observability (the canonical Permit2 contract is deployed at the same address on multiple EVM chains; cross-chain grantee-cluster propagation is currently single-chain in most detection feeds), and ERC-4337 account-abstraction approval semantics (where the signing artefact is a UserOperation rather than a 65-byte signature, complicating wallet-UX layer interception).
