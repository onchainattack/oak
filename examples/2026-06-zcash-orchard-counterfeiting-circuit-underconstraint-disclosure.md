# Zcash Orchard counterfeiting vulnerability — under-constrained zk-circuit soundness flaw (responsible disclosure) — 2026-05-29 / 2026-06-03

**Loss:** **no proven realised loss; a latent *unlimited-counterfeit* capability in the core shielded pool, disclosed and patched before any known exploitation.** Security researcher **Taylor Hornby** discovered a soundness flaw in the **Orchard** shielded-pool proving circuit that would have let an attacker **forge an unlimited number of counterfeit ZEC** without detection. A full proof-of-concept exploit (written with assistance from the **Anthropic Opus 4.8** model) **generated infinite counterfeit ZEC in local testing**. Because Orchard is privacy-shielded, it is **cryptographically impossible to prove whether the bug was exploited in the ~4 years it was live** — the Zcash Foundation states there is **no evidence of exploitation and no unauthorised value created**, but the property is unprovable by construction. The market priced the *uncertainty*: **ZEC fell ~31%** (to ≈\$409) within 24 hours of disclosure, and Arthur Hayes publicly announced he **dumped his entire ZEC position**, citing that non-exploitation "cannot be formally cryptographically proved impossible."
**OAK Techniques observed:** **OAK-T9.014** (Protocol-Client Consensus Bug — the canonical anchor. The flaw is in the **reference-client consensus code** (the Orchard proving/verifying circuit), not in a smart contract; it would permit transactions that **create value from nothing** in violation of the protocol's supply rules; and the fix mechanism was a **network-wide emergency soft-fork + hard-fork upgrade (NU6.2)**, not a contract patch — the exact T9.014 structural signature established by the Bitcoin value-overflow bug and CVE-2018-17144. This case extends T9.014 with a **zk-circuit-soundness sub-shape**: the inflation surface is an *under-constrained zero-knowledge proving circuit*, not an integer-overflow in a C++ arithmetic check. See [`techniques/T9.014-protocol-client-consensus-bug.md`](../techniques/T9.014-protocol-client-consensus-bug.md)). **OAK-T9.004** (Access-Control Misconfiguration — cross-referenced in the "missing-validation" reading: an *under-constrained* circuit is a verification predicate that fails to constrain an input it should have constrained, so an arbitrary/false witness is accepted. This is the same "missing constraint = missing validation" logic OAK applies to proof-verifier cases such as Veil Cash's Groth16 verifier and FOOM Cash. See [`techniques/T9.004-access-control-misconfiguration.md`](../techniques/T9.004-access-control-misconfiguration.md)). **OAK-T9** (Smart-Contract / Protocol Exploit parent).
**Attribution:** **unattributed** — this is a **responsible disclosure with no attacker**, not an exploited incident. Discovered 2026-05-29 by Taylor Hornby, disclosed to the **Zcash Open Development Lab (ZODL)** and the broader ecosystem; the security advisory ("The Orchard Counterfeiting Vulnerability — and next steps") was authored by **Zooko Wilcox, Jason McGee, and Taylor Hornby** (Shielded Labs). The case is a *capability/disclosure* entry, structurally adjacent to OAK's other unexploited-vulnerability anchors (CVE-2018-17144 duplicate-input inflation; ERC-4337 EntryPoint griefing disclosure; Coinbase Smart Wallet pre-launch H-01), not a deployed attack.

**Key teaching point:** Zcash Orchard is the **first zk-circuit-soundness anchor in OAK's protocol-client consensus-bug class — and the cleanest worked example of "privacy makes non-exploitation unprovable."** The supply-integrity guarantee of a shielded pool rests entirely on the *soundness* of its proving circuit: every shielded value must be backed by a valid spend, enforced not by a visible balance check but by the zero-knowledge proof that the transaction is well-formed. An **under-constrained circuit** — here, an element of the Orchard circuit that allowed **arbitrary/false inputs to an elliptic-curve multiplication to still be accepted as valid** — silently removes that backing requirement for a crafted witness, letting an attacker mint counterfeit shielded value that is indistinguishable, on-chain, from legitimate value. Two lessons compound: (1) **circuit soundness is consensus-critical** — a missing constraint in a proving circuit is a supply-inflation bug of the same class as a consensus-code integer overflow, and must be audited and fixed with the same severity and the same hard-fork machinery; (2) **privacy inverts the forensics** — in a transparent ledger you can prove after the fact that no inflation occurred (Bitcoin's value-overflow block was identified and reorged); in a shielded pool you *cannot*, so the residual risk is permanent uncertainty, which is itself a market-moving and trust-eroding outcome even when no exploitation occurred.

## Summary

**Zcash** is a privacy coin whose **Orchard** shielded pool (launched in the **NU5 upgrade, May 2022**) uses a Halo2-based zero-knowledge proving system to let users transact with amounts and addresses hidden, while the proof system enforces that no value is created or destroyed. The integrity of ZEC's supply inside Orchard depends on the **soundness** of that circuit: a valid proof must exist only for transactions that conserve value.

On **2026-05-29**, researcher **Taylor Hornby** discovered that an element of the Orchard circuit was **under-constrained** — specifically, the circuit permitted **arbitrary false inputs to an elliptic-curve multiplication to pass as valid**. This soundness gap meant an attacker could construct a proof that the verifier would accept for a transaction that **did not actually conserve value**, minting **counterfeit ZEC with no backing**. Hornby, assisted by the **Anthropic Opus 4.8** model, wrote a **complete exploit** and **generated unlimited counterfeit ZEC in local testing**, confirming the bug was practically exploitable, not merely theoretical.

The vulnerability had existed since **Orchard's launch in May 2022 — roughly four years undetected.** Because Orchard transactions are shielded, **there is no way to cryptographically prove whether the bug was exploited during that window.** The Zcash Foundation's position is that there is **no evidence of exploitation, no unauthorised value created, and user privacy was not affected** — but the unprovability is inherent to the shielded design.

Hornby disclosed privately to the **Zcash Open Development Lab (ZODL)**, which coordinated an emergency, two-phase response:

1. **Soft fork (2026-06-02, mainnet block 3,363,426):** disabled **Orchard actions** network-wide, halting the exploitable surface while a corrected circuit was prepared. Shielded Orchard transactions were temporarily unavailable.
2. **Hard fork — NU6.2 (2026-06-03, block 3,364,600):** activated **corrected circuit code** that properly constrains the previously-vulnerable element, and restored shielded transactions.

The public disclosure (Zooko Wilcox's advisory thread, 2026-06-05) coincided with a **~31% ZEC drawdown** and high-profile capitulation (Arthur Hayes announced selling his entire ZEC bag), as the market repriced the now-explicit uncertainty about historical supply integrity.

## Why this is structurally significant

T9.014 (Protocol-Client Consensus Bug) covers vulnerabilities in the reference-client code that determines consensus, where an attacker can produce a network-accepted transaction or block that violates the protocol's economic rules, and where the fix is a network-wide client upgrade rather than a contract patch. Zcash Orchard matches every structural criterion — and extends the class:

1. **Location.** The flaw is in the **Orchard proving/verifying circuit** shipped in the reference client (zcashd / zebra), not in on-chain smart-contract bytecode. ✓
2. **Exploit surface.** The attacker exploits a **consensus-level validation flaw** — the circuit's soundness — to create value from nothing, exactly as the Bitcoin value-overflow bug exploited `CheckTransaction()` arithmetic. The difference is the *form* of the validation: a **zk-proof constraint** rather than an integer-bounds check. ✓ (new sub-shape)
3. **Fix mechanism.** Patched via **emergency soft fork then hard fork (NU6.2)** with network-wide validator/node adoption — the T9.014 coordination-problem fix, not a governance/proxy upgrade. ✓
4. **Unexploited-disclosure pipeline shape.** Like **CVE-2018-17144** (duplicate-input inflation, found and patched before known exploitation), Orchard is a *capability* anchor: the inflation was demonstrated in a local PoC, not observed on mainnet. ✓

The **new sub-shape** is important enough to record explicitly. OAK's existing T9.014 anchors are **arithmetic** consensus bugs (Bitcoin 2010 integer overflow; CVE-2018-17144 duplicate-input). Zcash Orchard is the first **zk-circuit-soundness** consensus bug: the supply invariant is enforced by a proving circuit, and an *under-constrained* circuit is the proof-system analogue of an unchecked integer add. As more chains move supply-integrity and validity into zk circuits (zk-rollups, shielded pools, validity proofs), this sub-shape is the forward-looking risk surface — a single missing constraint in a circuit can be a network-wide unbounded-mint bug, and it is *harder* to audit than integer arithmetic because the failure is the silent acceptance of an invalid witness rather than a visibly wrong number.

The case also anchors a property OAK should track across all privacy/shielded systems: **forensic asymmetry under privacy.** Transparent-ledger consensus bugs (Bitcoin) are *recoverable and provable* — the bad block is identifiable and can be reorged, and the absence of prior exploitation can be checked. Shielded-pool soundness bugs are *neither* — exploitation is indistinguishable from legitimate use, so the residual is permanent uncertainty about supply. This makes circuit soundness a higher-stakes audit target in privacy systems than the equivalent check in a transparent one, and it makes "no evidence of exploitation" a weaker assurance than it sounds.

## What defenders observed

- **Pre-event (circuit-audit layer):** the vulnerability was an **under-constrained circuit element** — a constraint the Orchard circuit *should* have enforced on an elliptic-curve multiplication but did not, so a false witness satisfied the proof. Defender lesson: **circuit soundness review is consensus-critical and must target *missing* constraints, not just present ones.** Under-constraint bugs are the dominant class of zk-circuit vulnerability precisely because they are invisible to functional testing (valid inputs still produce valid proofs); they surface only under adversarial witness construction. Formal constraint-coverage tooling and adversarial circuit fuzzing are the relevant controls.
- **Detection asymmetry (the load-bearing observation):** in a transparent ledger, an inflation bug leaves an on-chain trace (Bitcoin's 184.5B-BTC output). In Orchard, **a counterfeit-mint proof is indistinguishable from a legitimate shielded transaction**, so there is no on-chain signal and no way to retroactively bound exploitation. Defender lesson: for shielded systems, the only assurance is *soundness of the circuit*, not *monitoring of the chain* — there is nothing to monitor.
- **At-disclosure (response design):** the two-phase **soft-fork-then-hard-fork** response is the model T9.014 remediation: a fast soft fork to *disable the vulnerable feature* (buy time, stop the bleeding) followed by a hard fork to *ship the corrected code*. The coordination cost is the network-wide upgrade, exactly as T9.014 predicts. Defender lesson: protocols with consensus-critical circuits should pre-plan a "disable the feature" soft-fork lever, because shipping a corrected circuit safely takes longer than halting an exploitable one.
- **Market layer:** the ~31% drawdown and high-profile exits occurred on *disclosure of a patched, no-evidence-of-exploitation* bug. Defender lesson: for privacy assets, the *unprovability* of non-exploitation is itself the risk the market prices — "we fixed it and saw no evidence" is a structurally weaker reassurance for a shielded pool than for a transparent chain, and communications should acknowledge that asymmetry rather than overclaim certainty.

## What this example tells contributors writing future Technique pages

- **T9.014 needs a documented zk-circuit-soundness sub-shape.** It is mechanically distinct from the integer-overflow (Bitcoin 2010) and duplicate-input (CVE-2018-17144) sub-shapes: the inflation surface is an *under-constrained proving circuit*, and the audit discipline is constraint-coverage / adversarial-witness review, not integer-bounds checking. Record which sub-shape a new T9.014 case belongs to.
- **Circuit soundness is a consensus bug, not a "smart-contract bug."** When a validity/shielded system's supply invariant is enforced by a circuit, a missing constraint is a network-wide unbounded-mint vulnerability of the same severity class as a consensus-code arithmetic bug, and it is fixed by hard fork, not by contract upgrade. Map it under T9.014, not T9.004 alone (cross-reference T9.004 for the "missing-validation" reading).
- **Record forensic asymmetry for privacy systems.** For any shielded/privacy case, note explicitly that exploitation may be *unprovable* and that "no evidence of exploitation" is a weaker assurance than on a transparent chain. This belongs in the defender section of every privacy-pool vulnerability.
- **AI-assisted vulnerability research is now a recurring detail.** The PoC was built with assistance from the Anthropic Opus 4.8 model — mirroring OAK's note that the Moonwell cbETH oracle case involved a Claude-Opus-co-authored commit. Contributors should record AI-assistance where reported, as a neutral fact about how both offensive research and defensive disclosure are increasingly conducted, not as an attribution.

## Discussion

Zcash Orchard joins OAK's protocol-client consensus-bug lineage (`examples/2010-08-bitcoin-value-overflow-bug.md` and the CVE-2018-17144 pipeline anchor) as its **first zero-knowledge sub-shape** and its **first privacy-coin** case. Where Bitcoin's 2010 bug is the canonical *transparent-ledger arithmetic* inflation — exploited on mainnet, identified, and reorged — Orchard is the canonical *shielded-circuit soundness* inflation — demonstrated in a PoC, disclosed responsibly, and patched by hard fork, but with the defining twist that **exploitation can never be proven or disproven**. The pairing teaches the class boundary precisely: same structural failure (reference-client validation lets value be created from nothing; fixed by network upgrade), opposite forensic posture (provable vs unprovable).

The case is also a forward-looking marker for the corpus. As validity proofs and shielded pools carry more of the industry's supply-integrity guarantees, under-constrained-circuit soundness bugs become the highest-leverage consensus-bug surface — a single missing constraint is an unbounded mint, it is invisible to functional testing, and in a privacy system it is invisible to chain monitoring as well. Contributors building out T9.014 and any future zk-soundness technique should treat Orchard as the anchor for "circuit soundness is consensus-critical, and privacy makes the residual risk permanent." The AI-assistance detail (PoC built with Anthropic Opus 4.8) is worth carrying as a neutral observation about the changing economics of both vulnerability discovery and responsible disclosure, consistent with how OAK already records the Moonwell cbETH AI-co-authored commit.
